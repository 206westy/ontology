## 0. 한줄요지

**결정함수·SPC가 쏟아내는 "판정"을 사람이 볼 수 있는 화면이 지금 하나도 없다 — 차트 라이브러리를 처음 들이고, 대시보드(모니터링용 위젯)와 액션보드(처리 큐)를 분리해서, "이상 웨이퍼만 바로 보이고, 클릭하면 왜 이상인지 온톨로지 캔버스에서 바로 확인되는" 화면을 코드 없이 조립하게 만든다.**

---

## 1. 목적

### 1-1. 지금 무엇이 없는가 (코드 근거)

- `dashboards`, `action_items` 테이블이 **존재하지 않는다.** "액션"은 현재 `add_class`류 그래프 편집 액션뿐이며, 판정 결과를 담는 큐 개념이 없다.
- 유일한 시각화는 **Cytoscape 캔버스**와 `HealthDashboardSheet.tsx`(`lib/graph-health.ts` 기반) 그래프 헬스 스코어카드뿐이다. 이 스코어카드는 그래프 구조 품질(고아 노드·중복 엣지·CQ 통과율)을 보는 것이지, **도메인 데이터(웨이퍼·로트·설비)의 판정 결과**를 보는 화면이 아니다.
- **차트/BI 라이브러리가 전무하다.** recharts·d3·chart.js·visx 어느 것도 의존성에 없다. 관리도(control chart), 추세선, KPI 카드 하나도 못 그린다.
- PRD-PF-B(결정함수)·PRD-PF-F(SPC/FDC)가 완성돼도 그 출력이 **API 응답으로만 존재**하고 이를 사람이 소비할 화면이 없으면 제품이 안 팔린다. 이 PRD가 그 마지막 고리다.

### 1-2. 두 화면을 분리하는 이유

사용자 요구를 그대로 따르면 "대시보드"와 "액션보드"는 목적이 다르다.

- **대시보드(모니터링)**: "지금 공정이 어떻게 돌아가고 있나"를 준실시간으로 훑어보는 화면. 관리도·추세·분포·KPI. **보는 것.**
- **액션보드(처리 큐)**: "지금 내가 뭘 처리해야 하나"만 남기는 화면. 결정함수가 불통과/이상으로 판정한 대상만 카드/표로. **처리하는 것.**

팔란티어 Workshop(대시보드형 앱 빌더)과 Quiver(작업 큐)의 대응 관계와 같다. 우리는 이 둘을 **뷰 빌더 하나**로 조립하되, 위젯 유형과 상호작용 모델은 명확히 다르게 설계한다.

### 1-3. 정직한 경계 (전편 원칙 계승)

- **완전자동 금지.** 액션보드의 "처리"는 항상 엔지니어의 확정(컨펌)을 거친다. AI/결정함수는 "이 웨이퍼는 이상입니다"까지만 말하고, "폐기 처리했습니다"는 말하지 않는다.
- **진짜 실시간이 아니다.** 현재 아키텍처는 Supabase↔Neo4j 커밋·배치 왕복이다. 이 PRD는 폴링/구독 기반 **준실시간**(측정값 유입→함수 재계산→보드 갱신)까지만 다룬다.
- **차트 라이브러리 신규 도입이 필수다.** 없는 걸 없다고 인정하고, 후보를 비판적으로 비교해 하나를 고른다(§3).

## 3. 기술 스택 — 차트 라이브러리 도입 결정

### 3-1. 후보 비교

현재 코드베이스 정합성(shadcn/Tailwind4/다크모드 CSS 변수 체계)과 관리도(control chart) 커스텀 드로잉 요구를 기준으로 4개 후보를 비교했다.

| 기준 | **Recharts** | **visx** | **Tremor (Raw)** | **ECharts** |
|---|---|---|---|---|
| 렌더 방식 | SVG (D3 wrapper) | SVG (D3 프리미티브, 조립형) | SVG (내부적으로 Recharts) | **Canvas** |
| 번들 크기(gzip, 실사용 기준) | 대략 100~130KB | 사용 모듈만 임포트 시 20~50KB | Recharts와 유사(+컴포넌트 자체 코드) | 모듈 임포트 최적화해도 150~250KB, 풀번들 1MB+ |
| shadcn/Tailwind4 정합 | ✅ **shadcn/ui 공식 Charts 레지스트리**가 Recharts 기반. `npx shadcn add chart`로 우리 톤에 바로 흡수 | ⚠️ 로우레벨 프리미티브라 직접 Tailwind로 감싸야 함(작업량↑, 자유도↑) | ✅ Tremor Raw 자체가 "복붙형 + Tailwind" 철학이라 shadcn과 설계 사상이 같음 | ❌ 자체 테마 객체 시스템. Tailwind CSS 변수와 직접 안 붙음(어댑터 레이어 필요) |
| 다크모드 | ✅ CSS 변수(`--chart-1`~`--chart-5`) 기반, 기존 `cssVar()` 패턴(Cytoscape POC에서 이미 씀)과 동일 사상 | ⚠️ 색상을 직접 전달, 다크모드는 우리가 설계 | ✅ Tailwind `dark:` 클래스 그대로 동작 | ⚠️ JS 테마 객체 전환 로직 별도 필요, 런타임 전환 시 재초기화 비용 |
| 접근성 | 🟡 기본 aria 속성 약함, 수동 보강 필요 | 🟡 SVG라 스크린리더 보강은 가능하나 수동 | 🟡 Recharts와 동일 한계 | ❌ Canvas라 DOM에 요소가 없어 스크린리더 기본 지원 최약(별도 aria 모듈 필요, 성숙도 낮음) |
| 관리도(control chart) 적합성 | ✅ `ReferenceLine`(UCL/LCL)·`ReferenceArea`(관리구간)·커스텀 `Dot`(이상점 강조)로 충분히 구현 가능 | ✅✅ 가장 자유로움 — Nelson Rules 위반 구간의 임의 하이라이트·커스텀 밴드 등 **완전 커스텀 요구엔 최적** | ✅ Recharts 상속(동일), KPI 카드·간단 추세엔 최적이나 세밀한 관리도 주석은 Recharts로 폴백 필요 | ✅✅ `markLine`/`markArea`/`markPoint`가 관리한계·이상점 표기에 **네이티브로 최적화**, `dataZoom`으로 대량 시계열 탐색 강함 |
| 대량 데이터 성능(수만~수십만 포인트) | 🟡 SVG DOM 노드 비례 증가 → 수천 포인트 넘으면 느려짐 | 🟡 Recharts와 동일 한계(SVG) | 🟡 동일 | ✅✅ **Canvas라 대량 포인트에 압도적으로 강함**(웨이퍼 수만 개 x 공정변수 시계열 같은 케이스) |

### 3-2. 권고안

**1차 도입 = Recharts (shadcn/ui 공식 Charts 컴포넌트 경로).**

- 이유: 이 제품의 북극성은 "shadcn 톤 유지 + UIUX 우선 + HITL 카드형 UI"다. shadcn이 Recharts를 공식 채택했으므로 **재배치 우선 원칙**(기존 CSS 변수 체계·다크모드·Radix 톤을 그대로 흡수)에 가장 잘 맞는다. 다크모드·접근성·shadcn 정합에서 유일하게 "그냥 붙는" 후보.
- 관리도(§5의 핵심 위젯)는 `ReferenceLine`/`ReferenceArea`/커스텀 `Dot`으로 MVP 요구(UCL/LCL 표시 + 이상점 강조)를 충분히 만족한다. Nelson Rules 8종처럼 매우 복잡한 패턴 하이라이트가 필요해지면 그 위젯 하나만 **visx로 이스케이프**하는 하이브리드 전략을 허용한다(전면 교체 아님).
- KPI 카드 로우(§5 위젯 유형)는 Tremor Raw의 카드 패턴을 **참고해 shadcn 톤으로 재구현**한다(라이브러리 자체를 추가 설치하지 않고 패턴만 차용 — 의존성 최소화).
- **ECharts는 이번 PRD 스코프에서 채택하지 않는다.** 다크모드/Tailwind 정합·접근성이 약하고, 이 단계(§4 M1~M4)에서 다루는 데이터량(로트/배치 단위 준실시간)은 SVG로 충분하다. 단, §8에 "대량 실시간 승격(웨이퍼 단위 초 단위 스트리밍, 수만 포인트 캔버스)" 시나리오가 오면 ECharts를 **부분 도입**(해당 위젯 한정)하는 것을 열린 결정으로 남긴다.

### 3-3. 기존 자산 재배치

| 신규로 필요한 것 | 재배치할 기존 자산 |
|---|---|
| 위젯 카드 프레임 | `HealthDashboardSheet.tsx`의 `MetricCard`/`ViolationList` 패턴(shadcn `Card` + `Sheet` 조합) 그대로 컴포넌트 분리 재사용 |
| 근거경로 클릭 → 캔버스 하이라이트 | `useOntologyStore.highlightNodes`(이미 `Text2CypherTab.tsx`, `AIAssistantTab.tsx`에서 쓰는 액션) 그대로 재사용. 신규 훅 불필요 |
| 위젯/보드의 스코프 경계 | `partitions`(구획) 테이블 — PRD-PF-A의 `ontology_id`가 아직 없으므로 MVP는 `partition_id` 스코프로 대시보드를 구획별로 분리 |
| 색·상태 배지 | 기존 시각언어(원=클래스, 점=인스턴스, 파선=암묵/약한관계, 실선=명시관계)를 액션 카드 상태 배지에도 확장 — 예: 파선 테두리=AI 초안 판정(미확정), 실선 테두리=엔지니어 확정 |
| 다크모드 CSS 변수 | `app/poc/cytoscape/page.tsx`의 `cssVar()` 패턴을 `--chart-1`~`--chart-5` 정의에 그대로 적용 |

## 4. 방향 (마일스톤)

의존성 순서: **차트 인프라 → 대시보드(읽기 전용) → 액션보드(쓰기/확정) → 뷰 빌더(조립) → 준실시간화**. 각 마일스톤은 단독으로 가치를 준다.

- **M1 — 차트 인프라 도입.** Recharts + shadcn Charts 컴포넌트 설치, `--chart-*` CSS 변수 정의(라이트/다크), 위젯 프레임(Card 래퍼) 컴포넌트 3종(추세/분포/KPI) 정적 프로토타입. 이 시점엔 목데이터로만 렌더.
- **M2 — `dashboards`/`dashboard_widgets` 테이블 + 최소 대시보드.** PRD-PF-B/F 출력을 위젯으로 바인딩. 고정 레이아웃(그리드) 1개 대시보드부터. 읽기 전용, 폴링 갱신.
- **M3 — `action_items` 테이블 + 액션보드(처리 큐).** 결정함수 "불통과/이상" 판정만 카드/표로 채움. 상태(대기/처리중/확정/기각) + HITL 확정 흐름 + 감사 로그. **일괄 처리 UX**(다중 선택 → 일괄 확정/기각).
- **M4 — 뷰 빌더(노코드 조립).** 비FDE가 위젯 추가/삭제/필터 설정을 UI로 조립. 위젯 config를 jsonb로 저장. 근거경로 클릭 → 캔버스 하이라이트 연결.
- **M5 — 준실시간 갱신 고도화.** 폴링(M2~M4 기본) → Supabase Realtime 구독으로 전환. 갱신 지연 30초→5초.
- **M6(선택, 후행) — 대량 성능 최적화.** 액션 아이템 가상 스크롤, 위젯 데이터 페이지네이션/샘플링. 사용량이 실제로 이 임계를 넘을 때 착수(선제 최적화 금지).

## 5. 방법론

### 5-1. 뷰 빌더

**설계 원칙: 드래그앤드롭 프리폼 캔버스가 아니라 "그리드 슬롯 + 설정 패널" 방식.** 이유 — (a) 프리폼 캔버스는 구현·QA 비용이 크고 이 단계 ROI가 낮다, (b) shadcn `Sheet`/`Dialog` 기반 설정 폼이 우리 기존 패턴(ActionCard, HealthDashboardSheet)과 UX 일관성이 높다, (c) 비FDE 사용자에게 "칸에 위젯을 고른다"가 "자유배치"보다 인지 부하가 낮다.

**위젯 유형(초기 카탈로그, 확장 가능):**

| 위젯 유형 | 데이터 소스 | 용도 |
|---|---|---|
| 관리도(Control Chart) | PRD-PF-F SPC 함수 출력(시계열 + UCL/LCL) | 공정변수 추세 + 관리한계 이탈 시각화 |
| 추세(Trend Line) | 임의 수치 속성 시계열 | MTBF·수리시간 등 일반 추세 |
| 분포(Histogram/Box) | 인스턴스 속성 집계 | Cp/Cpk류 공정능력 분포 |
| KPI 카드 | 결정함수 최신 판정 집계(통과율 등) | 한눈에 보는 요약 숫자 |
| 이상치 목록(Anomaly List) | 결정함수 "불통과" 판정 목록 | 대시보드에서 액션보드로 넘어가는 진입점 |

**근거경로 표기(필수):** 모든 위젯·액션 카드는 "어느 함수(function_id)·어느 데이터셋·언제 계산됨"을 항상 카드 하단에 1줄로 노출. 클릭 시 두 가지 동작을 제공: (1) 함수 정의 자체를 보여주는 팝오버(PRD-PF-B 화면으로), (2) 판정 근거가 된 인스턴스를 캔버스에서 하이라이트(`highlightNodes([instanceId])` 호출 + 해당 노드로 `focusNode`).

**뷰 빌더 UX 흐름:** 대시보드 진입 → "위젯 추가" 버튼 → 유형 선택(카탈로그) → 데이터 소스 선택(연결된 결정함수/데이터셋 드롭다운, 프리뷰 즉시 렌더) → 필터 조건(선택) → 저장. 코드 작성 없이 5분 이내 완료를 목표(§2 지표).

### 5-2. 데이터 모델 (신규 테이블 스케치)

> PRD-PF-A 완료 전이므로 워크스페이스/온톨로지 스코프 컬럼은 `partition_id`로 임시 대체하고, 마이그레이션 시 `ontology_id`로 승격하는 것을 전제로 설계한다(컬럼명은 확정, 값 소스만 나중에 교체).

```
dashboards
  id                  uuid pk
  partition_id        uuid fk → partitions.id            -- (PRD-PF-A 완료 시 ontology_id로 승격)
  name                text
  description         text
  layout              jsonb            -- 그리드 슬롯 배치(행/열/위젯 순서)
  is_default          boolean default false
  created_by          uuid             -- 사용자(추후 auth 연동)
  created_at          timestamptz
  updated_at          timestamptz

dashboard_widgets
  id                  uuid pk
  dashboard_id        uuid fk → dashboards.id (cascade)
  widget_type         text check in ('control_chart','trend','histogram','kpi_card','anomaly_list')
  title               text
  source_kind         text check in ('decision_function','spc_series','instance_property')  -- PRD-PF-B/F 출력 참조
  source_ref          jsonb            -- { functionId | seriesId, propertyId, filters }
  config              jsonb default '{}' -- 축 범위, 임계선, 색상 매핑 등 위젯별 옵션
  position            jsonb            -- { row, col, w, h } (그리드 슬롯)
  refresh_interval_s  integer default 30  -- 폴링 주기(M5에서 구독 전환 시 0으로 표시 가능)
  created_at          timestamptz
  updated_at          timestamptz

action_items
  id                  uuid pk
  partition_id        uuid fk → partitions.id            -- (PRD-PF-A 완료 시 ontology_id로 승격)
  source_function_id  uuid             -- PRD-PF-B constraints(kind=decision) 참조
  subject_instance_id uuid fk → instances.id  -- 판정 대상(예: 웨이퍼 인스턴스)
  verdict             text check in ('fail','warn','pass')  -- 원칙상 'pass'는 보드에 노출하지 않음(불통과/이상만)
  score               real             -- 결정함수 스코어(있는 경우)
  evidence            jsonb            -- 근거경로: 참조 데이터포인트·규칙명 등
  status              text check in ('pending','in_review','confirmed','dismissed') default 'pending'
  assigned_to         uuid             -- 담당 엔지니어(선택)
  resolved_by         uuid             -- 확정/기각한 사용자 (감사추적 필수 컬럼)
  resolved_at         timestamptz
  resolution_note     text             -- 확정/기각 사유(HITL 필수 입력)
  created_at          timestamptz
  updated_at          timestamptz

  -- 인덱스: (partition_id, status, verdict) 복합 인덱스 — "이상 웨이퍼만" 필터의 핵심 쿼리 경로
```

`action_items.status`는 명시적으로 자동 전이를 허용하지 않는다 — `pending → confirmed/dismissed` 전이는 반드시 `resolved_by` + `resolution_note`가 함께 쓰여야 커밋되도록 API 레벨에서 강제(완전자동 금지의 스키마 차원 구현).

### 5-3. UX — "이상 웨이퍼만 보이게"

- **기본 뷰 = 미처리 이상만.** 액션보드 진입 시 기본 필터는 `status=pending AND verdict IN ('fail','warn')`. "전체 보기" 토글은 있지만 기본값이 아니다.
- **정렬 기본값 = 스코어 내림차순(가장 심각한 것 위)**, 보조 정렬로 발생시각.
- **일괄 처리:** 체크박스 다중 선택 → "일괄 확정"/"일괄 기각" 버튼. 단, 일괄 처리도 `resolution_note` 1개(공통 사유)를 요구해 감사추적 공백을 만들지 않는다.
- **카드 vs 표 전환:** 소량(≤50)일 땐 카드형(HITL 컨펌카드 톤 계승), 대량일 땐 표형(밀도 우선) — 사용자가 전환 가능, 기본값은 건수 기준 자동 선택.
- **상태 배지 시각언어:** 파선 테두리 = AI/함수가 초안 판정(미확정), 실선 테두리 = 엔지니어 확정 — 기존 "특색"(원/점/파선/실선)을 액션 카드까지 일관 확장.
- **근거 클릭 → 캔버스 하이라이트:** 액션 카드의 `subject_instance_id` 클릭 시 메인 캔버스로 이동 없이 동일 세션에서 `highlightNodes` + `focusNode` 호출(모달/사이드시트 유지, 컨텍스트 전환 최소화).

### 5-4. 리스크

| 리스크 | 내용 | 완화책 |
|---|---|---|
| 차트 라이브러리 락인 | Recharts에 위젯 로직이 묶이면 후일 ECharts 전환 비용 큼 | 위젯 `config`를 라이브러리 중립 스키마(축·시리즈·임계선)로 저장, 렌더러만 갈아끼우는 어댑터 계층을 M2에서부터 분리 |
| 번들 증가 | 캔버스 중심 제품에 차트 라이브러리 추가가 메인 라우트를 무겁게 할 수 있음 | 대시보드/액션보드를 Next.js 라우트 분리 + 동적 임포트(`next/dynamic`)로 메인 캔버스 번들에 영향 0 유지(§2 지표) |
| 대량 항목 성능 | action_items 수만 건 시 목록 렌더·필터 지연 | M3는 서버사이드 페이지네이션 우선 적용, 가상 스크롤은 M6로 후행(선제 최적화 금지 원칙 준수) |
| 실시간 갱신 부하 | 폴링 주기 과다 시 DB 부하, 구독 전환 시 커넥션 관리 복잡도 | M2~M4는 위젯별 `refresh_interval_s`(기본 30초) 고정 폴링, M5에서 구독 전환은 별도 용량 테스트 후 단계적 롤아웃 |
| 확정의 감사추적 누락 | HITL 확정이 "그냥 버튼 클릭"이 되면 팔란티어 AIP식 승인 흐름의 신뢰가 무너짐 | §5-2 스키마 레벨에서 `resolved_by`+`resolution_note` 없는 상태 전이를 API가 거부하도록 강제 |
| 멀티테넌시 미완성 | PRD-PF-A 이전엔 `partition_id` 스코프가 사실상 워크스페이스 격리를 대신 못함(RLS deny-all 단일사용자) | 이 PRD의 신규 테이블도 PRD-PF-A 완료 전까지는 **단일 사용자 MVP 한계**를 그대로 상속함을 §8에 명시, 컬럼명만 승격 가능하게 설계 |

## 6. 수용 기준

- [ ] `dashboards`/`dashboard_widgets`/`action_items` 테이블이 마이그레이션되고 기존 `partitions`·`instances`와 FK로 연결된다.
- [ ] Recharts + shadcn Charts 컴포넌트가 도입되고, `--chart-1`~`--chart-5` CSS 변수가 라이트/다크 모드 모두 정의된다.
- [ ] 관리도 위젯이 PRD-PF-F 출력(시계열 + UCL/LCL)을 렌더하고, 이상점이 시각적으로 구분된다.
- [ ] 액션보드 기본 진입 화면이 `status=pending AND verdict IN ('fail','warn')`만 보여준다(전체 보기는 별도 토글).
- [ ] 모든 액션 카드/위젯에 근거경로(함수·데이터·시각)가 표기되고, 클릭 시 `highlightNodes`로 캔버스가 하이라이트된다.
- [ ] 액션 확정/기각 시 `resolved_by`+`resolution_note` 없이는 상태 전이가 API에서 거부된다.
- [ ] 뷰 빌더로 위젯 1개를 코드 없이 추가·저장할 수 있고, 사용자 테스트 기준 5분 이내 완료된다.
- [ ] 대시보드/액션보드 라우트가 코드 스플리팅되어 메인 캔버스(`app/page.tsx`) 초기 번들 크기가 증가하지 않는다.
- [ ] 일괄 확정/기각이 다중 선택 + 공통 사유 입력으로 동작한다.

## 7. 결론

지금 제품은 온톨로지를 "정교하게 저작"하는 데는 세계 수준이지만, 그 온톨로지 위에서 계산된 결정을 **사람이 볼 화면이 없다.** PRD-PF-B/F가 판정을 만들어도 이 PRD 없이는 API 응답에서 죽는다. 차트 라이브러리 도입은 더 이상 미룰 수 없는 인프라 부채이며, Recharts(shadcn 공식 경로)를 1차로 채택하는 것이 이 제품의 정체성(shadcn 톤·다크모드·접근성·재배치 우선)과 가장 정합적이다. 대시보드(모니터링)와 액션보드(처리 큐)를 분리해 "보는 것"과 "처리하는 것"의 UX 부하를 나누고, 뷰 빌더는 프리폼이 아닌 그리드+설정 패널로 구현 비용을 낮춘다. 완전자동 확정은 스키마 레벨에서부터 막아 HITL 북극성을 지킨다. 이 PRD가 끝나면 "센싱→결정→액션→writeback" 루프의 마지막 조각(액션 소비 화면)이 채워진다.

## 8. 열린 결정 / 불가능한 기능

**열린 결정 (팀 논의 필요):**

1. **차트 라이브러리 최종 확정.** 이 PRD의 권고는 Recharts(1차) + 필요시 관리도 한정 visx 이스케이프다. 최종 확정 전 실제 SPC 데이터 볼륨(웨이퍼당 포인트 수, 동시 표시 시계열 개수)을 PRD-PF-F 쪽과 맞춰 재검증 필요.
2. **ECharts 부분 도입 여부.** 대량 실시간(수만 포인트, 캔버스 필요) 시나리오가 실제로 발생하면 해당 위젯 한정으로 ECharts를 추가할지 — 지금은 보류.
3. **뷰 빌더의 커스텀 쿼리 허용 범위.** MVP는 사전정의 `source_kind`(decision_function/spc_series/instance_property)만 허용. 비FDE가 임의 필터 조건(예: "특정 속성값 범위")까지 노코드로 구성하게 할지는 UX 복잡도 대비 가치를 더 봐야 함.
4. **`partition_id` → `ontology_id` 마이그레이션 시점.** PRD-PF-A 완료 시점에 맞춰 일괄 전환할지, 이 PRD 릴리스 시점에 컬럼을 이중으로 둘지 결정 필요.
5. **일괄 처리의 사유 단일화가 감사추적 품질을 떨어뜨리는지.** 대량 확정 시 공통 사유 1개가 개별 판단을 흐릴 위험 — 처리 건수 상한(예: 일괄 처리 최대 50건)을 둘지 논의 필요.

**불가능한 기능 (이번 PRD 스코프 아님, 과대약속 금지):**

- **완전자동 확정/폐기 처리.** 결정함수의 판정은 항상 제안이며, 엔지니어 확정 없이 상태가 바뀌지 않는다.
- **진짜 실시간 스트리밍(초 단위, 센서 결합).** 현재 아키텍처는 배치/폴링 준실시간까지다. 센서 스트림 결합은 별도 이벤트 처리 인프라가 필요하며 이 PRD 밖이다.
- **완전 자유배치 프리폼 대시보드 캔버스.** 그리드 슬롯 방식만 지원. Figma급 자유배치는 스코프 아웃.
- **데이터 통합·ETL.** 위젯은 이미 연결된 데이터셋(PRD-PF-B/F 출력)만 소비한다. 신규 데이터소스 연동은 이 PRD의 일이 아니다.
- **완전한 멀티테넌시 격리.** PRD-PF-A 이전까지는 `partition_id` 스코프 수준의 논리적 분리만 제공하며, 진짜 워크스페이스 격리(RLS)는 보장하지 않는다.
- **Nelson Rules 8종 전체를 위젯 UI에서 자동 하이라이트.** 규칙 판정 자체는 PRD-PF-F(결정함수) 책임이며, 이 PRD는 그 출력을 "표시"만 한다 — 표시 위젯이 규칙을 재계산하지 않는다.
