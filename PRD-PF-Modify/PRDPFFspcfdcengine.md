# PRD-PF-F — SPC/FDC 통계 엔진 모듈: 공정 스펙관리, 온톨로지는 계산하지 않는다

> **모듈:** SPC(통계적 공정관리) · FDC(설비 이상탐지·분류) — 문제/워크스페이스 단위 토글
> **의존:** PRD-PF-B(kinetic Function이 통계엔진을 호출) · PRD-PF-D(공정 데이터셋 레지스트리) · PRD-PF-G(판정 결과 시각화·액션보드)
> **작성일:** 2026-07-12
> **원칙:** 비판적 옹호 · 기존 자산 재배치 우선 · shadcn/ui · 한국어 UI · HITL · 시각언어(원/점/파선/실선) 보존 · **역할 경계 최우선**

---

## 0. 한 줄 요지

제조 현장(특히 SAP/ERP가 없고 **공정 데이터만 있는** 다수 중소·중견 제조사)이 우리에게 원하는 것은 "그래프가 예뻐지는 것"이 아니라 **"이 웨이퍼/이 로트가 통과인지 아닌지, 왜 그런지 설명 가능한가"**다. 이 요구를 정직하게 풀려면 두 가지를 동시에 인정해야 한다. 첫째, **관리도 계산·공정능력지수·이상판정 룰은 통계학의 영역**이며 그래프 순회로 대체할 수 없다 — X-bar/R, Cp/Cpk, Nelson Rules는 수십 년간 검증된 수식과 부분군(subgroup) 이론 위에 서 있고, 우리가 재발명하거나 어설프게 흉내 낼 대상이 아니다. 둘째, 그럼에도 **온톨로지가 할 일은 분명히 있다** — "이 공정변수에 어떤 규칙을 적용할지", "이 스펙은 어떤 설비·제품·개정이력에 속하는지", "이 판정이 왜 나왔는지"를 조직하고, 재사용하고, 설명하고, 근거화(provenance)하는 것. 이 PRD는 **별도의 통계 엔진 모듈**을 신설하고, 온톨로지 kinetic Function(PRD-PF-B)이 그 엔진을 "호출"하는 얇고 명확한 경계선을 긋는다. 그래프는 계산기가 아니다 — 그래프는 계산이 왜 그 계산이어야 했는지를 기억하는 곳이다. SPC(제품 측정값)와 FDC(설비 센서)는 서로 다른 데이터·다른 판정 로직을 가지므로 별개 모듈로 분리하고, 공정데이터가 없는 워크스페이스(텍스트/ERP 중심)에서는 둘 다 끌 수 있는 **토글 아키텍처**로 시작한다.

## 1. 목적 (Purpose)

**왜 지금 SPC/FDC인가.** 현재 코드베이스 실태는 명확하다 — SPC/FDC 관련 로직 전무, 결정함수(kinetic Function) 엔진 없음(PRD-PF-B에서 도입 예정), 공정 데이터셋 레지스트리 없음(PRD-PF-D에서 도입 예정), 차트/통계 라이브러리 전무(recharts·d3·통계 패키지 전무). 즉 우리는 "온톨로지 구축 스튜디오"로서는 이미 스케치북→운영 전환·Git-for-data·Text2Cypher 소비까지 쌓았지만, **제조 도메인의 실제 첫 유스케이스(공정 스펙관리)에 필요한 계산 인프라가 통째로 비어 있다.**

사용자 요구를 재정리하면: (1) 어떤 제조업이든 "공정 스펙관리·공정 데이터"로 먼저 들어간다 — ERP/SAP가 있는 대기업보다 **공정 데이터만 있는 현장이 압도적으로 많다.** (2) 반대로 ERP/SAP만 있고 텍스트 문서 위주인 조직(구매·물류·정비 이력 등)에는 SPC/FDC가 무의미하거나 오히려 UI 잡음이므로 **꺼져야 한다.** 이 비대칭이 이 PRD의 존재 이유다 — SPC/FDC는 "코어 기능"이 아니라 **선택적으로 켜지는 도메인 모듈**이어야 한다.

**정직한 경계가 핵심인 이유.** 온톨로지 도구가 SPC/FDC 시장에 들어갈 때 가장 흔한 함정은 "그래프가 통계를 대체한다"는 과대약속이다. 이는 두 가지로 무너진다 — (a) 품질 엔지니어는 이미 Minitab·JMP·사내 SPC 시스템을 신뢰하며, 우리가 "그래프로 관리도를 계산한다"고 하면 즉시 신뢰를 잃는다. (b) Cp/Cpk 계산은 정규성 가정·부분군 크기·규격 재계산 등 통계적 엄밀성이 요구되는 영역이라, 그래프 순회 로직으로 흉내 내면 **틀린 판정을 자동화**하는 사고로 이어진다(제조 현장에서 이는 불량 유출·리콜로 직결). 따라서 이 PRD는 "온톨로지 스튜디오가 SPC/FDC 계산기를 만든다"가 아니라 **"온톨로지 스튜디오가 어떤 규칙을 어떤 변수에 적용할지 조직하고, 통계 엔진에 위임하고, 그 결과를 그래프 근거로 되받는다"**로 스코프를 못박는다. Function이 통계엔진을 호출·조직하는 것이지, Function이 통계엔진인 것은 아니다.

## 3. 기술 스택 (신규 통계·차트 도입 + 기존 재배치)

현재 우리 스택에는 통계·차트 라이브러리가 **전무**하다. 이 PRD는 그 공백을 처음으로 메우되, "온톨로지 쪽 재배치"와 "신규 통계 인프라 도입"을 분리해 과대 스코프를 막는다.

| 레이어 | 선택 | 상태 | 비고 |
|---|---|---|---|
| **통계 계산 코어** | 신규: Python 서비스(FastAPI) + `scipy`/`statsmodels`/`pyspc` 계열, 또는 JS `simple-statistics`+자체 관리도 룰 구현 | 🆕 도입 필요 | **결정 미확정(8절 열린결정) — Python 서비스 유력**. 이유: 통계 라이브러리 성숙도(scipy 생태계)·향후 다변량 FDC(PCA) 확장성. Node/Next.js 프로세스와 분리된 별도 서비스로 계산 신뢰성과 온톨로지 계층을 물리적으로도 분리. |
| **통계엔진 ↔ Function 통신** | 신규: 내부 REST/RPC(예: `/spc/evaluate`, `/fdc/classify`) | 🆕 도입 필요 | PRD-PF-B의 kinetic Function이 이 엔드포인트를 "호출 대상"으로 등록. Function 정의 자체에는 계산 로직 없음 — 파라미터(변수 ID·룰셋 ID·기간)만 전달. |
| **결과 저장** | 기존 Supabase(Drizzle) 확장 | 🔁 재배치 | 신규 테이블만 추가(5절 데이터모델), 기존 SoT·커밋 구조 그대로 사용. 별도 저장소 신설 금지. |
| **판정 결과 그래프 표현** | 기존 Neo4j 발행 파이프라인 | 🔁 재배치 | "판정 결과" 자체는 그래프 노드가 아니라 **그래프에 연결된 이벤트/근거**로 발행(6.2 데이터모델 참조). 그래프가 계산 결과를 저장하는 캐시가 아니라, 계산 결과를 설명하는 컨텍스트가 되도록 설계. |
| **차트/시각화(관리도·트렌드)** | 신규: `recharts` 또는 `visx` 채택 | 🆕 도입 필요 | 관리도(UCL/LCL 라인 포함 시계열), 공정능력 히스토그램은 그래프 캔버스(Cytoscape)로는 표현 불가 — **PRD-PF-G(결과 시각화)에서 별도 대시보드 뷰로 구현**, Cytoscape 캔버스와 명확히 분리(캔버스=관계, 대시보드=수치). |
| **AI 초안(룰셋 추천)** | 기존 AI SDK v6 | 🔁 재배치 | 신규 공정변수 등록 시 변수명·단위·데이터 특성을 보고 관리도 종류·룰셋 후보를 초안 — 기존 Critic/컨펌 카드 패턴 재사용, 확정은 엔지니어. |
| **데이터 소스** | PRD-PF-D 공정 데이터셋 레지스트리 | 🔁 의존 | 이 PRD는 데이터셋 자체를 관리하지 않음. 데이터셋 등록·스키마 매핑은 PRD-PF-D 책임, 이 PRD는 "등록된 데이터셋에서 어떤 컬럼이 SPC 변수/FDC 센서인지"만 매핑. |

**스택 정합 원칙.** 통계 엔진은 "온톨로지 스튜디오 제품"의 일부이지만 **아키텍처적으로 그래프 발행 경로(Supabase→Neo4j)와 별도 데이터 경로**를 가진다. 이는 재배치가 아니라 신규 계층이며, 그 이유를 8절에서 열린결정으로 다시 명시한다.

## 4. 방향 (마일스톤 로드맵)

> 순서 근거: **역할 경계 문서화·인터페이스 확정(M1) → SPC 최소 기능(M2) → 토글 아키텍처(M3) → FDC 최소 기능(M4) → AI 초안·설명가능성(M5)**. SPC를 FDC보다 먼저 놓은 이유: SPC(제품 측정값)가 데이터 구조가 단순하고(단변량 시계열+스펙) 표준 룰이 산업 전반에 확립돼 있어 첫 신뢰 구축에 유리한 반면, FDC(설비 센서 트레이스)는 다변량·설비별 특이성이 커서 정확도 리스크가 크다.

### M1 — 역할 경계 인터페이스 확정 (통계엔진 ↔ 온톨로지)
- 통계엔진 호출 인터페이스(`/spc/evaluate`, `/fdc/classify`) 스펙 확정: 입력(변수 ID·측정값 배열·스펙/룰셋 파라미터), 출력(판정 결과·근거 수치·적용 룰 ID).
- kinetic Function(PRD-PF-B) 쪽에서 "이 Function은 통계엔진을 호출한다"를 선언하는 Function 타입 신설(계산 로직 없음, 호출 파라미터 매핑만).
- **가드레일:** 코드 리뷰 체크리스트에 "그래프/Cypher 쿼리 내 통계 계산식 금지" 명시 — 이 PRD 전체의 최우선 원칙을 기술 부채로 새지 않게 고정.

### M2 — SPC 모듈 최소 기능
- 관리도 5종(X-bar/R, I-MR, p/np/c/u) 계산 엔드포인트.
- 공정변수별 스펙(USL/LSL/target) 및 관리한계(UCL/LCL) 등록·저장(5절 데이터모델).
- Cp/Cpk/Pp/Ppk 계산.
- 이상판정: Western Electric 4룰 우선 구현(가장 널리 쓰이고 오탐률이 상대적으로 낮음), Nelson 8룰은 옵션으로 켤 수 있게(과탐지 리스크 안내 문구 포함).
- 온톨로지 kinetic Function에서 "변수 X에 관리도 Y, 룰셋 Z 적용" 매핑을 저작·저장.

### M3 — 토글 아키텍처
- 문제/워크스페이스 설정에 SPC·FDC 모듈 on/off 스위치. off 시 관련 메뉴·대시보드·Function 타입 자체가 UI에서 숨김(플러그인 로딩 방식, 코드는 남되 비활성).
- on 전환 시 온보딩: "공정 데이터셋을 먼저 PRD-PF-D에 등록하세요" 가드.
- off 워크스페이스는 기존 Text2Cypher·RAG·정비 트러블슈팅류 소비 경로 그대로 — SPC/FDC 부재가 다른 기능을 막지 않음을 회귀 테스트로 보장.

### M4 — FDC 모듈 최소 기능
- 설비 센서 트레이스(온도·압력·전류 등 시계열) 등록 매핑(PRD-PF-D 의존).
- 탐지 방식: (a) 임계값 기반(정적/동적 상하한), (b) 단변량 트렌드 기반(급변·드리프트 탐지) — 두 가지로 시작, **다변량(PCA 등) 방식은 M4 스코프 아님**(8절 불가기능).
- SPC와의 명확한 UI/개념 분리: "제품 측정값(metrology) = SPC", "설비 센서 = FDC" — 같은 화면에서 혼용 표기 금지, 라벨링 규칙을 디자인 시스템에 반영.

### M5 — AI 초안 & 설명가능성
- 신규 공정변수 등록 시 AI가 변수 특성(연속/이산, 데이터 패턴)을 보고 관리도 종류·룰셋 후보를 초안 제시 → 엔지니어 컨펌(HITL, 기존 Critic/컨펌 카드 패턴 재사용).
- 판정 결과 카드에 "왜 이상으로 판정됐는지"(적용 룰·초과 σ·연속 위반 포인트) 근거를 그래프 provenance 체인과 연결해 노출(PRD-PF-G와 연동).

## 5. 방법론 (통계엔진 ↔ 온톨로지 경계 · 데이터모델 · 리스크)

### 5.1 경계 원칙 (반복 명시)

**그래프는 계산하지 않는다.** 관리도 UCL/LCL 산정, 부분군 통계, Cp/Cpk, Nelson/Western Electric 룰 판정은 전부 통계 엔진(Python 서비스)이 수행한다. 온톨로지·그래프의 역할은 정확히 세 가지로 한정한다.

1. **의미(시멘틱) 조직**: 공정변수·웨이퍼·로트·설비·스펙 간의 관계를 그래프로 표현(예: `공정변수 --속함--> 설비`, `로트 --측정됨--> 공정변수`, `스펙 --버전--> 개정이력`).
2. **규칙 배치(kinetic Function)**: "이 변수에는 이 관리도, 이 룰셋을 쓴다"는 결정을 Function으로 선언·재사용·버전관리. Function은 통계엔진 호출의 **오케스트레이터**이지 계산기가 아니다.
3. **설명·근거화**: 판정 결과가 나오면 "어떤 규칙, 어떤 데이터, 어떤 스펙 버전에 근거했는가"를 그래프 provenance로 연결해 감사 가능하게 만든다. 계산 결과 수치 자체의 정확성은 통계엔진이 책임지고, 그 결과의 **맥락과 이력**은 온톨로지가 책임진다.

이 경계는 협상 대상이 아니다 — 위반(그래프 쿼리에 통계 계산 임베딩)은 제품 신뢰성 리스크이자 이 PRD의 실패 조건으로 간주한다.

### 5.2 데이터모델 (신규)

Supabase(Drizzle) 신규 테이블 초안:

| 테이블 | 역할 | 핵심 컬럼 |
|---|---|---|
| `spec_limits` | 공정변수별 스펙 | `variable_id`, `usl`, `lsl`, `target`, `revision`, `effective_from` |
| `control_limits` | 공정변수별 관리한계(통계엔진 산출 결과 저장) | `variable_id`, `chart_type`(X-bar/R 등), `ucl`, `lcl`, `centerline`, `subgroup_size`, `computed_at`, `computed_by`(통계엔진 버전) |
| `spc_rulesets` | 적용 룰셋 정의(Western Electric/Nelson 중 어떤 룰 on/off) | `ruleset_id`, `rules_enabled`(배열), `owner_function_id` |
| `spc_runs` | 개별 판정 실행 결과 | `run_id`, `variable_id`, `lot_id`, `batch_id`, `chart_type`, `verdict`(pass/fail/warn), `violated_rules`, `evaluated_at` |
| `fdc_traces` | FDC 센서 트레이스 판정 결과 | `trace_id`, `equipment_id`, `sensor_id`, `detection_method`, `fault_flag`, `evaluated_at` |

그래프(Neo4j) 발행 원칙: 위 테이블의 **원본 수치는 그래프로 복제하지 않는다.** 대신 `spc_runs`/`fdc_traces`의 판정 결과 요약(이상/정상 플래그 + 근거 링크)만 그래프 노드/관계로 발행해, 그래프가 "무엇이 이상이었고 왜인지"를 가리키는 인덱스 역할을 하게 한다. 원본 시계열·상세 계산은 Supabase(SoT)와 통계엔진 로그에만 존재.

### 5.3 리스크

- **R1. 통계 정확성(관리한계 산정·부분군 이론).** 부분군 크기 선택, 정규성 가정 위반, 초기 관리한계 산정 시점(공정 안정화 이전 데이터 오염) 등은 통계학적으로 까다로운 문제다. **완화:** 자체 알고리즘 재발명 금지, 검증된 라이브러리(scipy/statsmodels 또는 산업 표준 SPC 라이브러리) 사용, 관리한계 재계산 시점을 명시적으로 엔지니어가 트리거(자동 재계산 금지 — 공정 변경을 관리한계가 조용히 흡수하면 이상 탐지력을 잃는다).
- **R2. 준실시간 한계.** 현재 배치 아키텍처상 완전 실시간 스트리밍 판정은 불가. **완화:** 로트/배치 단위 처리로 스코프를 명확히 한정하고 UI에 "준실시간(배치 단위)"임을 항상 표기, "실시간 SPC"라는 과대 마케팅 문구 금지.
- **R3. 통계엔진 인프라 선택 미확정.** Python 서비스 vs JS 통계 라이브러리 — 8절 열린결정으로 이관, M1에서 인터페이스만 확정하고 구현체는 M2 착수 직전 재검토.
- **R4. 도메인 파라미터 초기설정 부담.** 스펙·룰셋을 엔지니어가 처음부터 전부 입력해야 하면 온보딩 마찰이 크다. **완화:** M5 AI 초안 + 기존 Critic/컨펌 패턴 재사용, 단 최종 확정은 항상 엔지니어(HITL, 완전자동 금지 — 오판정의 책임소재를 흐리지 않기 위함).
- **R5. SPC/FDC 개념 혼용.** 제품 측정값과 설비 센서를 사용자가 같은 것으로 오인하면 잘못된 모듈에 데이터를 넣는다. **완화:** M4에서 명시한 라벨링 규칙, 온보딩 시 "측정값이냐 센서냐" 분기 질문.
- **R6. 그래프-통계엔진 동기화 지연/불일치.** 통계엔진 재계산과 그래프 발행 사이 지연으로 판정 결과가 그래프상 최신이 아닐 수 있음. **완화:** `computed_at` 타임스탬프를 그래프 발행 노드에 필수 포함, PRD-PF-G 대시보드에서 "최신 계산 기준 시각" 항상 노출.

- [ ] **경계:** 그래프/Cypher 쿼리·Function 정의 코드에 통계 계산식(표준편차·관리한계 산식 등)이 하드코딩된 인스턴스 0건(코드 리뷰 게이트 통과).

- [ ] **이상판정:** Western Electric 4룰 기본 적용, Nelson 8룰 옵션 토글 가능. 판정 결과에 위반 룰 ID가 기록된다.
- [ ] **토글:** 워크스페이스 설정에서 SPC/FDC를 각각 독립적으로 켜고 끌 수 있으며, off 상태에서 기존 Text2Cypher/RAG 소비 경로가 회귀 없이 동작한다.
- [ ] **FDC:** 임계값 기반·단변량 트렌드 기반 탐지 각 1종 이상 동작, SPC와 UI/라벨상 명확히 분리(제품 측정값 vs 설비 센서).
- [ ] **준실시간 표기:** 모든 판정 결과 UI에 "배치/로트 단위, 실시간 아님"이 표기된다.
- [ ] **AI 초안:** 신규 공정변수 등록 시 관리도·룰셋 초안이 제시되고, 엔지니어 컨펌 없이는 확정되지 않는다(HITL).
- [ ] **근거화:** 이상 판정 1건마다 적용 규칙·데이터 출처·스펙 버전을 잇는 provenance 카드가 노출된다.

- [ ] **공통:** shadcn/ui·한국어 UI, 기존 배지/카드 패턴 준수, lint·빌드·기존 테스트 회귀 0.

## 7. 결론 (비판적 옹호 요약)

SPC/FDC는 온톨로지 스튜디오가 "그럴듯한 그래프 데모"에서 "제조 현장이 실제로 돈을 내는 도구"로 넘어가는 관문이다. 공정 데이터만 있는 현장이 SAP를 갖춘 현장보다 많다는 사용자 판단은 시장 크기 논리로 타당하다. 그러나 이 기회를 잡는 방식이 "그래프로 관리도를 계산한다"는 과대약속이어서는 안 된다 — 통계적 공정관리는 수십 년간 검증된 수학 위에 서 있고, 우리가 그 위에 얹을 가치는 계산이 아니라 **조직·재사용·설명·근거화**다. 이 PRD는 통계 엔진을 명확히 분리된 모듈로 신설하고, kinetic Function을 그 호출자로 한정하며, 그래프를 "판정 결과가 왜 나왔는지"를 가리키는 인덱스로 설계함으로써 이 경계를 아키텍처 수준에서 강제한다. SPC를 FDC보다 먼저, 단변량을 다변량보다 먼저, 배치 판정을 실시간보다 먼저 놓는 순서는 모두 같은 원칙에서 나온다 — **검증 가능한 것부터, 정직하게.** 텍스트/ERP 중심 조직을 위한 토글은 이 모듈이 "제조 특화 기능"이지 "필수 코어"가 아님을 제품 구조로도 증명한다. 완전자동 판정과 실시간 마케팅은 명시적으로 금지한다 — 오판정의 책임소재를 흐리는 순간 이 모듈은 제조 현장에서 신뢰를 잃는다.

## 8. 열린 결정 / 불가능한 기능

**열린 결정 (다음 단계에서 확정 필요)**

1. **통계엔진 구현체: Python 서비스 vs JS 통계 라이브러리.** Python(scipy/statsmodels) 쪽이 통계 생태계 성숙도·향후 다변량 FDC 확장성에서 유리하지만, 별도 서비스·배포·운영 부담(현재 스택은 Next.js/Node 단일 경로)이 추가된다. JS 라이브러리(`simple-statistics` 등)는 배포는 간단하지만 산업 표준 SPC 룰 구현체가 성숙하지 않아 자체 구현 리스크가 크다. **M1 완료 시점에 인터페이스만 확정하고, M2 착수 직전 PoC로 재결정.**
2. **부분군(subgroup) 크기 및 관리한계 재계산 트리거 정책.** 자동 재계산 시 공정 변경을 관리한계가 흡수해 이상 탐지력이 약화될 위험 — 엔지니어 수동 트리거로 할지, 주기적 검토 알림으로 할지 미확정.
3. **Nelson 8룰 기본값(on/off).** 오탐지(false positive) 증가 우려로 기본 off, 옵션 on을 제안했으나 최종 UX 문구·기본값은 PRD-PF-G와 함께 재검토 필요.
4. **PRD-PF-D(데이터셋 레지스트리)와의 정확한 인터페이스 계약.** 이 PRD는 "등록된 데이터셋에서 컬럼을 SPC 변수/FDC 센서로 매핑"만 가정했으나, PRD-PF-D의 스키마가 확정되지 않아 매핑 API 세부는 PRD-PF-D 확정 후 조정 필요.

**불가능한 기능 (이번 스코프에서 명시적으로 제외)**

1. **완전 실시간 스트리밍 SPC/FDC.** 현 배치 아키텍처상 불가. 로트/배치 단위 준실시간까지만 지원.
2. **다변량 FDC(PCA 등 통계적 차원축소 기반 이상탐지).** 반도체 업계 표준 기법이지만 이 PRD 스코프 밖 — 단변량 임계·트렌드 탐지로 시작 후 별도 PRD로 확장 검토.
3. **완전자동 판정·조치.** 이상 판정 후 자동으로 설비를 정지시키거나 로트를 자동 폐기하는 등의 자동 조치는 금지 — 모든 조치는 사람의 확인을 거친다(HITL 불변식, 완전자동 절대 금지).
4. **정규성 위반·비정상 공정에 대한 자동 통계 기법 전환.** 데이터가 정규분포를 따르지 않을 때 자동으로 다른 관리도/변환을 적용하는 지능형 전환은 이번 스코프 밖 — 엔지니어가 수동으로 관리도 종류를 선택.
5. **그래프 쿼리를 통한 통계 계산.** Cypher/그래프 순회로 관리한계나 Cpk를 계산하는 기능은 설계상 영구 금지(경계 원칙 위반).
