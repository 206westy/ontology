# PRD-PF 시리즈 — 온톨로지 스튜디오 → "문제해결 플랫폼" 확장 (인덱스 · 로드맵 · 의존성 · 결정사항)

> 작성: 2026-07-12 · 온톨로지 스튜디오 플랫폼 확장 마스터 인덱스

> 네이밍: **플랫폼(Platform) 시리즈 = PRD-PF-A ~ I**. 기존 제품 PRD-A~N(레거시)과 **구분되는 새 시리즈**. 이 인덱스가 9개 PRD의 순서·의존·미결 결정을 총괄한다.

## 0. 이 시리즈가 만드는 것 (한 화면)

지금: **단일 페이지에서 온톨로지만 그리는 에디터.**
목표: **비FDE 도메인 전문가가, 문제를 정의하고 정제된 데이터를 연결하면, AI가 온톨로지·결정함수·액션보드를 초안으로 지어 주고, 사람은 단계마다 확정만 하며, 그 온톨로지는 다음 문제에서 재사용·분기되며 복리로 자라는 플랫폼.**

전체 워크플로우(멀티페이지, 단계별 엔지니어 확정):

```
[문제정의] → [데이터 연결·재사용] → [온톨로지 구축] → [키네틱(함수)] → [SPC/FDC(토글)] → [대시보드·액션보드] → [AIP·자동화]
   PF-C          PF-D                  PF-B(레이어)      PF-B/E          PF-F              PF-G                 PF-H/I
   └──────────────── 전 단계에 AI 코파일럿 부착: PF-E ────────────────┘
   └──────────────── 전 기능의 토대: 멀티온톨로지·테넌시 PF-A ─────────┘
```

## 1. PRD 목록 (A ~ I)

| PRD | 제목 | 한 줄 | 의존 | 핵심 신규 |
|---|---|---|---|---|
| **A** | 기초공사 — 멀티 온톨로지·워크스페이스·테넌시·RLS | 단일 전역 그래프 → 다중 온톨로지 SaaS 토대 | 없음(최선행) | `workspaces`·`ontologies`·`memberships` + 전 테이블 `ontology_id` 소급 + RLS |
| **B** | 온톨로지 레이어 재정의 — Semantic / Kinetic(=Function) / Dynamic 처리 | 라벨뿐인 kinetic을 **실행되는 결정 함수**로 승격, dynamic 레이어 분해 | A | `functions`·`decision_results`(권장) or `constraints.kind=decision` |
| **C** | 문제정의 진입점 + 다단계 워크플로우 셸(멀티페이지) | 단일 캔버스 → 문제 최상위 스코프 + 단계별 페이지·확정 게이트 | A | `problems`·`problem_ontology_links` + App Router 스텝퍼 |
| **D** | 데이터 연결 & 데이터셋 재사용 | 매번 재파싱 → 한 번 연결해 여러 문제가 공유 | A·C | `datasources`·`datasets`·`dataset_columns`·`problem_datasets` |
| **E** | AI 코파일럿(전 단계 부착) | 데이터 충분성 진단·결측 추천·문제기반 함수 추천 | B·C·D | 배선(orchestration), 충분성/함수추천 2개 신규 툴콜 |
| **F** | SPC/FDC 통계 엔진(토글) | 공정 스펙관리 — 통계는 엔진, 온톨로지는 조직·설명 | B·D·G | `spec_limits`·`control_limits`·`spc_rulesets`·`spc_runs`·`fdc_traces` + 토글 |
| **G** | 액션보드 & 대시보드(뷰 빌더 + 차트) | 이상 웨이퍼만 처리 큐로, 관리도·KPI 대시보드 | B·F·A | `dashboards`·`dashboard_widgets`·`action_items` + 차트 라이브러리 |
| **H** | AIP — 답변엔진·진단형 RAG·제안형 에이전트 | 감사가능 모델 위 근거경로 답변, 에이전트는 제안만 | B·G·A | 구획요약·전역/국소 검색·`rag/entrypoint` 완성 |
| **I** | 자동화·트리거·상태 라이프사이클(="다이나믹"의 실체) | 이벤트→함수→액션 제안, 상태머신, 준실시간 | B·F·G | `triggers`·`automation_runs`·`object_state_defs`·`instance_state_log` |

## 2. 빌드 순서 & 의존성 (권장 시퀀싱)

```
Phase 0 · 기초        A ─────────────────────────────┐  (없으면 SaaS·다중 온톨로지 불가)
                                                      ↓
Phase 1 · 재구성      B(레이어/Function) ── C(문제·멀티페이지 셸)   (병행 가능)
                                    │            │
Phase 2 · 골격                     ├─ D(데이터) ─┤
                                   └─ E(AI 코파일럿, B·C·D에 얹힘)
                                                      ↓
Phase 3 · 버티컬      F(SPC/FDC) ── G(보드)          (공정 데이터 우선 유스케이스)
                                                      ↓
Phase 4 · 지능·자동화 H(AIP) ── I(자동화·라이프사이클)
```

- **Phase 0(A)는 미룰 수 없다.** 멀티 온톨로지·테넌시가 없으면 "문제마다 온톨로지가 쌓인다"는 비전 자체가 성립 안 함. 가장 크고 아픈 소급 마이그레이션.
- **Phase 1의 B가 지적 핵심.** 전편 결론(kinetic=실행 함수)을 코드로. B가 서면 F·G·H·I가 의미를 갖는다.
- **Phase 3부터 "당신이 바로 해보고 싶은" SPC 수직 슬라이스**가 나온다(F→G). 단 B·D 선행 필요.

> **대안 경로(데모 우선):** 만약 투자·내부설득용으로 SPC 데모를 빨리 봐야 하면, A를 "온톨로지 1개 하드코딩"으로 임시 우회하고 B(축소)+D+F+G만 얇게 관통하는 **수직 슬라이스**를 먼저 만들 수 있다. 단 이건 기술부채이며 나중에 A로 정식화해야 함(§4 결정).

## 3. 온톨로지 스튜디오 "특색" 보존 원칙 (전 PRD 공통)

확장하되 정체성은 지킨다: (1) 시각 언어(클래스=원·인스턴스=점·is-a=파선·관계=실선) — 함수·상태도 이 언어에 녹인다, (2) HITL 컨펌 카드 — 모든 AI 산출은 초안+사람 확정, (3) Critic — 품질 수호자로 전 단계 검수, (4) 온톨로지의 Git(commits/branches/merge_requests) — 문제별 재사용·확장·분기의 엔진, (5) 자연어+컨펌 — 팔란티어의 코드 저작 대비 우리의 차별점.

각 PRD 8번 섹션에서 올라온 미결 중, **당신 결정이 시리즈 구조를 바꾸는** 상위 항목:

| D2 | **Kinetic 함수 데이터모델** | (A) 별도 `functions`+`decision_results` 테이블 / (B) `constraints.kind=decision` 확장 | ✅ **(A) 별도 테이블 확정** |

| D4 | **SPC 통계 엔진 구현체** | (A) Python 통계 서비스(scipy/statsmodels) / (B) JS 라이브러리 | ⏳ PoC 후(F의 M1) |

## 5. 플랫폼 레벨 "안되는 것" (전 PRD 관통, 정직)

1. **완전자동 금지** — 팔란티어 AIP도 "제안+사람 검토". 우리도 자동 *초안*까지, 확정은 사람. (전 PRD 불변식)
2. **온톨로지가 SPC/통계를 계산하지 않는다** — 통계는 엔진, 온톨로지는 규칙 조직·설명·근거화. (PF-F 경계)
3. **제로샷 범용 불가** — 버티컬 템플릿(SPC/FDC/정비) 축적으로 점진 확장. (PF-E 성공조건)
4. **데이터 통합 정면승부 회피** — 정제된 데이터(CSV/DB뷰/파케이) 전제, 무거운 ETL·SAP 직접연동 아웃. (PF-D)
5. **실시간은 준실시간부터** — 현 배치 아키텍처상 로트/배치·폴링. 초저지연 스트림은 별도 인프라. (PF-F·I)
6. **멀티테넌시는 선행 기초공사** — 현재 단일사용자 MVP. SaaS 전 필수. (PF-A)
7. **에이전트 자율실행·소비표면 writeback 금지** — 읽기전용, 모델 변경은 생성 흐름으로. (PF-H)

## 6. 결론

9개 PRD는 하나의 이야기다: **A(토대) 위에 B(결정 함수로 온톨로지를 "운영"으로) 를 세우고, C·D·E(문제-우선 워크플로우 + 데이터 재사용 + 전단계 AI)로 비FDE가 쓰게 만들고, F·G(공정 SPC/FDC + 액션보드)로 첫 버티컬 가치를 내고, H·I(AIP 답변·제안형 자동화)로 운영 루프를 닫는다.** 각 PRD는 "기존 자산 재배치 우선"을 지켰고(신규 인프라 최소), 6개 정직 경계를 전부 반영했다. 팔란티어 풀스택을 흉내 내지 않고, **팔란티어의 ③④⑥을 문제-우선으로 압축해 비FDE 셀프서비스로** — 그리고 "문제마다 온톨로지가 복리로 자라는" 우리 고유 포지션으로 간다.
