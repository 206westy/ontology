# Ontology Studio — PRD-L: 구성요소 간소화 & 확신-초안 확정 (Element Simplification & Confident-Draft Confirmation)

> **문서 성격**: 목적 · 기능 · UX 설명 중심. 코드·구현·스키마는 제외(구현은 Claude Code). 기존 요소를 **삭제가 아니라 통합·은닉·자동화**하는 관점 — 컴포넌트명·파일:줄번호는 마이그레이션 기준점으로만 인용.
> **버전**: L-1.0 · 2026-07-07
> **선행/기반**: PRD-I(공통 ConfirmCard·Guided 여정·역할 토큰) 완료, PRD-K(스케일 업·체크박스 부분 반영·검토 표면·배치 확정 UI) 완료 상태를 전제로 그 위에 얹는다. PRD-K가 여정의 **물리적 사용성**(크기·발견성·피드백)을 해결했다면, 본 PRD는 그 위에서 **개념적 복잡도**(사용자가 내려야 하는 판단의 수·종류)를 줄인다.
> **분석 방법**: 온톨로지 7요소가 스키마·AI 프롬프트·스토어·UI·검증에 어떻게 퍼져 소비되는지 2관점 코드 전수 추적(2026-07-07). 각 요소의 토큰 비용·중복·애매성 신호를 파일:줄번호 근거로 확보.

## 0. 목적

**한 줄 요약**: 온톨로지 구성요소는 강력하지만, 지금은 **AI가 판단해도 될 구분을 사용자에게 팝오버에서 하나씩 결정하게** 만들고(체감 복잡도↑), **전략 생성에 쓰이지 않는 분류에 LLM 토큰을 태우며**(추출 비용↑), **같은 개념을 두 요소로 중복 구현**(공리 vs 제약)한다. 요소를 얇게 만드는 게 목표가 아니다 — **① 사용자 판단은 줄이고, ② 기계용 구조 풍부함은 지키고, ③ 확정은 신뢰도로 트리아지(고신뢰 자동수락·저신뢰만 표면화)**하여, "구체적인 온톨로지"와 "쉬운 확정"을 동시에 얻는다.

**핵심 전제 — 두 종류의 복잡도를 분리한다**:

| 종류 | 정의 | 처리 원칙 |
|---|---|---|
| **(A) 사용자 결정 복잡도** | 사람이 내려야 하는 *판단*의 수·종류 | **순수 비용.** 검색·전략에 도움 안 되면서 판단만 시키는 구분은 통합·은닉·AI 위임 |
| **(B) 기계용 구조 풍부함** | 그래프에 담긴 *타입 디테일*(클래스·타입 관계·domain/range·속성) | **자산.** 상용 LLM GraphRAG + 노드 벡터서치 + 문서 답변형 전략 생성기에서 검색 grounding을 높임 → 유지·강화 |

**대상 사용자**: 온톨로지·그래프 지식이 없는 도메인 전문가(CLAUDE.md 전제). "이걸 클래스로 넣을지 인스턴스로 넣을지"를 스스로 판단하도록 요구하면 안 된다 — AI가 확신 초안을 제시하고, 사용자는 원탭으로 수락·수정만 한다.

**하류 소비처(설계 제약)**: 완성된 온톨로지는 **상용 LLM 기반 GraphRAG**(text-to-cypher를 LangGraph 노드로) + **노드 벡터서치** + **문서 기반 답변**을 결합한 전략 생성기에 즉시 투입된다. 따라서:
- 관계 카테고리는 text-to-cypher가 "증상/원인/조치" 계열을 필터할 수 있을 만큼만 있으면 되고, 표면 동의어 recall은 **벡터 인덱스가 상당 부분 대체**한다(Neo4j 벡터 인덱스 기존 존재).
- 관계 **이름**(`type`, 자유 문자열)은 자연어 의미를 이미 운반하므로 LLM-read 단계에서 대부분의 뜻이 복원된다.

**성공의 정의**:
- 초안 확정 시 사용자가 내리는 **평균 판단 수**가 유의하게 감소(고신뢰 항목은 기본 수락, 저신뢰만 검토).
- "제약조건"을 만들 때 사용자가 마주하는 개념이 **하나**다(공리/제약 이중 시스템 소멸).
- parse 프롬프트에서 관계 분류에 드는 토큰·오분류율이 감소(5분류→3레이어).
- 클래스/인스턴스 결정이 **AI 확신 초안 + 원탭 전환**으로 이뤄지고, 사용자가 개념을 학습하지 않아도 완주한다.
- 위 변경으로 **온톨로지의 구조적 구체성(B)은 유지 또는 향상**된다(요소를 지우지 않음).

7요소가 코드 전반에 어떻게 소비되는지 추적한 결과, **AI가 대화에서 실제로 "판단"하는 결정은 사실상 2가지(클래스 vs 인스턴스, 관계 카테고리)뿐**이고, 나머지는 구현 아티팩트이거나 중복이다.

### 보존할 강점 (건드리지 않음)

- **클래스/인스턴스 요소 자체**(TBox/ABox 분리 — GraphRAG cypher·전략 walk의 근간).
- **관계 이름(`type`) 자유 문자열** — 도메인 어휘를 손실 없이 담는 층. 카테고리 변경과 무관하게 유지.
- **출처·확신도(Provenance/Attribution)**, **구획(Partition)** — 저비용·비-AI 요소. 현행 유지.
- **PRD-I/K 확정 UI 자산**(ConfirmCard·체크박스·검토 표면·스케일) — 재활용, 재작성 금지.

## 2. 변경 목표 (6개 축)

> 원칙: **삭제할 것은 오직 — 사용자에게 판단을 시키면서 검색·전략엔 도움 안 되는 구분뿐.** 나머지는 통합·은닉·AI 위임.

### L1. 공리 + 제약 → 단일 "규칙(Rule)" 개념으로 통합

- **문제**(L-D1): 한 개념("지켜야 할 규칙")이 두 요소·두 UI·두 저장소로 갈라져, 사용자는 "제약조건"이 때론 강제되고 때론 아무것도 안 하는지 알 수 없다.
- **목표**: 사용자·AI에게 노출되는 규칙은 **하나의 모델**. 타입은 기존 제약의 유형 체계(cardinality/disjoint/domain_range/property_value)를 정본으로 삼고, free-text 공리는 **"미강제 설명 메모"로 명시적으로 강등**하거나 규칙 모델의 한 종류로 흡수.
- **UX**: 규칙 생성 진입점을 하나로. 각 규칙에 **"강제됨/설명용" 상태를 평문으로 표기**(사용자가 "이 규칙이 실제로 막아주는지"를 즉시 앎). 거버넌스 AI 제안(`suggest-governance`)도 단일 모델로 라우팅.
- **비고**: 데이터 마이그레이션은 구현 단계에서 `axioms`→`constraints` 흡수 또는 통합 테이블로 결정. 기존 데이터 무손실 보존.

### L2. 관계 카테고리 5 → 3 레이어 (시멘틱 / 다이나믹 / 키네틱)

- **문제**(L-D2, L-D3): 5분류는 프롬프트 토큰이 크고 diagnostic/procedural 혼동이 최다 오류인데, 앱 내부에선 2분류로만 소비된다. 전략 생성기가 필요로 하는 것은 "지형/원인/조치"를 가를 수 있는 **최소 축**이다.
- **목표**: 5분류를 전략 walk에 정확히 대응하는 **3레이어**로 재편.

  | 레이어 | 뜻(전략 walk 대응) | 흡수하는 기존 카테고리 |
  |---|---|---|
  | **시멘틱(semantic)** | "무엇인가 / 지형" — 구조·구성·서술 | structural + descriptive |
  | **다이나믹(dynamic)** | "무엇이 원인·변화인가" — 인과·진단 | causal + diagnostic |
  | **키네틱(kinetic)** | "무엇을 실행하는가" — 정해진 조치·절차 | procedural |

- **효과**: 3버킷은 MECE라 LLM이 5개보다 안정적으로 라벨링(diagnostic/procedural 혼동의 상당수가 "다이나믹 vs 키네틱" 단일 경계로 축소). 프롬프트에서 5종 정의·전용 구분 문단·CSV 반복 축소. `categoryConfidence`는 **삭제가 아니라 L6 트리아지 신호로 용도 전환**.
- **전략 활용**: text-to-cypher는 `WHERE r.layer IN ('dynamic')`로 증상·원인 계열만, `'kinetic'`으로 조치 계열만 필터. 표면 동의어 recall은 노드/엣지 벡터서치가 보완.
- **결정 필요**: `diagnostic`을 다이나믹에 흡수(권고)할지 독립 유지할지는 확정 전 사용자 승인 대상(§4 미결).

### L3. 관계유형 + 관계 → 사용자·AI에게 단일 "관계 추가" 액션

- **문제**(L-D4): LLM·사용자 어느 쪽도 RelationType과 Edge를 구분하지 않는데, assist 액션·plan 로직·일부 UI(`ConstraintsPanel`의 RelationType 원시 드롭다운)가 둘을 노출한다.
- **목표**: **인터랙션·AI 액션 계층에서 관계는 하나.** `RelationPopover`가 이미 하는 "이름 입력 → 유형 자동 생성 + 엣지 생성" 패턴을 표준으로 삼고, assist 액션도 단일 "관계 추가"로 합친다. `ConstraintsPanel`의 관계 선택은 **원시 RelationType id가 아니라 사용자가 만든 관계/쌍**을 고르게.
- **비고**: **데이터모델의 TBox/ABox 분리(RelationType 테이블 유지)는 보존** — GraphRAG 스키마 쿼리·domain/range에 유용. "UX 통합"이지 "스키마 통합"이 아님.

### L4. 클래스 vs 인스턴스 — 확신-초안 + 원탭 전환 확정

- **문제**(L-D5): 이 결정은 제거 불가(데이터모델·RAG 근간)이나, 지금은 사용자가 개념을 이해해 판단하도록 요구된다. 예: *"내가 키우는 호랑이의 이름은 범이야"* — 범이는 클래스가 아니라 **호랑이 클래스의 인스턴스**여야 한다. AI가 이를 판정해 초안으로 제시하되, 확정 UI가 약하다(작은 팝오버 토글).
- **목표**:
  1. **AI가 확신을 갖고 커밋** — 애매 회피("헷갈리면 인스턴스" 기본값)로 대부분을 "그냥 수락" 원탭이 되게.
  2. **전환을 1급 어포던스로** — PRD-K가 스케일·발견성을 이미 올렸으니, 그 위에 "이건 **카테고리(클래스)**인가요, **실제 하나(인스턴스)**인가요?" 평문 마이크로카피 + 원탭 토글을 확정 카드에 상시 노출.
  3. 흩어진 3곳 설명(L-D5)을 **일관된 하나의 확정 어포던스**로 수렴.

### L5. 속성 vs 속성값 — 확신-초안 확정

- **문제**(L-D6): 개념 구분은 타당하나(정의 vs 데이터), AI가 이미 값에서 정의를 파생하므로 사용자가 스키마를 손으로 따로 정의할 필요가 없다.
- **목표**: AI가 속성 **정의 + 값**을 함께 추출해 초안으로 제시, 사용자는 확정만. 프롬프트의 "속성을 노드로 만들지 마라" 3중 반복(L-D6)을 1회로 정리. 인스턴스 값 입력은 원바이원 대신 **표/그리드형 일괄 편집**(백로그, 선택).

### L6. 확정 = 신뢰도 트리아지 (핵심 레버리지)

- **문제**(L-D7): "값싼 확정"은 AI 초안이 확신 있고 대체로 맞을 때만 이득이다. 원탭 × 오류 다수 = 여전히 무겁다. PRD-K가 체크박스·검토 표면을 넣었으나 기본 체크 상태가 신뢰도로 구동되지 않는다.
- **목표**: 확정 화면을 **"고신뢰 전부 수락 + 저신뢰 몇 개만 결정"** 구조로.
  - **고신뢰**(높은 confidence / `categoryConfidence ≥ 0.7` / Critic 무이슈): 기본 체크 = 수락. 접힌 요약("N개 자동 반영 예정")으로.
  - **저신뢰**(저 confidence / `categoryConfidence < 0.7` / Critic 플래그): 검토 표면으로 자동 승격, 사용자 결정 요청.
  - 이로써 앞서 "비용"으로 보였던 `categoryConfidence`·Critic 큐가 **확정 UX의 트리아지 연료로 용도 전환**된다.
- **효과**: "200개 검토"가 "훑어보고 저신뢰 N개만 손봄"으로. 사용자가 느끼는 "결정할 게 많다"의 근본 해소.

## 3. 비목표 (Non-Goals)

- **요소 삭제로 온톨로지를 얇게 만들기** — 아님. 구조 풍부함(B)은 유지·강화한다.
- **클래스/인스턴스·속성/속성값 개념 폐지** — 아님. AI 위임 + 확정 UX만 바꾼다.
- **관계 이름(`type`) 통제어휘화** — 아님. 자유 문자열 유지(도메인 어휘 손실 방지).
- **비즈니스 로직·그래프 의미 변경** — 아님. 표현·분류·인터랙션 계층만 변경(기존 세션 원칙).
- **외부 myATHENA/전략 엔진 구현** — 본 PRD 범위 밖. 본 PRD는 그 입력이 될 온톨로지의 표현을 정리한다.

## 4. 마일스톤 (제안)

| 마일스톤 | 범위 | 선행 |
|---|---|---|
| **M1** | L1 공리+제약 → 단일 규칙 모델 통합(UX·거버넌스 라우팅·상태 평문·무손실 마이그레이션) | 독립 |
| **M2** | L2 카테고리 5→3 레이어(스키마 enum·프롬프트 축소·매핑·백필·eval 채점 갱신) + `categoryConfidence` 용도 전환 준비 | 독립 |
| **M3** | L3 관계유형+관계 단일 액션(assist 액션·plan·ConstraintsPanel 관계 선택) — 데이터모델 보존 | 독립 |
| **M4** | L4 클래스/인스턴스 확신-초안 + 원탭 전환 확정 어포던스(마이크로카피·토글 1급화) | PRD-K |
| **M5** | L6 신뢰도 트리아지 확정(기본 체크·자동 승격·요약) + L5 속성/속성값 확정 정리 | PRD-K, M2 |

> 마일스톤 순서는 독립적 3개(M1~M3, 스키마·추출 계층)를 먼저, UX 계층(M4~M5)을 뒤에 두는 것을 권고. M2는 M5의 트리아지 신호(`categoryConfidence`) 재활용과 맞물림.

### 확정 전 미결 사항 (사용자 승인 필요)

1. **L2**: `diagnostic`을 **다이나믹에 흡수**(권고) vs **독립 4번째 레이어 유지**? — 진단 추론을 별도 cypher 필터로 자주 쓸 계획이면 독립 유지.
2. **L1**: free-text 공리를 **제약 모델에 흡수** vs **"설명 메모"로 강등 후 규칙과 분리**? — 강제 규칙과 메모를 섞지 않으려면 후자.
3. **L3**: 데이터모델 RelationType 테이블 **유지**(권고, TBox/ABox) vs 완전 통합?

## 6. 성공 지표

- **판단 감소**: 초안 확정 1건당 사용자 클릭·결정 수(고신뢰 자동수락 비율↑).
- **개념 감소**: "규칙" 생성 시 노출 개념 1개(공리/제약 이중성 소멸), 관계 추가 시 액션 1개.
- **토큰·정확도**: parse 프롬프트 관계 분류 토큰↓, 카테고리 오분류율↓(3레이어 혼동행렬).
- **완주율**: 비전문가가 안내 없이 "입력→초안→확정" 완주(클래스/인스턴스 학습 불요).
- **무회귀**: 기존 테스트 그린 + 구조 풍부함(B) 유지 확인.
