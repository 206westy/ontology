## 0. 목적

**한 줄 요약**: 온톨로지 구성요소를 **팔란티어 온톨로지의 3레이어(Semantic 지식 · Kinetic 행동 · Dynamic 자동화)로 재조직**하여, ① AI가 대화에서 내리는 분류 결정을 단순·정확하게 만들고(토큰·오분류↓), ② 사용자가 팝오버에서 하나씩 판단하던 것을 **AI 확신 초안 + 신뢰도 트리아지 확정**으로 바꾸며(체감 복잡도↓), ③ 같은 개념의 중복(공리 vs 제약)을 Dynamic 레이어의 단일 "규칙"으로 통합한다. 요소를 얇게 만드는 게 목표가 아니다 — **구조는 팔란티어식으로 더 또렷해지고(기계 풍부함↑), 사용자 판단·AI 토큰은 줄인다.**

**핵심 전제 — 두 종류의 복잡도를 분리한다**:

| 종류 | 정의 | 처리 원칙 |
|---|---|---|
| **(A) 사용자 결정 복잡도** | 사람이 내려야 하는 *판단*의 수·종류 | **순수 비용.** 검색·전략에 도움 안 되면서 판단만 시키는 구분은 통합·은닉·AI 위임 |
| **(B) 기계용 구조 풍부함** | 그래프에 담긴 *타입 디테일*(클래스·타입 관계·domain/range·레이어) | **자산.** 상용 LLM GraphRAG + 노드 벡터서치 + 문서 답변형 전략 생성기의 검색 grounding을 높임 → 유지·강화 |

**대상 사용자**: 온톨로지·그래프 지식이 없는 도메인 전문가(CLAUDE.md 전제). "이걸 클래스/인스턴스로 넣을지", "이 관계가 무슨 분류인지"를 스스로 판단하게 요구하면 안 된다 — AI가 확신 초안을 제시하고, 사용자는 원탭으로 수락·수정만 한다.

**하류 소비처(설계 제약)**: 완성된 온톨로지는 **상용 LLM 기반 GraphRAG**(text-to-cypher를 LangGraph 노드로) + **노드 벡터서치** + **문서 기반 답변**을 결합한 전략 생성기에 즉시 투입된다. 3레이어는 이 전략 사이클(지식 조회 → 행동 실행 → 자동화)과 정확히 대응한다.

**성공의 정의**:
- 관계 분류가 5-way에서 **Semantic/Kinetic 2-way**로 단순화되어, parse 프롬프트 최대 블록(diagnostic/procedural 구분)이 삭제되고 오분류가 감소한다.
- "제약조건"을 만들 때 사용자가 마주하는 개념이 **하나**(Dynamic 레이어의 "규칙")다 — 공리/제약 이중 시스템 소멸.
- 클래스/인스턴스·속성/속성값이 **AI 확신 초안 + 원탭 확정 + 신뢰도 트리아지**로 이뤄지고, 사용자가 개념을 학습하지 않아도 완주한다.
- 온톨로지가 팔란티어 3레이어로 정리되어 **구조적 구체성(B)이 향상**된다(요소를 지우지 않음).

## 1. 팔란티어 3레이어 아키텍처 (본 PRD의 뼈대)

전통 온톨로지는 지식을 표현하는 **Semantic 한 층**뿐이지만, 팔란티어는 그 위에 **Kinetic(행동)**·**Dynamic(자동화)**를 얹어 "지식 → 행동 실행 → 학습·진화"의 운영 사이클을 이룬다. 본 PRD는 이 개념을 도입해 기존 요소를 재조직한다.

| 레이어 | 팔란티어 정의 | 본 앱에서의 귀속 |
|---|---|---|
| **Semantic (지식)** | 엔티티·속성·관계 = "무엇이 존재하고 어떻게 연결되나" | 클래스·인스턴스·속성·속성값 + **서술형 관계**(구조·인과·서술) |
| **Kinetic (행동)** | Actions/Functions = 온톨로지를 *작동*시키는 조치 | **행동형 관계**(증상→점검, 정해진 조치 실행) |
| **Dynamic (자동화)** | 비즈니스 규칙·정책·워크플로·권한 = 온톨로지에 *행위* 부여 | **통합 "규칙"**(구 공리+제약) + (후속) 에이전트 자동화 훅 |

> 근거: [Palantir Ontology Overview](https://www.palantir.com/docs/foundry/ontology/overview) · [Understanding Palantir's Ontology (Medium)](https://pythonebasta.medium.com/understanding-palantirs-ontology-semantic-kinetic-and-dynamic-layers-explained-c1c25b39ea3c) · [Palantir vs Classic Ontology (Pebblous)](https://blog.pebblous.ai/project/CURK/ontology/palantir-vs-classic-ontology/en/)

**관계 5분류 → 2레이어 재배치**:

| 기존 카테고리 | → 레이어 | 성격 |
|---|---|---|
| `structural` (부품/포함/위치) | **Semantic** | 무엇이 무엇과 구성/연결 |
| `causal` (A↑→B) | **Semantic** | 도메인 인과 지식(사실) |
| `descriptive` (서술/명명) | **Semantic** | 단순 서술 |
| `diagnostic` (증상→점검) | **Kinetic** | 조사·점검 액션 |
| `procedural` (정해진 조치 실행) | **Kinetic** | 실행 액션 |

**본 PRD의 범위(스코프 (가))**: 3레이어로 **조직·태깅·명명 + 규칙 통합**까지. Dynamic 레이어의 **실제 워크플로/에이전트 자동화 실행은 후속(v6 Operator·전략 생성기 PRD)**으로 분리한다. 이렇게 해야 지금 온톨로지가 팔란티어 구조로 "정리"되면서 범위가 터지지 않는다.

## 2. 현재 구성요소 진단 (코드 근거)

7요소 추적 결과, **AI가 대화에서 실제로 "판단"하는 결정은 사실상 2가지(클래스 vs 인스턴스, 관계 카테고리)뿐**이고, 나머지는 구현 아티팩트이거나 중복이다.

| # | 신호 | 근거 | 성격 |
|---|---|---|---|
| L-D1 | **공리(Axiom)와 제약(Constraint)이 둘 다 "제약조건"인데 별개 시스템** — `axioms`/`axiom_classes`(free-text `rule_logic`) vs `constraints`(4타입). **실제 강제되는 건 `constraints.cardinality` 하나뿐**, `axioms`는 `/api/validate`가 읽지 않음 | `drizzle/schema.ts:313,337,510-542`, `api/validate/route.ts`, `RightPanel.tsx`(인라인) vs `ConstraintsPanel.tsx`(10필드) | **중복** — 한 개념이 두 요소·두 UI |

### 보존할 강점 (건드리지 않음)

- **클래스/인스턴스 요소**(TBox/ABox — GraphRAG cypher·전략 walk의 근간).
- **관계 이름(`type`) 자유 문자열** — 도메인 어휘를 손실 없이 담는 층. 레이어 태깅과 무관하게 유지(세밀 필터는 이름+벡터로 복원).
- **출처·확신도**, **구획(Partition)** — 저비용·비-AI 요소. 현행 유지.
- **PRD-I/K 확정 UI 자산**(ConfirmCard·체크박스·검토 표면·스케일) — 재활용, 재작성 금지.

## 3. 변경 축 (L1~L6)

> 원칙: **삭제할 것은 오직 — 사용자에게 판단을 시키면서 검색·전략엔 도움 안 되는 구분뿐.** 나머지는 3레이어로 재조직·통합·AI 위임.

### L1. Dynamic 레이어 — 공리+제약 → 단일 "규칙(Rule)" 통합 **[Dynamic]**

- **문제**(L-D1): 한 개념("지켜야 할 규칙")이 두 요소·두 UI·두 저장소로 갈라져, 사용자는 "제약조건"이 실제로 막는지 알 수 없다.
- **목표**: 규칙을 **Dynamic 레이어의 단일 시민**으로. 사용자·AI에게 노출되는 규칙은 **하나의 모델**. 타입 규칙(cardinality/disjoint/domain_range/property_value)을 정본으로 삼고, 자유서술 규칙은 **"설명 메모(비강제)"로 흡수**해 무손실 보존(전략 생성기 LLM이 그 문장을 참고 지식으로 읽을 수 있음).
- **UX**: 규칙 생성 진입점 하나. 각 규칙에 **"강제됨(typed) / 설명 메모(비강제)" 상태를 평문 표기** → 사용자가 "이 규칙이 실제로 막아주는지"를 즉시 앎. 거버넌스 AI 제안(`suggest-governance`)도 단일 모델로 라우팅.
- **확정**: 자유서술 공리는 **흡수(폐지 아님)**. RelationType 유지와 함께 아래 §5 확정 로그 참조.

### L2. 관계 레이어링 — 5분류 → Semantic/Kinetic 2레이어 **[Semantic·Kinetic]**

- **문제**(L-D2, L-D3): 5분류는 프롬프트 토큰이 크고 diagnostic/procedural 혼동이 최다 오류인데, 앱 내부에선 사실상 2분류로만 소비되고, 값이 정체성에 박혀 있다.
- **목표**: 관계를 팔란티어 2레이어로 태깅.
  - **Semantic**: structural + causal + descriptive (지식·서술)
  - **Kinetic**: diagnostic + procedural (행동·조치)
- **효과(토큰·정확도)**: parse 프롬프트에서 **diagnostic/procedural 구분 전용 문단(최장 블록) 삭제**, 5종 정의를 2줄로 축소, **`categoryConfidence` 필드 제거(또는 L6 트리아지용으로만)**, CSV 반복 축소. **최다 오분류 클래스(diagnostic↔procedural)가 같은 Kinetic이 되어 증발.** 남는 판단은 "서술이냐/행동이냐" 단일 이분 → LLM이 5-way보다 안정.
- **정체성 정리(그린필드)**: `category`(5값)를 **`layer`(semantic/kinetic) 2값으로 정본 교체**. **`layer`를 엣지 안정 ID·중복키에서 제외** — 정체성 = (source, target, 관계이름). 데이터 폐기 가능하므로 drop & recreate. PRD-H 패턴·eval 혼동행렬·Neo4j 푸시(`r.layer`)도 2레이어로 일괄 정리(보존 데이터 없어 회귀 부담 없음).
- **전략 활용**: text-to-cypher는 `WHERE r.layer='kinetic'`로 조치 계열, `'semantic'`으로 지식 계열 필터. "증상만(diagnostic)" 같은 세밀 필터는 관계 이름 + 노드/엣지 벡터서치로 복원.

### L3. 관계유형 + 관계 → 사용자·AI 단일 "관계 추가" 액션 **[Semantic]**

- **문제**(L-D4): LLM·사용자 어느 쪽도 RelationType과 Edge를 구분하지 않는데, assist 액션·plan 로직·`ConstraintsPanel`의 RelationType 원시 드롭다운이 둘을 노출한다.
- **목표**: **인터랙션·AI 액션에서 관계는 하나.** `RelationPopover`의 "이름 입력 → 유형 자동 생성 + 엣지 생성" 패턴을 표준으로. assist 액션도 단일 "관계 추가"로. `ConstraintsPanel`의 관계 선택은 원시 RelationType id가 아니라 **사용자가 만든 관계/쌍**을 고르게.
- **확정**: **데이터모델의 TBox/ABox 분리(RelationType 테이블)는 보존** — GraphRAG text-to-cypher가 "이 온톨로지에 어떤 타입 관계가 있나"를 스키마·domain/range로 조회하는 데 유용. "UX 통합"이지 "스키마 통합"이 아님(§5).

### L4. Semantic 레이어 — 클래스 vs 인스턴스 확신-초안 + 원탭 전환 **[Semantic]**

- **문제**(L-D5): 이 결정은 제거 불가(데이터모델·RAG 근간)이나, 지금은 사용자가 개념을 이해해 판단하도록 요구된다. 예: *"내가 키우는 호랑이의 이름은 범이야"* — 범이는 클래스가 아니라 **호랑이 클래스의 인스턴스**. AI가 판정해 초안 제시하되 확정 UI가 약함(작은 팝오버 토글).
- **목표**:
  1. **AI가 확신 커밋** — 애매 회피("헷갈리면 인스턴스" 기본값)로 대부분을 원탭 수락으로.
  2. **전환을 1급 어포던스로** — PRD-K가 올린 스케일·발견성 위에, "이건 **카테고리(클래스)**인가요, **실제 하나(인스턴스)**인가요?" 평문 마이크로카피 + 원탭 토글을 확정 카드에 상시 노출.
  3. 흩어진 3곳 설명(L-D5)을 **일관된 하나의 확정 어포던스**로 수렴.

### L5. Semantic 레이어 — 속성 vs 속성값 확정 정리 **[Semantic]**

- **문제**(L-D6): 개념 구분은 타당하나, AI가 이미 값에서 정의를 파생하므로 사용자가 스키마를 손으로 따로 정의할 필요가 없다.
- **목표**: AI가 속성 **정의 + 값**을 함께 추출해 초안 제시, 사용자는 확정만. "속성을 노드로 만들지 마라" 3중 반복(L-D6)을 1회로 정리. 인스턴스 값 입력의 표/그리드 일괄 편집은 **백로그(선택)**.

### L6. 확정 = 신뢰도 트리아지 (핵심 레버리지) **[교차]**

- **문제**(L-D7): "값싼 확정"은 AI 초안이 확신 있고 대체로 맞을 때만 이득. 원탭 × 오류 다수 = 여전히 무겁다.
- **목표**: 확정 화면을 **"고신뢰 전부 수락 + 저신뢰 몇 개만 결정"** 구조로.
  - **고신뢰**(높은 confidence / Critic 무이슈): 기본 체크 = 수락. 접힌 요약("N개 자동 반영 예정").
  - **저신뢰**(저 confidence / Critic 플래그 / 클래스·인스턴스 근거 약함): 검토 표면으로 자동 승격, 결정 요청.
  - 이로써 `categoryConfidence`·Critic 큐가 **확정 UX의 트리아지 연료로 용도 전환**(L2에서 관계 분류용으로는 축소되지만, 신뢰 신호 자체는 여기서 재활용).
- **효과**: "200개 검토"가 "훑어보고 저신뢰 N개만 손봄"으로. "결정할 게 많다"의 근본 해소.

## 4. 비목표 (Non-Goals)

- **요소 삭제로 온톨로지를 얇게 만들기** — 아님. 3레이어로 구조 풍부함(B)을 오히려 강화.
- **Dynamic 레이어의 워크플로/에이전트 자동화 실행 구현** — 아님(스코프 (가)). 이번엔 규칙 통합·명명·귀속까지. 실행은 후속 PRD.
- **클래스/인스턴스·속성/속성값 개념 폐지** — 아님. AI 위임 + 확정 UX만 바꿈.
- **관계 이름(`type`) 통제어휘화** — 아님. 자유 문자열 유지(세밀 필터는 이름+벡터).
- **비즈니스 로직·그래프 의미 변경** — 아님. 표현·분류·인터랙션 계층만(기존 세션 원칙).

## 5. 확정 로그 & 미결

**확정(2026-07-07)**:
1. **데이터**: 그린필드(테스트 데이터 폐기) → drop & recreate, 정체성에서 `layer` 제외.
2. **관계 레이어**: Semantic/Kinetic **2레이어**(diagnostic·procedural=Kinetic, 나머지=Semantic). Dynamic은 관계가 아니라 규칙/자동화 레이어.
3. **공리 처리**: 단일 규칙 모델에 **"설명 메모(비강제)"로 흡수**(폐지 아님).
4. **RelationType 테이블**: **데이터모델 유지**(TBox/ABox), UX·AI 액션만 단일화.
5. **스코프**: (가) — 3레이어 조직·태깅·규칙 통합까지. Dynamic 자동화 실행은 후속.

**미결(구현 착수 전 확인 권장)**:
- `descriptive` 관계를 Semantic으로 흡수하되, 미리보기의 "actionable vs descriptive" 이분(`parse-mapping.ts:209-219`)을 "Semantic vs Kinetic"으로 대체할지 / descriptive를 별도 약한신호로 남길지.
- Dynamic 레이어를 UI에 **명시적 3번째 탭/뷰로 노출**할지, 내부 조직으로만 둘지(사용자 판단 증가 방지 관점).

## 6. 마일스톤 (제안)

| 마일스톤 | 레이어 | 범위 | 선행 |
|---|---|---|---|
| **M1** | Dynamic | L1 공리+제약 → 단일 규칙(강제/메모 표기·거버넌스 라우팅·drop&recreate) | 독립 |

## 8. 성공 지표

- **토큰·정확도**: parse 관계 분류 토큰↓(diagnostic/procedural 블록·categoryConfidence 삭제), 관계 오분류율↓(2레이어).
- **판단 감소**: 초안 확정 1건당 사용자 결정 수↓(고신뢰 자동수락 비율↑).
- **개념 감소**: "규칙" 생성 시 노출 개념 1개(공리/제약 이중성 소멸), 관계 추가 시 액션 1개.
- **완주율**: 비전문가가 안내 없이 "입력→초안→확정" 완주(클래스/인스턴스·카테고리 학습 불요).
- **구조 향상**: 온톨로지가 Semantic/Kinetic/Dynamic로 정리되어 GraphRAG grounding·전략 사이클 대응(B 유지·향상).
