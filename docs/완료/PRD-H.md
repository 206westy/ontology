## 0. 이 제품이 구현하는 패러다임

**패턴-시드 스키마-적응형 온톨로지 구축.** 입력을 인지해 도메인에 맞는 **설계 패턴(ODP)을 시드**로 온톨로지를 그리고, 새 개념·관계가 나오면 `기존에 매핑 / 패턴 확장 / 새 패턴·구획으로 분기` 중 하나로 적응한다. AI는 도메인·패턴·관계·병합·확장/분기·용어 뜻을 **판정**하고, 사용자는 **컨펌만** 한다. 패턴은 단순 "모양"이 아니라 `역할(노드 타입) + 관계 타입 + competency question + 소비(traversal) 템플릿` 번들이며, 이 번들이 제품의 판매 단위이자 차별점이다.

---

## 배경: 단일 제네릭 추출의 3중 실패 (실측)

1. **파편화** — 같은 대상이 표면형만 달라도(에어압 ↔ air pressure, IO 입력 ↔ IO board) 별개 노드로 튀어 그래프가 여러 조각으로 분리됨. 근본 원인은 관계 끝점을 정확 문자열로만 맞추고 실패 시 부모 없는 노드를 즉석 생성하는 처리.
2. **평탄화** — 도메인 형태를 모른 채 `증상 → (확인한다) → 전부`로 depth-1 평탄. 진단이 요구하는 증상→원인→점검→조치의 인과 계층이 사라짐.
3. **도메인 무지** — 추출이 단일 제네릭. 도메인 템플릿은 "시작점"일 뿐 입력별로 패턴을 골라 주입하는 경로가 없음. 또한 미정의·모호 용어(예 `VV`)를 맥락 없이 방치.

**목표**: 도메인에 맞는 패턴을 시드로, 정체성(병합)·형태(인과 계층)·용어(맥락 해소)를 함께 정확히 세우고, 신규 도메인은 컨펌 한 번으로 확장/분기한다.

---

## 설계 원칙

1. **Supabase가 최신 기준(SoT), Neo4j는 발행본.** 생성·병합·대조·렌더는 최신 기준을 본다.
2. **패턴 = 번들.** 역할·관계·CQ·traversal을 한 단위로 관리·버전.
3. **컨펌만으로 진행.** 자동 확정 0. 모든 AI 판정은 사용자 승인 게이트를 통과.
4. **구획으로 도메인 격리.** 서로 다른 도메인은 별 구획(교차오염 방지). 구획 간 동일 대상은 **브릿지**로 연결.
5. **스키마 적응 3분기.** 신규 요소는 `매핑 / 확장 / 분기`. 충실히 정렬 불가할 때만 분기.
6. **관계는 근거 기반.** 단지 같은 문서에 등장했다는 이유로 잇지 않는다. 근거 없으면 섬으로 두고 "연결 제안"만. 모든 관계에 근거 스팬·신뢰도.
7. **용어는 맥락으로 해소.** 미정의·모호 용어는 도메인 + 현재 온톨로지 맥락을 주입해 뜻을 좁힌다. 전역 검색 금지, 도메인-스코프.
8. **진행형 생성.** 온톨로지가 그려지는 과정을 실시간으로 보여준다(노드가 순차로 캔버스에 나타남).
9. **모델 절약.** 추출은 경량 모델, 판정(동일성·드리프트·용어 뜻)만 정밀. 대량 처리 비용 보호.

## 전체 동작 흐름

```
입력(텍스트/데이터)
 → [H2] 도메인 인지 · 신뢰도 · 혼합 감지 · CQ 생성
 → [H8-a] 도메인 요약 카드 ──[컨펌: 이 패턴으로 / 다른 패턴 / 혼합→분할]──▶
 → [H1] 패턴을 시드로 로드  (없으면 검색·리서치로 신규 패턴 초안)
 → [H3] 패턴-시드 생성 : 근거 기반 관계 + 역할 타이핑 + 배치 canonical 병합 ─ 진행형 렌더
 → [H4] 미정의·모호 용어 감지 → 맥락 주입 해소 → [H8-e] 용어 확인 카드 ──[컨펌]──▶
 → [H5] 스키마 드리프트 판정: 매핑 / 확장 / 분기 → [H8-d] 결정 카드 ──[컨펌]──▶
 → [H6] 구획 착지 + 크로스-구획 브릿지 제안 → [H8-f] 브릿지 카드 ──[컨펌]──▶
 → [H7] 연결성 + CQ 통과 검증
 → 커밋 → 발행(Neo4j)
```

# H1. 패턴 레지스트리

**목적** — 도메인 패턴을 재사용·버전 가능한 번들 자산으로 보유.

**동작 방향** — 각 패턴은 `역할(노드 타입) · 관계 타입 · competency question 세트 · 소비 traversal 템플릿`을 담는다. 초기 5종: 진단/FMEA(증상·고장모드·원인·영향·부품·파라미터·점검·조치 / indicates·caused_by·detected_by·resolved_by·part_of), 행정/프로세스, 카탈로그/BOM, 조직/역할, 이벤트/타임라인. 기존 도메인 템플릿을 패턴으로 승격. 패턴 버전업 시 이전 버전 보존(구획 마이그레이션 추적).

**통과조건**
- 패턴이 번들(역할+관계+CQ+traversal)로 로드된다.
- 패턴마다 CQ 세트가 존재한다(예 FMEA: "증상 X의 원인?", "원인 Y의 조치?").
- 버전업 후에도 이전 버전이 조회된다.

---

# H2. 도메인 인지 + 패턴 라우터

**목적** — 입력을 보고 어떤 도메인·패턴인지 판단하고 요약을 제시.

**동작 방향** — 입력에서 도메인·신뢰도·혼합 비율(예 진단 70% + 행정 30%)·추천 패턴·CQ 미리보기를 산출. 레지스트리에 맞는 패턴이 없으면 검색·리서치로 해당 도메인의 온톨로지 관례를 조사해 **신규 패턴 초안**을 제안(초안 상태, 확정은 컨펌). 이 단계는 추출을 시작하지 않고 "요약 → 컨펌" 게이트까지만.

**UX** — H8-a 도메인 요약 카드로 노출.

**통과조건**
- 진단 노트 → 도메인=진단, 신뢰도 높음, 추천=FMEA.
- 혼합 문서 → 두 패턴 비율 반환.
- 레지스트리에 없는 도메인 → 신규 패턴 초안 제안.
- 컨펌 전에는 그래프가 생성되지 않는다.

# H3. 패턴-시드 온톨로지 생성 (정확한 관계 · 역할 타이핑 · 배치 병합)

**목적** — 선택 패턴을 시드로, **정체성이 깨끗하고 관계가 정확한** 온톨로지를 진행형으로 그린다.

**동작 방향**
- **패턴 주입 추출**: 엔티티는 패턴의 역할 타입으로, 관계는 패턴의 관계 타입으로 뽑는다. 진단이면 `증상→원인→점검/조치`의 인과 계층으로 추출(평탄 리스트 아님). 부모 없는 무타입 노드 생성 금지.
- **정확한 관계 제시(핵심)**:
  - 관계는 **원문 근거 스팬 + 신뢰도**를 항상 동반한다. 근거 없는 연결은 만들지 않고 섬으로 두며 "연결 제안"만 한다(같은 문서 등장 = 근거 아님).
  - 방향·의미가 정확해야 한다(source→target, 어떤 관계 타입인지). 모호하거나 저신뢰 관계는 자동 추가하지 않고 확인 대상으로 표시.
  - 사용자가 각 관계의 근거·신뢰도를 보고 검증할 수 있어야 한다(관계 옆에 근거 미리보기).
- **배치 canonical 병합**: 한 번의 생성 안에서 표면형이 다른 동일 대상을 하나로 해소(정규화 + 도메인 별칭 + 의미/어휘 하이브리드 후보 + 동일성 판정). 판정은 병합 "제안"으로만 뜨고 자동 병합하지 않음(H8-c).
- **진행형 렌더**: 생성이 진행되는 동안 노드·관계가 캔버스에 순차로 나타난다.

**UX** — 진행형 캔버스, 관계별 근거·신뢰도 표시, 저신뢰 관계는 점선/확인 배지.

**통과조건**
- 진단 노트가 증상→원인→점검→조치 역할 구조로 나온다(평탄 아님).
- 모든 관계에 근거·신뢰도가 붙는다. 근거 없는 강제 연결 0건.
- 표면형이 다른 동일 대상(에어압↔air pressure 등)이 병합 제안으로 잡히고, 승인 시에만 하나가 된다.
- 무타입·부모없음 노드 0건.
- 생성 과정이 진행형으로 보인다.

# H4. 맥락 주입형 용어 해소 (Context-Injected Term Resolution)

**목적** — 미정의·모호 용어(약어·도메인 은어)를 **현재 온톨로지 맥락에 맞는 뜻**으로 정확히 해소해 이해도를 높인다.

**동작 방향**
- **트리거**: 정의 없는 노드, 모호한 약어(예 `VV`, `EMO`), 저신뢰 타입 판정. 생성 중 자동 감지해 "미정의/모호 용어" 목록으로 모음(건별 남발 아님).
- **해소 순서**: ① 내부 도메인 용어집/별칭 → ② 현재 온톨로지·구획 맥락(형제 노드, 상위 패턴, 인접 개념) → ③ (opt-in) 웹 검색 → ④ 사용자 확인.
- **맥락 주입 검색(핵심)**: 웹 검색 시 **키워드만 던지지 않는다.** 반드시 함께 주입:
  - 도메인(예: 반도체 설비 메인트),
  - 현재 온톨로지 맥락(형제/인접 노드, 소속 패턴·구획),
  - 후보 타입(부품/파라미터/신호 등).
  - 예) `VV` 단독 검색 금지 → "반도체 설비 유지보수 맥락의 VV(밸브 후보) — 인접 노드: 솔레노이드·에어 실린더·레귤레이터" 형태의 맥락 질의로 뜻을 좁힌다.
- **도메인-스코프 원칙**: `VV`는 다른 분야에선 다른 뜻이므로, 해소 결과는 **이 온톨로지/구획의 도메인에 한정**해 채택한다(전역 사실로 확정하지 않음).
- **결과 재주입**: 확정된 뜻은 (1) 노드 정의에 기록, (2) 세션/도메인 용어집에 등록, (3) **이후 추출·검색 맥락에 계속 주입**되어 같은 문서·구획에서 `VV=밸브`로 일관 적용.
- **근거·신뢰도·출처**: 해소는 후보(랭킹된 뜻)로 제시하고 출처(내부/웹)와 신뢰도를 붙인다. 자동 확정 없음.

**UX** — H8-e 용어 확인 카드. 주입한 맥락을 투명하게 보여줌(무엇을 근거로 이 뜻을 골랐는지). 확정 시 "이후 이 온톨로지에서 VV=밸브로 사용" 안내.

**가드레일(비판적 반영)** — 미정의 용어가 대량이면 검색 폭주·지연·오해소 위험. 따라서 (a) 모호·미정의로 판정된 것만, (b) 배치로 묶어, (c) 웹은 opt-in, (d) 확정 결과를 용어집에 캐시해 재검색 방지.

**통과조건**
- `VV` 같은 약어가 도메인+온톨로지 맥락을 주입한 질의로 해소되어 이 맥락의 뜻(밸브) 후보를 제시한다.
- 확정된 뜻이 노드 정의에 남고, 같은 세션의 이후 생성·검색에 재주입된다.
- 웹 검색 질의에 도메인·인접 맥락이 포함된다(키워드 단독 아님).
- 사용자 확인 전에는 뜻이 확정되지 않는다.
- 도메인이 다른 문서에 같은 약어가 나와도 이전 뜻을 전역 강제하지 않는다(구획-스코프).

# H5. 스키마 드리프트 — 매핑 / 확장 / 분기

**목적** — 패턴에 없는 신규 개념·관계를 만났을 때 확장할지 새로 그릴지 AI가 판단, 사용자는 컨펌.

**동작 방향** — 신규 요소를 3분기로 판정: **매핑**(기존 역할/관계에 정렬 → 정상), **확장**(도메인 내부 확장 → 패턴에 역할/관계 추가, 같은 구획, 패턴 버전업 제안), **분기**(충실한 정렬 불가 = 다른 도메인 → 새 패턴/구획 제안). 여러 도메인에 걸치는 요소는 브릿지 후보로 표시. 확정은 컨펌 시에만.

**UX** — H8-d 확장 vs 분기 결정 카드(각 선택의 미리보기 포함).

**통과조건**
- 진단 패턴에 자연스러운 신규 원인 → 확장 제안(같은 구획).
- 진단 문맥에 행정 승인 흐름 유입 → 분기 제안(새 구획).
- 컨펌 전 패턴·구획 불변.

# H6. 구획 + 크로스-구획 브릿지

**목적** — 도메인별로 격리하되, 구획을 넘는 동일 대상은 브릿지로 잇는다.

**동작 방향** — 분기 확정 시 새 구획 생성·패턴 바인딩. 서로 다른 구획에 나타난 동일 대상(예 `펌프447`)을 크로스-구획 동일성 판정으로 찾아 브릿지 후보로 제시. 브릿지도 타입·근거를 갖는다(무분별 연결로 hairball 되지 않게).

**UX** — H8-f 브릿지 제안 카드.

**통과조건**
- 같은 대상이 두 구획에 등장 → 브릿지 후보 제안.
- 컨펌 시에만 브릿지 생성, 근거·타입 기록.

# H7. 연결성 + CQ 검증

**목적** — "정상 생성"의 기준을 파편화 없음(연결성) + 소비 가능(CQ 통과)으로 정의.

**동작 방향** — 그래프가 여러 조각으로 분리됐는지(도달성) 검사해 분리 시 명시적으로 경고(현행 "섬 없음" 오탐 교정). 인스턴스까지 고아 탐지 확장. 그리고 패턴의 CQ 세트로 그래프가 실제 진단 질의에 답할 경로가 있는지 점검해 통과율을 검수에 표시.

**통과조건**
- 병합 전 노트 → "N개로 분리" 경고가 뜬다.
- 병합 후 → 단일 연결, CQ 통과율 표시(예 4/4).
- CQ에 답할 경로가 없는 경우 해당 CQ가 실패로 표시된다.

# H8. HITL 컨펌 UX (제품 시그니처 — 최우선)

**목적** — "AI가 판정 → 사용자는 컨펌만"을 최소 클릭으로. 아래 카드는 모두 확정 인터랙션이며, 이 흐름이 제품의 제1 데모.

**확정된 인터랙션(요구사항 반영)**
- **(a) 도메인 요약 카드**: "이 내용은 **진단**으로 보입니다(신뢰 82%) — [진단/FMEA] 패턴으로 만들까요? [행정/프로세스] [카탈로그/BOM] [직접 선택]"
- **혼합 감지**: "이 문서는 진단 70% + 행정 30% — 두 구획으로 나눌까요? [분할] [하나로]"
- **(b) 진행형 캔버스**: 컨펌 후 노드·관계가 순차로 그려짐(생성이 보이는 경험).
- **(c) 병합 프리뷰 카드**: "에어압 → air pressure로 통합" — 근거·신뢰도·배지(동일/연관/중복가능/신규) + [통합 승인]/[별개 유지]/[해소 큐로]. `별개 유지`는 세션 내 재제안 억제.
- **(d) 확장 vs 분기 결정 카드**: "새 개념 N개가 현재 패턴 밖 — [패턴 확장]/[새 구획으로 분리]/[무시]" + 각 선택 미리보기.
- **(e) 용어 확인 카드**: "VV — 이 맥락(반도체 설비 메인트)에서 후보: ① 밸브(Valve) 92% ② …  [이 뜻으로]/[다른 뜻]/[직접 입력]/[건너뛰기]". 주입한 맥락과 출처를 함께 표시.
- **(f) 브릿지 제안 카드**: "펌프447이 [메인트]·[행정] 양쪽에 등장 — 브릿지로 연결? [연결]/[별개]"

**UX 원칙** — 사용자는 컨펌만. 각 카드는 근거·신뢰도·출처를 투명하게. 관계는 근거와 함께, 저신뢰는 확인 배지로. 다크모드·접근성·CSS 변수 준수(하드코딩 색·이모지 금지), 한국어 문구.

**통과조건**
- 입력 → 요약 카드 → 컨펌해야 생성 시작.
- 생성 중 진행형 렌더가 보인다.
- 병합·용어·확장/분기·브릿지가 각각 카드로 뜨고, 컨펌 시에만 반영된다.
- `별개 유지`·`무시`·용어 확정이 세션 내 유지·재주입된다.

# H9. 모델·기술 방향 (개념 수준)

- **추출**: 경량 모델 + 진행형(부분 결과 스트리밍) 렌더.
- **판정**: 동일성·스키마 드리프트·용어 뜻 판정만 정밀 모델(소량·고정밀). 전면 상향 금지(대량 처리 비용).
- **정체성 후보**: 의미 유사 + 어휘(오타·복합명사) + 도메인 별칭의 하이브리드. 한↔영 기술동의어는 별칭이 신뢰 경로. 임베더 교체는 측정 후 판단.
- **용어 해소 검색**: 맥락 주입 질의 · opt-in · 도메인 스코프 · 결과 용어집 캐시.

## 전체 수용 시나리오 (스모크)

1. **진단 노트 투입** → 요약 "진단(FMEA) 82%, CQ 4개" → [이 패턴으로] → 진행형 생성 → `에어압≡air pressure` 병합 승인 → 미정의 `VV` 감지 → 맥락 주입 해소 "밸브 후보" → [이 뜻으로] → 단일 연결 그래프 + CQ 4/4 통과 → 커밋·발행.
2. **이어서 행정 문서 업로드** → 요약 "행정/프로세스, 현재 패턴 밖" → 분기 결정 → [새 구획으로 분리].
3. **`펌프447` 두 구획 등장** → 브릿지 제안 → [연결].
4. **혼합 문서** → "70/30" → [분할] → 두 구획 + 브릿지 제안.
5. **다른 도메인 문서에 `VV` 재등장** → 이전 뜻을 전역 강제하지 않고 그 구획 맥락으로 재해소.
6. 전 과정 사용자는 **컨펌만**, 자동 확정 0.
