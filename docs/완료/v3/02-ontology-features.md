# 온톨로지 전문가 — v3 신규 기능 제안서

> **작성일**: 2026-03-22
> **작성자**: 온톨로지/지식그래프 전문 기획자 (v3 기획단)
> **관점**: Palantir Foundry/Gotham 대규모 온톨로지 구축 경험 기반

---

## 1. 현재 앱의 온톨로지 관점 한계

### 1.1 스키마 표현력 부족

현재 앱은 클래스/프로퍼티/인스턴스/관계/공리의 기본 CRUD만 지원한다. 실제 온톨로지 구축에서 핵심인 **제약 조건의 형식적 표현**이 거의 없다.

- **카디널리티 제약 없음**: "장비는 반드시 1개 이상의 Site에 위치해야 한다" 같은 관계 다중성을 표현할 수 없다. `edges` 테이블에 카디널리티 컬럼이 없고, 프로퍼티의 `is_required` 플래그만 존재한다.
- **도메인/레인지 미약**: `relation_types`에 `source_class_id`/`target_class_id`가 있으나 SET NULL로 느슨하게 관리되어, 실질적인 도메인/레인지 강제가 불가능하다. 사용자가 아무 노드 간에나 관계를 만들 수 있다.
- **Disjoint/Covering 없음**: 클래스 간 배타성(DryAsher와 WetStation은 동시에 속할 수 없다)이나 완전성(모든 Equipment는 반드시 하위 타입 중 하나)을 선언할 수 없다.
- **공리가 자유 텍스트**: `axioms.rule_logic`이 JSON이지만 실제로 파싱/실행 로직이 없다. 자연어 설명만 저장되고 검증은 수행되지 않는다.

### 1.2 온톨로지 품질 보장 메커니즘 부재

- **유효성 검사 없음**: 고아 노드(관계 없는 인스턴스), 순환 참조(A is-a B is-a A), 타입 불일치(integer 프로퍼티에 문자열 값) 등을 탐지하는 로직이 전혀 없다.
- **중복 탐지 없음**: "DryAsher"와 "Dry Asher"를 서로 다른 클래스로 생성해도 경고가 없다. LLM이 기존 클래스를 컨텍스트로 받지만, 유사도 기반 매칭이 아닌 정확 일치만 확인한다.
- **일관성 검사 없음**: 커밋이나 Neo4j 푸시 전에 온톨로지 전체의 논리적 일관성을 검증하는 단계가 없다.

### 1.3 네임스페이스 및 상호운용성 부재

- 모든 엔티티가 단일 플랫 네임스페이스에 존재한다. 대규모 온톨로지에서는 도메인별 네임스페이스 분리가 필수적이다.
- OWL/RDF/SKOS 같은 표준 온톨로지 포맷의 임포트/익스포트가 불가능하다. Protege에서 만든 기존 온톨로지를 가져올 수 없고, 구축한 온톨로지를 다른 도구에서 활용할 수 없다.
- schema.org, Dublin Core 등 외부 온톨로지를 참조하거나 매핑할 방법이 없다.

### 1.4 LLM 활용의 제한성

- 단일 텍스트 입력 → 구조화만 지원한다. CSV/PDF/위키 등 대량 문서 일괄 파싱이 불가능하다(파일 첨부 버튼은 있으나 미구현).
- LLM이 기존 온톨로지 전체 구조를 깊이 이해하지 못한다. 클래스 이름 목록만 전달하고, 프로퍼티/관계/제약조건 컨텍스트는 포함하지 않는다.
- 자연어 쿼리 → Cypher 변환, 온톨로지 패턴 추천 등 고급 LLM 활용이 없다.

### 1.5 버전 관리의 불완전함

- `commits` 테이블과 `commit_details`로 변경 이력은 추적하지만, **브랜칭/머지**가 없다. "실험적으로 구조를 바꿔보고 안 되면 되돌리기"가 undo/redo(50단계)에 의존한다.
- 커밋 간 **diff 비교**가 UI에서 불가능하다(commit_details에 before/after snapshot은 있으나 시각화 없음).
- 특정 시점으로의 **롤백**이 Supabase 레벨에서 불가능하다(Neo4j 롤백만 존재).

### 1.6 단일 사용자 한계

- MVP 설계대로 1인 사용이지만, 실제 도메인 온톨로지 구축은 여러 전문가의 협업이 필수적이다. 현재 아키텍처에 인증, RLS, 동시 편집 고려가 전혀 없다.

## 2. v3 핵심 신규 기능 (Must-have)

### 2.1 제약 조건(Constraints) 관리 시스템

**왜 필요한가**: 제약 조건 없는 온톨로지는 "구조화된 메모장"에 불과하다. 도메인 전문가가 "이건 반드시 이래야 해"라고 아는 규칙을 시스템이 강제하지 않으면, 데이터가 쌓일수록 온톨로지의 신뢰성이 급격히 하락한다.

**사용자 시나리오**:
1. 사용자가 Equipment 클래스의 `located_at` 관계를 선택하고, "최소 1개, 최대 1개" 카디널리티를 설정한다.
2. 이후 Equipment 인스턴스를 생성할 때, Site 관계가 없으면 CommitBar에 경고가 표시된다.
3. 사용자가 "DryAsher와 WetStation은 배타적이다"를 자연어로 입력하면, LLM이 disjoint 제약으로 변환하고, 위반 시 경고한다.

**구현 범위**:
- 카디널리티 제약: 관계별 min/max 설정 (UI: 숫자 입력 또는 드롭다운)
- 도메인/레인지 강제: 관계 생성 시 허용된 소스/타겟 클래스만 선택 가능
- Disjoint 선언: 클래스 쌍/그룹 간 배타 관계 설정
- 프로퍼티 값 제약: range(min/max), pattern(정규식), enum 확장
- 자연어 → 제약 변환: LLM이 사용자 입력을 구조화된 제약으로 해석

**기대 효과**: 온톨로지 품질을 구축 시점에서 보장. Neo4j 푸시 전 검증 게이트 역할.

### 2.2 스키마 유효성 검사 & 일관성 검사

**왜 필요한가**: Palantir Foundry에서 가장 자주 겪는 문제는 "온톨로지가 커진 후 발견되는 구조적 결함"이다. 고아 노드, 순환 참조, 타입 불일치는 조기에 발견할수록 수정 비용이 낮다.

**사용자 시나리오**:
1. 사용자가 Toolbar의 "검증" 버튼을 클릭한다.
2. 시스템이 전체 온톨로지를 스캔하여 문제 목록을 severity별로 표시한다.
3. 각 문제를 클릭하면 해당 노드로 이동하고, "자동 수정" 옵션을 제안한다.
4. 커밋 또는 Neo4j 푸시 시 자동으로 검증이 실행되고, error 수준 문제가 있으면 진행을 차단한다.

**검증 규칙**:
- **Error** (차단): 순환 is-a 참조, 필수 프로퍼티 값 누락, 카디널리티 위반
- **Warning** (경고): 고아 노드(관계 없는 인스턴스), 빈 클래스(인스턴스 0개), 프로퍼티 정의만 있고 값 없는 인스턴스
- **Info** (참고): 유사 이름 중복 가능성, 미사용 관계 타입, 단일 인스턴스 클래스

**기대 효과**: Neo4j에 결함 있는 데이터가 푸시되는 것을 방지. 온톨로지 품질 지표 대시보드화 가능.

### 2.3 중복 탐지 및 병합 (Entity Resolution)

**왜 필요한가**: LLM이 텍스트를 구조화할 때 동일 개념을 다른 이름으로 생성하는 일이 빈번하다. "Dry Asher", "DryAsher", "건식 애셔"가 모두 같은 것인지 사용자가 매번 판단해야 한다.

**사용자 시나리오**:
1. 사용자가 새 텍스트를 입력하면 LLM이 구조화한다.
2. 프리뷰 Phase에서 기존 엔티티와 유사도가 높은 항목에 "기존 'DryAsher'와 동일한가요?" 배지가 표시된다.
3. 사용자가 "병합"을 선택하면 기존 엔티티에 정보가 추가되고, "별개"를 선택하면 새 엔티티로 생성된다.
4. 주기적으로 "중복 검사" 실행 시, 편집 거리/임베딩 유사도 기반으로 후보 쌍을 제시한다.

**기대 효과**: 온톨로지 내 중복 엔티티 최소화. LLM 구조화의 정확도 체감 향상.

### 2.4 대량 문서 일괄 파싱

**왜 필요한가**: 현실에서 도메인 지식은 단일 텍스트 입력이 아니라, CSV 테이블, PDF 매뉴얼, 위키 문서, 스프레드시트 등에 분산되어 있다. 이를 하나씩 복사-붙여넣기하는 것은 비현실적이다.

**사용자 시나리오**:
1. 사용자가 "가져오기" 버튼으로 CSV 파일을 업로드한다.
2. 시스템이 헤더를 프로퍼티로, 행을 인스턴스로 자동 매핑하는 프리뷰를 보여준다.
3. 사용자가 매핑을 조정하고 확정하면 대량 인스턴스가 생성된다.
4. PDF 업로드 시, LLM이 문서를 청크 단위로 파싱하여 클래스/관계를 추출한다.

**지원 포맷 (v3 범위)**:
- CSV/TSV: 구조화된 테이블 → 인스턴스 일괄 생성
- 자유 텍스트 (대량): 청크 분할 → LLM 병렬 파싱 → 결과 병합
- 클립보드 붙여넣기: 현재 미구현 버튼 활성화

**기대 효과**: 온톨로지 구축 속도 10배 이상 향상. 기존 데이터 자산 활용 가능.

### 2.5 자연어 쿼리 → Cypher 변환

**왜 필요한가**: 도메인 전문가는 Cypher를 모른다. "DryAsher 중에 2024년 이후 설치된 것은?"이라고 물으면 바로 답을 얻을 수 있어야 한다. 이것이 "코드/쿼리 없이 온톨로지를 다룬다"는 제품 비전의 완성이다.

**사용자 시나리오**:
1. Toolbar의 "AI 어시스턴트" 버튼(현재 disabled)을 활성화한다.
2. 사용자가 자연어로 질문을 입력한다.
3. LLM이 현재 온톨로지 스키마를 참조하여 Cypher 쿼리를 생성한다.
4. 생성된 Cypher를 프리뷰로 보여주고, 사용자가 실행을 확인한다.
5. 결과를 테이블 또는 서브그래프로 시각화한다.

**기대 효과**: 구축한 온톨로지의 즉각적 활용. "만들고 끝"이 아니라 "만들고 바로 질문"하는 워크플로우.

### 2.6 온톨로지 임포트/익스포트 (기본 포맷)

**왜 필요한가**: 온톨로지는 폐쇄적 도구에 갇혀서는 가치가 없다. Protege, TopBraid 등 기존 도구 사용자가 마이그레이션할 수 있어야 하고, 구축 결과를 표준 포맷으로 공유할 수 있어야 한다.

**사용자 시나리오**:
1. 사용자가 Protege에서 작업하던 OWL 파일을 "가져오기"로 업로드한다.
2. 시스템이 OWL을 파싱하여 클래스/프로퍼티/관계 구조로 변환한 프리뷰를 보여준다.
3. 사용자가 확정하면 기존 캔버스에 병합된다.
4. "내보내기" 버튼으로 현재 온톨로지를 JSON-LD, Turtle, OWL/XML로 다운로드한다.

**v3 지원 포맷**:
- **임포트**: OWL/XML, JSON-LD, CSV (인스턴스)
- **익스포트**: JSON-LD, Turtle, OWL/XML, CSV (인스턴스 테이블)

**기대 효과**: 기존 온톨로지 자산 재활용. 타 도구/시스템과의 상호운용성 확보.

## 3. v3 Nice-to-have 기능

### 3.1 온톨로지 패턴 추천 (도메인별 템플릿)

**왜 필요한가**: 빈 캔버스에서 시작하는 것은 도메인 전문가에게 인지 부하가 높다. "제조업 장비 온톨로지는 보통 이런 구조로 시작합니다"라는 가이드가 있으면 초기 구축 속도가 크게 향상된다.

**사용자 시나리오**:
1. EmptyState에서 "템플릿으로 시작" 옵션을 선택한다.
2. 도메인별 템플릿 목록이 표시된다 (제조업, 의료, IT인프라, 조직 등).
3. 템플릿 선택 시 기본 클래스 구조 + 관계 타입 + 프로퍼티가 캔버스에 생성된다.
4. 사용자가 자기 도메인에 맞게 수정/확장한다.

**기대 효과**: 온보딩 시간 단축. 온톨로지 설계 모범 사례 전파.

### 3.2 온톨로지 자동 완성 / 추천

**왜 필요한가**: 온톨로지 구축 중 "이 클래스에 어떤 프로퍼티를 넣어야 하지?"라는 질문에 LLM이 선제적으로 답할 수 있다.

**사용자 시나리오**:
1. 사용자가 새 클래스 "CMP장비"를 생성한다.
2. RightPanel AI 탭에 "이런 프로퍼티는 어떨까요?" 제안이 자동 표시된다 (slurry_type, pad_type, rotation_speed 등).
3. 사용자가 원하는 제안만 클릭하여 추가한다.
4. 관계도 마찬가지: "이 클래스는 보통 이런 관계를 갖습니다" 추천.

**기대 효과**: 도메인 전문가의 암묵지를 LLM이 보완. 온톨로지 완성도 향상.

### 3.3 변경 이력 비교 (Diff View)

**왜 필요한가**: 커밋 간 무엇이 변경되었는지 시각적으로 비교할 수 없으면, 변경 이력이 있어도 활용이 어렵다. commit_details에 before/after snapshot이 이미 저장되어 있으므로, UI만 추가하면 된다.

**사용자 시나리오**:
1. CommitBar에서 "변경 내역"을 클릭한다.
2. 커밋 목록에서 두 커밋을 선택한다.
3. 그래프 diff 뷰가 표시된다: 추가된 노드(초록), 삭제된 노드(빨강), 수정된 노드(주황).
4. 각 변경 항목을 클릭하면 before/after 상세를 볼 수 있다.

**기대 효과**: 변경 추적의 투명성. "언제 이걸 바꿨지?" 추적 가능.

### 3.4 RightPanel AI 탭 활성화 (컨텍스트 인식 AI 보조)

**왜 필요한가**: PRD에 명시된 핵심 기능이지만 현재 미구현이다. 선택된 노드의 컨텍스트를 인식한 AI 보조는 "코드/쿼리 없이 온톨로지 구축"이라는 핵심 가치의 직접적 구현이다.

**사용자 시나리오**:
1. Equipment 클래스를 선택한 상태에서 AI 탭에 "PM 주기 관련 프로퍼티 추가해줘"라고 입력한다.
2. LLM이 Equipment 클래스의 현재 프로퍼티/관계를 참조하여 `pm_cycle_days: integer`, `last_pm_date: date`를 제안한다.
3. 제안 항목이 해당 섹션에 하이라이트되어 나타나고, 사용자가 확정/삭제한다.

**기대 효과**: 기존 노드 편집의 생산성 향상. "AI가 이해하고 도와준다"는 핵심 경험 실현.

### 3.5 프로퍼티 상속 시각화

**왜 필요한가**: 계층적 클래스 구조에서 상위 클래스의 프로퍼티가 하위로 상속되는 것은 온톨로지의 기본이다. 현재는 각 클래스가 자기 프로퍼티만 보여주므로, "Equipment의 model_name이 SUPRA에도 적용되나?"를 확인하려면 상위 클래스를 일일이 확인해야 한다.

**사용자 시나리오**:
1. SUPRA 클래스를 선택하면 RightPanel에 "상속된 프로퍼티" 섹션이 접힌 상태로 표시된다.
2. 펼치면 Equipment → DryAsher → SUPRA 경로의 모든 프로퍼티가 출처와 함께 표시된다.
3. 상속된 프로퍼티는 읽기 전용이고, 원본 클래스를 클릭하면 이동한다.

**기대 효과**: 온톨로지 계층 구조의 의미를 명확히 전달. 중복 프로퍼티 정의 방지.

## 4. 장기 로드맵 기능

### 4.1 추론 규칙 (Inference Rules) — RDFS/OWL-lite 수준

**왜 필요한가**: "A is-a B이고 B is-a C이면, A is-a C이다" 같은 전이적 추론, "located_at의 역관계는 has_equipment이다" 같은 역관계 자동 생성은 온톨로지의 핵심 가치이다.

**구현 방향**:
- RDFS 수준: subClassOf 전이 추론, domain/range 추론
- OWL-lite 수준: 역관계(inverseOf), 전이관계(transitiveProperty), 대칭관계(symmetricProperty)
- 추론된 관계를 점선으로 시각화 (명시적 관계와 구분)
- Neo4j 푸시 시 추론된 관계도 포함할지 사용자 선택

**기대 효과**: 명시적으로 선언하지 않은 지식을 자동으로 도출. 온톨로지의 지식 밀도 향상.

### 4.2 네임스페이스 관리

**왜 필요한가**: 단일 프로젝트가 커지면 도메인 분리가 필요하다. 또한 외부 온톨로지(schema.org 등)를 참조할 때 네임스페이스가 필수이다.

**구현 방향**:
- 클래스/관계에 네임스페이스 프리픽스 지원 (예: `equip:DryAsher`, `loc:Site`)
- 네임스페이스 관리 패널 (프리픽스 → URI 매핑)
- 외부 네임스페이스 참조 (schema.org, Dublin Core, SKOS)
- 네임스페이스별 색상 구분

### 4.3 온톨로지 버전 관리 및 브랜칭

**왜 필요한가**: "온톨로지의 Git"이라는 아키텍처 컨셉을 완전히 실현하려면, 커밋/푸시를 넘어 브랜칭/머지가 필요하다.

**구현 방향**:
- 브랜치 생성: 현재 온톨로지 상태에서 분기
- 브랜치 간 전환: 각 브랜치의 커밋 이력 독립 관리
- 브랜치 머지: 충돌 감지 및 해소 UI
- 브랜치 비교: 두 브랜치 간 diff 시각화

### 4.4 다중 사용자 동시 편집

**왜 필요한가**: 실제 온톨로지 구축은 도메인 전문가, 데이터 엔지니어, 지식 관리자가 협업하는 작업이다.

**구현 방향**:
- Supabase Auth 기반 사용자 인증
- RLS 활성화 + 역할 기반 권한 (Owner, Editor, Viewer)
- 실시간 커서 공유 (Supabase Realtime)
- 동시 편집 충돌 해소 (CRDT 또는 OT)
- 변경 알림 및 @멘션

### 4.5 온톨로지 리뷰 워크플로우 (PR-like)

**왜 필요한가**: 온톨로지 변경의 품질 게이트가 필요하다. 특히 Neo4j 프로덕션 반영 전에 리뷰어의 승인을 거치는 프로세스가 있어야 한다.

**구현 방향**:
- 변경 요청(Change Request) 생성: 커밋 묶음 + 설명
- 리뷰어 지정 및 코멘트
- 승인/반려 프로세스
- Neo4j 푸시 시 승인 필수 옵션

### 4.6 SPARQL 엔드포인트 연결

**왜 필요한가**: SPARQL은 온톨로지 세계의 표준 쿼리 언어이다. 외부 트리플스토어와 연동하거나, 기존 SPARQL 기반 시스템과 통합할 때 필요하다.

**구현 방향**:
- SPARQL 엔드포인트 설정 UI
- SPARQL 쿼리 에디터 (코드 하이라이팅)
- 결과 시각화 (테이블/그래프)
- 연합 쿼리 지원 (로컬 + 외부 엔드포인트)

### 4.7 대규모 그래프 성능 최적화

**왜 필요한가**: 현재 전체 온톨로지를 메모리에 로딩하고 ELK 전체 레이아웃을 재계산하는 구조는, 수천 노드 규모에서 한계에 도달한다.

**구현 방향**:
- 가상화 렌더링: 뷰포트 내 노드만 렌더링
- 증분 레이아웃: 변경된 부분만 재배치
- WebGL 렌더링 (React Flow Pro 또는 커스텀)
- 서버사이드 페이지네이션 (전체 로딩 → 온디맨드 로딩)
- 클러스터링: 하위 트리를 접힌 노드로 표현

## 5. 도메인별 온톨로지 템플릿 제안

### 5.1 반도체 제조 (현재 타겟 도메인)

```
Equipment (장비)

├── WetProcess (습식 공정 장비)
│   ├── WetStation, CMP, Cleaner

프로퍼티: model_name, vendor, install_date, pm_cycle, fab_site, bay_location
제약조건: 모든 장비는 반드시 1개 Site에 위치, PM주기는 양의 정수
```

### 5.2 IT 인프라

```
System (시스템)
├── Server, Database, Application, Network
├── Service (마이크로서비스)

관계 타입: depends_on, deployed_on, connects_to, backed_by
프로퍼티: hostname, ip_address, os_version, owner_team, sla_tier
제약조건: 모든 Application은 최소 1개 Server에 deployed_on
```

### 5.3 조직 & 인사

├── Department, Team, Project

프로퍼티: name, email, role, hire_date, department
제약조건: 모든 Employee는 정확히 1개 Department에 belongs_to
```

### 5.4 의료 & 헬스케어

├── Disease, Symptom, Treatment, Drug, Procedure

관계 타입: treats, causes, indicates, prescribed_for, performed_by
프로퍼티: icd_code, drug_class, dosage_unit, contraindication
제약조건: 모든 Treatment는 최소 1개 Disease를 treats
```

### 5.5 공급망 & 물류

├── RawMaterial, Component, FinishedGood

프로퍼티: sku, unit_cost, lead_time_days, moq, origin_country

## 6. 경쟁 도구 벤치마킹

### 6.1 Protege (Stanford)

| 항목 | Protege | Ontology Studio (현재) | v3 목표 |
|------|---------|----------------------|---------|
| 대상 사용자 | 온톨로지 공학자 | 도메인 전문가 | 도메인 전문가 (변함없음) |
| 학습 곡선 | 매우 높음 (OWL 지식 필요) | 낮음 (자유 텍스트 입력) | 낮음 유지 |
| OWL/RDF 지원 | 완전 지원 (OWL 2 full) | 없음 | 기본 임포트/익스포트 |
| 추론 엔진 | HermiT, Pellet, FaCT++ | 없음 | 없음 (장기 로드맵) |
| 시각화 | OntoGraf 플러그인 (빈약) | React Flow (우수) | React Flow 강화 |
| LLM 연동 | 없음 | 핵심 기능 | 대폭 강화 |
| 협업 | WebProtege (기본) | 없음 | 장기 로드맵 |
| UX/UI | 2000년대 자바 데스크톱 | 모던 웹 | 모던 웹 (개선) |

**Ontology Studio의 차별점**: Protege는 온톨로지 전문가를 위한 도구이고, Ontology Studio는 도메인 전문가를 위한 도구이다. Protege의 표현력을 흉내낼 필요는 없지만, 기본적인 상호운용성(임포트/익스포트)은 확보해야 한다.

### 6.2 TopBraid Composer / EDG

| 항목 | TopBraid | Ontology Studio (현재) | v3 목표 |
|------|----------|----------------------|---------|
| 가격 | 고가 라이선스 | 무료/자체 호스팅 | 무료/자체 호스팅 |
| 엔터프라이즈 기능 | 거버넌스, RBAC, 감사 | 없음 | 기본 검증 |
| SHACL/SPIN | 완전 지원 | 없음 | 제약 조건 시스템으로 대체 |
| 그래프 DB 연동 | 다양한 트리플스토어 | Neo4j 전용 | Neo4j 전용 |
| 데이터 통합 | SPARQLMotion, ETL | LLM 파싱 | LLM + CSV/파일 파싱 |

**Ontology Studio의 차별점**: TopBraid는 엔터프라이즈 온톨로지 관리 도구이다. Ontology Studio는 "구축 경험"에 초점. TopBraid의 거버넌스 기능은 장기 로드맵에서 선택적으로 도입.

### 6.3 Palantir Foundry Ontology

| 항목 | Palantir Foundry | Ontology Studio (현재) | v3 목표 |
|------|-----------------|----------------------|---------|
| 철학 | 데이터 → 온톨로지 (데이터 중심) | 지식 → 온톨로지 (지식 중심) | 지식 + 데이터 양방향 |
| 온톨로지 정의 방식 | 코드 (Typescript/Python) | 비주얼 그래프 + LLM | 비주얼 + LLM 강화 |
| 데이터 연결 | 파이프라인 기반 (대규모) | 단건 입력 | 대량 파싱 + CSV |
| Action/Function | 온톨로지 기반 API 자동 생성 | 없음 | 자연어 쿼리 |
| 검색 | 벡터 + 키워드 + 온톨로지 | Explorer 텍스트 필터 | 벡터 검색 (장기) |
| 스케일 | 수십억 객체 | 수백~수천 | 수천 (v3 최적화) |

**Ontology Studio의 차별점**: Foundry는 데이터가 먼저 있고 온톨로지를 그 위에 씌우는 접근이다. Ontology Studio는 지식이 먼저 있고 그것을 구조화하는 접근이다. 이 차이는 유지하되, Foundry의 장점인 "데이터 연결의 용이성"과 "품질 검증 파이프라인"을 참고해야 한다.

### 6.4 종합 포지셔닝

```
                    표현력 (OWL 완전성)
                         ↑
                         │
            Protege ●    │    ● TopBraid
                         │
                         │
                         │         ● Foundry
                         │
         Ontology Studio │
              (v3) ●     │
                         │
              ──────────────────────→ 접근성 (비전문가 사용 가능)
                         │
```

Ontology Studio v3의 포지션은 **"적절한 표현력 + 최고의 접근성"**이다. Protege/TopBraid의 표현력 전쟁에 참여하지 않되, 도메인 전문가가 실질적으로 필요한 수준의 온톨로지 모델링은 지원한다. LLM을 통해 접근성에서 압도적 우위를 유지하는 것이 전략적 핵심이다.
