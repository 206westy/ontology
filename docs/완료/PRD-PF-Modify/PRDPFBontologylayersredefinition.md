# PRD-PF-B — 온톨로지 레이어 재정의: Semantic / Kinetic(=Function) / 그리고 "Dynamic" 처리

- **스코프**: 온톨로지의 레이어 개념을 재정의한다. (1) Semantic 레이어 명확화, (2) Kinetic 레이어를 라벨에서 **일급 결정 함수(Function)** 로 승격, (3) 추상적 "Dynamic 레이어" 도입 여부를 명시적으로 결정. 실행엔진의 완성 구현·이벤트 자동화·상태머신 상세는 후속 PRD로 위임(경계 명시).
- **의존 PRD**: PRD-PF-A (온톨로지 컨테이너 — 멀티 온톨로지·테넌시·RLS). 본 PRD의 모든 요소는 특정 온톨로지 컨테이너에 귀속된다.
- **연관 PRD**: PRD-PF-I (이벤트-트리거 자동화, 후술), PRD-PF-A (보안/권한).
- **작성일**: 2026-07-12
- **원칙**: 비판적 옹호 · 재배치 우선(신규 최소화) · shadcn · 한국어 · HITL 컨펌 · 시각언어 특색 보존(클래스=원, 인스턴스=점, is-a=파선, 관계=실선).

## 결정 반영 (2026-07-12 확정)

- **D2 확정 = (A) 별도 `functions`+`decision_results` 테이블** (constraints.kind 확장 아님). 무결성 제약과 결정 함수의 목적 정반대성을 스키마에 각인.
- **D1 확정 = Dynamic 레이어 미도입**. 보안=PF-A, 상태 라이프사이클·이벤트 자동화=PF-I로 분해.
- **D7 확정 = 함수 저작 "둘 다"(2-tier)** — 아래 §4-저작 모델을 이렇게 재정의한다:
  - **Tier 1 (도메인 전문가·기본): 자연어 → 선언적 AST.** AI가 자연어 규칙을 화이트리스트 연산자 AST로 초안 → HITL 컨펌. 부작용 0, 결정론, 완전 감사가능. 대다수 결정 함수는 여기서 끝난다.
  - **Tier 2 (FDE·파워유저): 샌드박스 코드(TypeScript).** 선언적 AST로 표현 안 되는 복잡 로직(다단계·외부 통계엔진 호출 등)용. **전제: 격리 실행(sandbox) — 네트워크·파일·전역 접근 차단, CPU/메모리·타임아웃 상한, 결정론 강제, 입력=속성 스냅샷/출력=판정만.** 코드 함수는 `functions.impl_type='code'`로 표시하고 Critic·리뷰 게이트를 Tier1보다 강하게.
  - 두 티어 모두 **출력 계약 동일**(판정: 통과/불통과/스코어/추천 + 근거), 실행결과는 `decision_results`에 동일 적재. UI는 티어를 뱃지로 구분.
  - **보안 리스크 상향**: Tier2 도입으로 코드 실행 안전이 최상위 리스크가 됨 → §5 리스크·§6 수용기준에 "샌드박스 탈출 불가·리소스 상한·결정론" 계약 테스트 필수.

## 0. 한 줄 요지

온톨로지는 **Semantic(무엇이 존재하는가)** 과 **Kinetic(무엇을 계산·판정하는가)** 두 레이어로만 정의하며, Kinetic의 심장을 "kinetic 라벨"이 아니라 **속성을 읽어 판정을 산출하는 일급 결정 함수(Function)** 로 삼는다. 흔히 말하는 "Dynamic 레이어"는 별도 레이어로 만들지 않고 보안·상태 라이프사이클·이벤트 자동화 3개 구체물로 분해한다.

---

## 1. 목적

### 1.1 문제

현재 코드는 **단일 전역 그래프**이며, 온톨로지의 "동적 측면"이 이름만 있고 실체가 없다.

- `relation_types.layer = 'kinetic'` 은 **표시 라벨**일 뿐 실행 로직이 아니다. "원인→결과" 엣지를 그린다고 값이 계산되지는 않는다.
- `constraints`(kind=enforced|memo)는 **무결성 검증 전용**이다. cyclic/orphan/cardinality 위반을 `validation_results`에 severity로 적재하는, 즉 "모델이 규칙에 맞는가"를 본다.
- Critic 9룰(`critic/review.ts`)은 **모델 품질검수**다. "이 온톨로지 설계가 좋은가"를 본다.
- **★값을 읽어 통과/불통과/스코어를 계산하는 함수형 실행엔진이 코드에 전무하다.★**

즉 우리 제품은 "웨이퍼의 결함밀도가 임계 이상이면 불량"이라는 **도메인 결정을 온톨로지 위에서 실행할 수단**이 없다. 이것이 팔란티어 온톨로지가 semantic 위에 얹는 **kinetic(actions/functions)** 의 부재다.

### 1.2 전략 근거 (반드시 반영)

- 팔란티어 온톨로지 = **semantic elements**(objects/properties/links) + **kinetic elements**(actions/functions/dynamic security). 공식 정의는 **2종뿐**이며 dynamic security는 kinetic **하위** 요소다.
- kinetic의 핵심은 **실행 가능한 Function**(속성을 읽어 → 로직 → 판정/writeback)이지 라벨 엣지가 아니다.
- 무결성 제약(integrity constraint)과 결정 함수(decision function)는 기술적 사촌(둘 다 규칙 평가)이나 **목적이 정반대**다: 전자는 "모델이 유효한가"를 막고, 후자는 "이 인스턴스가 어떤 판정인가"를 계산한다.
- 팔란티어는 함수를 **TypeScript로 코딩**한다. 우리의 차별점은 **자연어 규칙 → 조건식 AI 초안 + HITL 컨펌**이다.
- "세 번째 dynamic 레이어"는 **블로거·커뮤니티 해석이지 공식이 아니다.**

### 1.3 본 PRD가 하는 일

1. Semantic 레이어를 **변경 없이** 명확히 경계 짓는다.
2. Kinetic 레이어를 **일급 결정 함수(Function)** 개념으로 재정의하고, 데이터 모델 2안을 비교·권고하며, 순수·결정론·감사가능한 실행엔진을 **설계**한다(구현 상세는 마일스톤/후속).
3. "Dynamic 레이어" 도입 여부를 **명시적으로 결정**한다 — 도입하지 않음을 권고하고, 대체 분해안을 제시한다.

## 4. 방향 (마일스톤)

- **M0 — 용어·경계 확정 (문서)**: Semantic/Kinetic 2레이어, "Function=결정 함수" 용어 일원화. `relation_types.layer='kinetic'` 라벨의 위상 재정의(§5.5). Dynamic 미도입 결정.
- **M1 — 데이터 모델 채택 + 마이그레이션**: §5.2의 2안 비교 → 1안 확정 → 스키마 스케치 + Drizzle 마이그레이션.
- **M2 — 실행엔진 MVP**: 단일 클래스, 속성 N개 입력 → 조건식 평가 → {통과|불통과|스코어|추천} 산출. 순수·결정론·감사(입력 스냅샷 해시 기록). 결과를 판정 결과 테이블에 적재.
- **M3 — 자연어→조건식 AI 초안 + 컨펌카드**: 자연어 규칙 입력 → 구조화 조건식 초안 → 컨펌카드 → 승인 시 함수 저장. Critic 검수 훅.
- **M4 — 시각화 + 감사 UI**: 함수 노드/뱃지 그래프 표현, 판정 결과 조회(인스턴스별 최근 판정, 근거 속성값 스냅샷).
- **M5 — 경계 확인**: 상태 라이프사이클·이벤트 자동화는 본 PRD 밖임을 재확인, 후속 PRD 티켓 발행.

## 5. 방법론 (설계 · 데이터 모델 · 리스크)

### 5.1 레이어 재정의 요약

| 레이어 | 질문 | 우리 요소 | 상태 |
|--------|------|-----------|------|
| **Semantic** | 무엇이 존재하는가 | `classes`(is-a), `properties`, `instances`, `instance_values`, `edges`, `relation_types` | **변경 없음** |
| **Kinetic** | 무엇을 계산·판정하는가 | ★신규★ **Function(결정 함수)** — 속성 읽어 판정 산출 | **본 PRD의 심장** |
| ~~Dynamic~~ | (별도 레이어 아님) | 보안·상태·이벤트로 **분해** (§5.6) | **미도입 권고** |

#### 무결성 제약 vs 결정 함수 — 목적 구분 (핵심 표)

| 항목 | 무결성 제약 (integrity constraint) | 결정 함수 (decision function) |
|------|-----------------------------------|-------------------------------|
| 코드 위치 | `constraints` (kind=enforced\|memo) | ★신규★ Function (kind=decision 또는 `functions`) |
| 질문 | "온톨로지 **모델**이 유효한가?" | "이 **인스턴스**가 어떤 판정인가?" |
| 입력 | 그래프 구조(순환/고아/카디널리티) | 인스턴스의 속성 값 |
| 출력 | 위반 여부 → `validation_results` severity | 판정: 통과/불통과/스코어/추천 |
| 위반 시 | 저장/발행 차단 또는 경고 | 차단 아님 — **판정 결과를 산출·기록** |
| 시점 | 모델 편집·발행 검증 | 데이터(인스턴스) 평가 |
| 비유 | 문법 검사기 | 계산기/심판 |
| 자동화 여부 | 검증은 자동 | 판정은 자동, **후속 액션은 컨펌**(정직 경계) |

> 이 둘은 **기술적 사촌**(규칙을 평가한다)이나 목적이 정반대다. UI·스키마·문서에서 절대 한 통에 섞지 않는다.

### 5.2 데이터 모델 — 2안 비교 및 권고

**공통 요구**: 함수는 (a) 대상 클래스, (b) 입력 속성 목록, (c) 로직(선언적 조건식/수식), (d) 출력 스펙(판정 종류·라벨·스코어 범위), (e) 자연어 원문, (f) 버전을 갖는다. 특정 온톨로지(PRD-PF-A)에 귀속.

#### 안 A — `constraints.kind` 에 `decision` 추가 (재배치 최대)

- 기존 `constraints(kind, constraint_type, config jsonb)`에 `kind='decision'` 추가, `config`에 입력/로직/출력 JSON.
- **장점**: 신규 테이블 0, 감사·조회 UI 즉시 공유, "규칙을 저장한다"는 개념적 근접성.
- **단점**: enforced/memo(무결성)와 decision(결정)이 **목적 정반대인데 한 테이블**에 공존 → §5.1 표의 구분이 스키마에서 흐려짐. `config` jsonb가 두 성격을 다 담아 쿼리·검증 복잡. 출력(스코어/추천)이라는 결정 함수만의 개념이 constraint 의미론과 충돌.

#### 안 B — 별도 `functions` 테이블 신설 (개념 정합)

```text
functions
  id            uuid pk
  ontology_id   uuid fk   -- PRD-PF-A 귀속
  name          text
  target_class  uuid fk classes
  inputs        jsonb     -- [{property_id, alias}]
  logic         jsonb     -- 선언적 AST(조건/수식) — 코드 아님
  output_spec   jsonb     -- {kind: pass_fail|score|recommend, labels, range}
  nl_source     text      -- 자연어 원문(차별점)
  status        text      -- draft|confirmed|archived
  version       int
  created_by / created_at / ...

decision_results   -- validation_results 형제 구조(재배치)
  id            uuid pk
  function_id   uuid fk functions
  instance_id   uuid fk instances
  verdict       jsonb     -- {pass|fail|score|recommendation}
  input_snapshot jsonb    -- 감사: 판정에 쓰인 속성값
  input_hash    text      -- 결정론 검증(동일 입력→동일 해시)
  evaluated_at  timestamptz
```

- **장점**: §5.1 표의 목적 구분이 **스키마에 각인**됨. output_spec(스코어/추천)이 자연스러움. `nl_source`로 자연어 차별점을 일급 저장. `decision_results`는 `validation_results` 구조를 복제·재배치하여 UI 공유하되 의미 오염 없음.
- **단점**: 테이블 2개 신규(마이그레이션·RLS 추가).

#### 권고: **안 B (별도 `functions` 테이블)**

이유: 무결성 제약과 결정 함수는 **목적이 정반대**(§5.1)이며, 이를 같은 테이블에 넣으면 우리가 문서로 애써 구분한 개념이 데이터 층에서 다시 뭉개진다. 결정 함수만의 개념(출력=스코어/추천, 자연어 원문, 판정 결과 이력)이 constraint 의미론에 억지로 끼워지지 않는다. `decision_results`를 `validation_results`의 **형제 구조로 재배치**해 조회·감사 UI는 공유하므로 "재배치 우선" 원칙과도 부합한다. 마이그레이션: `functions`, `decision_results` 2개 테이블 + PRD-PF-A RLS 정책 상속.

> 비판적 옹호: 안 A가 "신규 0"으로 원칙에 더 맞아 보이지만, 원칙의 우선순위는 "재배치 우선"이지 "개념 오염 감수"가 아니다. 결과 저장은 이미 재배치한다. 순 신규 비용은 정의 테이블 1개뿐이며 그 대가로 개념 정합을 얻는다.

### 5.3 실행엔진 설계 — 순수 · 결정론 · 감사가능

- **순수(pure)**: 입력 = 인스턴스 속성 값 스냅샷 + 함수 정의. 부작용 없음(DB write는 엔진 밖 결과 적재기가 담당, 엔진은 판정만 반환).
- **결정론(deterministic)**: 동일 입력 → 동일 출력. `input_snapshot`를 해시화(`input_hash`)해 재현성 검증. 난수·현재시각·네트워크·전역상태 **참조 금지**(표현식 화이트리스트로 원천 차단).
- **감사가능(auditable)**: 모든 판정은 `decision_results`에 입력 스냅샷·함수 버전·타임스탬프와 함께 기록. "왜 이 판정인가"를 근거 속성값으로 재구성 가능.
- **로직 표현**: `logic`은 **선언적 AST**(비교/논리/산술/임계치)만. 자유 코드 실행 금지 → `eval`/`new Function` 금지, 화이트리스트 연산자만 평가. 이것이 팔란티어(TS 코딩) 대비 안전성·자연어 컨펌 흐름의 근거.
- **평가 흐름**: `instance_values` 조회 → alias 바인딩 → AST 평가 → `output_spec`에 맞춰 verdict 정규화 → 결과 반환 → 적재기가 `decision_results` write.

### 5.4 자연어 규칙 → 조건식 AI 초안 + HITL 컨펌 (차별점)

1. 사용자가 자연어로 규칙 입력: "결함밀도가 0.5 이상이면 불량".
2. AI SDK v6 + 구조화 출력(zod4)으로 `{inputs, logic(AST), output_spec}` **초안** 생성.
3. 초안은 즉시 저장되지 않고 **컨펌카드**로 렌더: 사람이 읽을 수 있는 조건식 + 근거 속성 + 예시 판정 미리보기.
4. Critic 훅(모델 품질검수)으로 "존재하지 않는 속성 참조", "임계치 단위 불일치" 등 점검.
5. 사용자가 **승인(컨펌)** 해야 `status=confirmed`로 승격. 정직 경계 준수: **완전 자동 금지 = 자동 초안 + 사람 컨펌**.

> 팔란티어는 TypeScript로 함수를 코딩한다. 우리는 **자연어 + 컨펌**으로 도메인 전문가가 코드 없이 결정 함수를 만든다 — 이것이 제품 북극성(스케치북→운영, AI=Critic)과 직결된 차별점이다.

### 5.5 `relation_types.layer` 라벨과 Function의 관계 정리

- `relation_types.layer='kinetic'` 라벨은 **서술(descriptive)**: "이 관계는 동적 성격이다"라는 사람용 태그일 뿐 실행되지 않는다.
- Function은 **실행(executable)**: 실제로 속성을 읽어 판정을 계산한다.
- 재정의: 라벨은 **문서적 힌트**로 유지하되, "kinetic"의 실체는 Function이 담당한다고 UI·문서에서 명시. 혼동을 막기 위해 라벨 툴팁에 "표시용 — 실행은 Function 참조" 문구 추가. (라벨 제거는 파괴적 변경이라 M0에서 보류, 재의미화만 수행 → §8 열린 결정.)

### 5.6 "Dynamic 레이어" 처리 — 미도입 권고 + 분해

**결정: 추상적 3번째 "dynamic 레이어"를 도입하지 않는다.** 근거: 팔란티어 공식 정의는 semantic + kinetic **2종뿐**이고 "dynamic security"는 **kinetic 하위** 요소다. "세 번째 dynamic 레이어"는 커뮤니티/블로거 해석이지 공식이 아니다. 추상 레이어를 만들면 실체 없는 개념 부채가 생긴다.

대신 흔히 "dynamic"으로 뭉뚱그려지는 것을 **3개 구체물로 분해**:

| # | 'dynamic'의 실체 | 우리 처리 | 담당 PRD |
|---|-----------------|-----------|----------|
| (a) | 보안/권한 (동적 접근제어) | 테넌시·RLS로 구현 | **PRD-PF-A** |
| (b) | 상태 라이프사이클 (객체 상태 전이: 웨이퍼 pending→pass/fail, 작업지시 open→closed) | **상태 속성 + 허용 전이 제약** 또는 경량 상태머신 | 본 PRD 인접(스케치만) / 후속 |
| (c) | 이벤트-트리거 자동화 (신규 데이터→함수 실행→액션 제안) | 이벤트 훅 → Function 실행 → **액션 제안(컨펌)** | **PRD-PF-I** |

> (b) 스케치: 상태는 하나의 속성(enum), "허용 전이"는 무결성 제약 또는 결정 함수로 표현 가능. 상세 상태머신은 본 PRD 범위 밖(경계 명시). (c)는 이벤트 인프라가 필요하므로 전면 위임.

### 5.7 특색 보존 — 함수의 시각화

- 함수는 그래프에 **함수 노드** 또는 대상 클래스(원)에 붙는 **함수 뱃지**로 표현. 기존 시각언어(클래스=원, 인스턴스=점, is-a=파선, 관계=실선)를 훼손하지 않고 **직교적으로 추가**.
- 함수 노드는 입력 속성 → 함수 → 판정으로 흐름을 시각화(방향 표시). Cytoscape 스타일 확장.
- 판정 결과는 인스턴스(점)에 상태 색/뱃지로 오버레이 가능(예: pass=녹, fail=적) — 단, 이는 표시일 뿐 SoT는 `decision_results`.

### 5.8 리스크

| 리스크 | 영향 | 완화 |
|--------|------|------|
| 함수 실행 안전(무한루프·부작용) | 서버 다운·데이터 오염 | 자유코드 금지, 선언적 AST만, 재귀 없는 평가, write는 엔진 밖 적재기로 분리 |
| 자연어→로직 정확도 | 잘못된 판정 | HITL 컨펌 필수, Critic 훅, 예시 미리보기, 승인 전 저장 금지 |
| 결정론 미보장 | 감사·재현 실패 | 시각·난수·네트워크 참조 금지(화이트리스트), input_hash 재현 검증 |
| 무결성 vs 결정 혼동 | 개념 부채 | 안 B(별도 테이블), §5.1 구분표, UI 분리 |
| Dynamic 레이어 재요구 | 개념 팽창 | §5.6 분해안 문서화, §8 선택지로 명시 |
| 함수 시각화가 그래프 오염 | 특색 훼손 | 직교 추가(뱃지/노드), 기존 시각언어 불변 |

## 7. 결론

온톨로지의 "동적 측면"은 지금까지 **라벨과 이름만** 있었다. 본 PRD는 그 실체를 **일급 결정 함수(Function)** 로 세운다 — 속성을 읽어 판정을 산출하는, 순수·결정론·감사가능한 실행 요소다. 무결성 제약(모델이 유효한가)과 결정 함수(인스턴스가 어떤 판정인가)를 목적으로 분리하고, 이를 위해 **별도 `functions` 테이블(안 B)** 을 권고하되 결과 적재는 `validation_results` 형제 구조로 재배치한다. 팔란티어가 TypeScript로 코딩하는 함수를, 우리는 **자연어 + HITL 컨펌**으로 도메인 전문가 손에 쥐어준다 — 이것이 제품의 차별점이다. "Dynamic 레이어"는 실체 없는 추상이므로 도입하지 않고 보안·상태·이벤트 3구체물로 분해한다. 정직 경계(자동 초안 + 컨펌, 완전 자동 금지)와 시각언어 특색은 전 구간 보존된다.

## 8. 열린 결정 / 불가 기능

### 8.1 열린 결정 (선택지 포함)

1. **데이터 모델 (본 PRD 권고: 안 B)**
   - (A) `constraints.kind='decision'` 확장 — 신규 0, 개념 오염 위험.
   - (B) ★권고★ 별도 `functions` + `decision_results` — 개념 정합, 테이블 2개 신규.
2. **"Dynamic 레이어" (본 PRD 권고: 분해안)**
   - (A) ★권고★ 미도입 + (a)보안/(b)상태/(c)이벤트 분해.
   - (B) 별도 dynamic 레이어 유지 — 실체 없는 추상, 개념 부채.
3. **상태 라이프사이클 (b) 구현 방식** — 열림
   - (A) 상태 enum 속성 + 허용 전이를 무결성 제약으로.
   - (B) 상태 enum 속성 + 허용 전이를 결정 함수로.
   - (C) 경량 상태머신 별도 도입(후속 PRD). — 본 PRD는 스케치만.
4. **`relation_types.layer='kinetic'` 라벨 최종 처리** — 열림
   - (A) ★M0 채택★ 유지하되 "서술용" 재의미화(파괴적 변경 회피).
   - (B) 향후 라벨 제거하고 Function 참조로 완전 대체(마이그레이션 비용).
5. **표현식 평가기 선택** — 열림: 자체 화이트리스트 AST vs 검증된 안전 표현식 라이브러리(샌드박스). 안전·결정론 요건 충족 전제.

### 8.2 불가 기능 (본 PRD 범위 밖 — 과대약속 금지)

- **완전 자동 판정→자동 조치**: 판정은 자동이나 **후속 액션은 컨펌 필수**. 자동 writeback/실행은 하지 않는다(정직 경계).
- **자유 코드(TypeScript 등) 함수 작성**: 안전·결정론·자연어 컨펌 흐름을 위해 선언적 AST만. 임의 코드 실행은 불가.
- **이벤트-트리거 자동화 인프라**: 신규 데이터 감지→함수 실행 파이프라인은 **PRD-PF-I**로 위임.
- **동적 접근제어(보안)**: 테넌시·RLS는 **PRD-PF-A** 소관.
- **완전한 상태머신 엔진**: (b)는 스케치만, 상세 전이 엔진은 후속.
- **자연어→로직 100% 정확 보장**: 초안 품질은 확률적이며 반드시 사람 컨펌으로 보정한다(무보정 신뢰 불가).
