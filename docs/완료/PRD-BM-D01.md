# PRD-BM-D01 · 패턴 마켓플레이스 — 학습 캐시를 성장 플라이휠로
 
> **시리즈**: 매일 벤치마킹 (Daily Benchmark) — Day 01
> **작성일**: 2026-07-09
> **카테고리**: 온톨로지/KG 전문 × AI/RAG × 노코드 성장전략 (교차)
> **벤치마크 대상**: Palantir Foundry **Marketplace**, Notion **Template Gallery**, Airtable **Universe**
> **한 줄 요지**: 이미 코드에 존재하는 `patterns`(학습형 패턴 캐시)를 **내부 최적화 자산**에서 → **공유·재사용·성장 플라이휠**로 격상시킨다. 신규 엔진을 만드는 게 아니라, 있는 자산의 *비즈니스 표면*을 여는 일.

## 0. 왜 오늘 이 주제인가 (사업 관점)
 
Ontology Studio의 가장 큰 사업적 난제는 **콜드 스타트(cold start)** 다. 도메인 전문가가 빈 캔버스 앞에서 "무엇부터 만들지"를 마주하는 순간이 이탈률이 가장 높은 지점이고, SaaS 활성화(activation) 곡선을 결정한다. README의 시퀀스도 이미 이 문제를 **"② 패턴 확보 — 공개 온톨로지 검색→적응→합성"** 으로 풀고 있다. 즉 *기술적 해법은 이미 코드에 있다* (`patterns` 테이블, `/api/llm/discover-pattern`, retrieve(LOV)›adapt›synthesize).
 
문제는 이 자산이 **개인 사용자 내부에 갇혀 있다**는 것. 벤치마크 3社의 공통 교훈은 명확하다: **재사용 가능한 "시작점"을 공유 자산으로 만들면, 그것이 활성화·바이럴·수익화를 동시에 푸는 단일 지렛대가 된다.**
 
| 플랫폼 | 재사용 단위 | 사업적 효과 |
|--------|-------------|-------------|
| **Palantir Foundry Marketplace** | 패키지 제품(오브젝트 타입·링크·액션·함수·파이프라인을 통째로) | 팀·인스턴스 간 온톨로지 **재사용**, 표준화, 배포 시간 단축. 설치 시 "입력 매핑"만 하면 즉시 가동 |
| **Notion Template Gallery** | 워크스페이스 템플릿 | 광고비 0원으로 20M(2021)→100M+ MAU(2024). 템플릿이 활성화 엔진 + 크리에이터 이코노미 + 앰버서더 바이럴 루프의 3중 플라이휠 |
| **Airtable Universe** | 베이스(스키마+뷰) | 커뮤니티가 도메인별 스키마를 공유 → 신규 유저의 "빈 페이지" 장벽 제거 |
 
> **통찰**: Palantir은 온톨로지를 *패키지 상품*으로 판다. Notion은 템플릿을 *성장 루프*로 쓴다. 우리는 이미 **두 가지의 재료(학습된 패턴 + 출처·라이선스 메타데이터)를 코드에 갖고 있다.** 이 PRD는 그것을 제품 표면으로 끌어올리는 실행안이다.

## 1. 목적 (Purpose)
 
Ontology Studio의 `patterns` 자산을, 세 층위로 열어 **활성화율·재방문율·(향후) 수익화**를 동시에 끌어올린다.
 
1. **Consume(소비)** — 신규/기존 사용자가 검증된 패턴을 "새 구획으로 시딩"하며 시작한다. → *콜드 스타트 제거, time-to-first-graph 단축.*
2. **Contribute(기여)** — 사용자가 자기 도메인에서 수렴시킨 패턴을 공유 카탈로그로 발행(publish)한다. → *네트워크 효과의 공급 측.*
3. **Curate(큐레이션)** — 출처·라이선스·사용 빈도(occurrence)·헬스 점수로 신뢰 가능한 패턴만 표면화한다. → *Critic 철학의 마켓플레이스 버전.*

## 2. 목표 & 성공지표 (Goals / North-Star Metrics)
 
| 축 | 지표 | 현재 | 목표 방향 |
|----|------|------|-----------|
| 활성화 | **첫 그래프까지 시간(TTFG)** — 가입→첫 커밋 | 빈 캔버스에서 자유 입력 의존 | 패턴 시딩 경로로 **↓** |
| 활성화 | 패턴으로 시작한 세션 비율 | 측정 안 됨 | 계측 시작 → **↑** |
| 유지 | 패턴 재사용 세션의 D7 재방문율 | — | 자유입력 세션 대비 **↑** 여부 검증 |
| 공급 | 발행된 공유 패턴 수 / 기여자 수 | 내부 캐시만 존재 | **0 → 유의미** |
| 신뢰 | 패턴별 provenance·occurrence 노출률 | 데이터는 있음, 표면 없음 | **100% 표면화** |
 
> **선결 과제(v6 로드맵 계승)**: 성능을 숫자로 말하려면 계측이 먼저다. M0에서 패턴 사용 이벤트 로깅부터 심는다.

## 3. 기술 스택 & 재사용 자산 (Tech Stack)
 
신규 인프라 최소화. **기존 스택 위에서 표면만 추가**한다.
 
- **프론트**: Next.js(App Router), shadcn/ui, Lucide, Tailwind/CSS 변수 — 기존 `<ConfirmCard>`·EmptyState·`BridgeSuggestCard` 컨벤션 준수, 이모지 금지, 한국어 UI.
- **데이터(SoT)**: Supabase — 기존 `patterns` 테이블 확장(공유 스코프 컬럼) + `term_glossary`/`relation_glossary` 메타 연계.
- **AI**: AI SDK — 패턴 명명·요약·적응은 기존 `discover-pattern` 파이프라인(retrieve›adapt›synthesize) 재사용.
- **발행 대상**: 패턴 시딩은 스테이징(Supabase)에만 반영 → 기존 커밋/푸시(Neo4j) 흐름과 호환. **AI 자동 확정 없음(항상 HITL).**
- **거버넌스**: 출처(source_type)·라이선스 경고(PRD-H)·occurrence_count·헬스 점수(HealthScoreBadge) — 이미 존재하는 신뢰 신호를 마켓플레이스 카드에 그대로 노출.
> **핵심 방침**: "새로 만드는 것보다 재배치+통합"(v6 §7 계승). 이 PRD의 신규 코드는 대부분 *조회 API + 카탈로그 UI + 발행 액션*이며, 지능 로직은 전부 기구현 자산 호출이다.

## 4. 방향 (Direction) — 3-Milestone 로드맵
 
### M0 · 계측 & 시딩 경로 노출 (기반)
- 패턴 사용 이벤트 로깅: `pattern_seeded`, `pattern_source`(cache/discovered/shared), TTFG 타임스탬프.
- EmptyState "패턴으로 시작"(PRD-H 기구현)에 **로컬 캐시 패턴 목록**을 카드로 노출 + "새 구획으로 시딩"(PRD-N M1 옵션과 정합).
- **Out**: 공유·발행 없음. 순수 내부 자산 표면화 + 계측.
### M1 · 공유 패턴 카탈로그 (소비 측 네트워크 효과)
- `patterns`에 공유 스코프(`visibility: private|org|public`) + 큐레이션 메타(occurrence, health, license) 추가.
- **패턴 브라우저**: 도메인·출처·사용빈도로 필터되는 카탈로그 시트. 각 카드 = 이름·도메인·출처 배지·라이선스·"이 패턴으로 시작" CTA.
- 시딩 시 기존 adapt 파이프라인으로 사용자 맥락에 맞게 자동 적응(억지 이식 금지) → HITL 컨펌.
### M2 · 패턴 발행 & 큐레이션 (공급 측 플라이휠)
- 사용자가 수렴시킨 패턴을 **"공유 패턴으로 발행"**(라이선스·출처 확인 게이트 통과 시). Palantir Marketplace의 "입력 매핑 후 배포" 모델 벤치마크 → 발행 전 민감 식별자 마스킹(PRD-H `web-verify` 마스킹 자산 재사용).
- 큐레이션: occurrence·health 임계 이하 패턴은 카탈로그에서 dim/하단 정렬(Critic의 마켓플레이스 적용).
- **Out(후순위)**: 유료 패턴 판매, 크리에이터 정산 — 수익화는 공급이 임계에 도달한 뒤 별도 PRD.

## 5. 방법론 (Methodology) — 벤치마크 → 우리 제품 매핑
 
| 벤치마크 교훈 | 출처 | 우리 실행 |
|---------------|------|-----------|
| 온톨로지를 "설치 가능한 패키지"로 → 입력 매핑만으로 즉시 가동 | Palantir Marketplace | 패턴 시딩 = adapt 파이프라인이 "입력 매핑"을 자동 수행. 사용자는 컨펌만 |
| 템플릿이 "빈 페이지" 장벽을 제거해 활성화 엔진이 됨 | Notion/Airtable | EmptyState를 자유입력이 아니라 **패턴 우선** 진입으로 |
| 공유+수익화가 공급자의 시간·자부심 투자를 유발(플라이휠) | Notion 크리에이터 이코노미 | M2 발행 → (후속) 조직 내/공개 공유. 수익화는 공급 임계 후 |
| 신뢰 없는 재사용은 오히려 오염 | Palantir 거버넌스 | occurrence·health·provenance·license를 카드에 상시 노출 (Critic 철학) |
 
**리스크 & 대응**
- *저품질 패턴 범람* → 큐레이션 임계(occurrence/health) + 출처 게이트. 자동 발행 없음.
- *라이선스 위반* → 발행 전 라이선스 경고(PRD-H 기구현) 필수 통과.
- *과잉 적응(원 패턴 왜곡)* → 시딩은 항상 HITL diff 프리뷰, 원본 패턴 불변.

## 6. 수용 기준 (Acceptance) — M0 우선
 
- [ ] 신규 세션이 EmptyState에서 로컬 캐시 패턴을 카드로 보고, 1클릭으로 새 구획에 시딩할 수 있다.
- [ ] `pattern_seeded` 이벤트와 TTFG가 계측되어, "패턴 시작 vs 자유입력"의 활성화 차이를 숫자로 비교할 수 있다.
- [ ] 각 패턴 카드에 출처·라이선스·사용빈도가 노출된다(신뢰 신호 100% 표면화).
- [ ] 시딩은 adapt 파이프라인을 거쳐 HITL 컨펌으로만 반영된다(자동 확정 없음).
