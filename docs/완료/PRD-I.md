# Ontology Studio — PRD-I: 코드베이스 통합 & 디자인 통일 (Integration & Design Unity)

> **문서 성격**: 목적 · 기능 · UX 설명 · UI 특징(마이그레이션 지향) 중심. **코드·구현·파일 구성·스키마는 제외**(구현은 Claude Code). 기존 컴포넌트를 **교체가 아니라 개선·재호스팅**하는 관점으로 서술 — 기존 컴포넌트명은 마이그레이션 기준점으로만 참조.
> **버전**: I-1.0 · 2026-07-01
> **선행/기반**: PRD-H-2(패턴-적응 기획) + 현행 코드베이스 UX 분석(3-패널 · NewNodePopover · CommitBar · RightPanel · EntityResolutionSheet · NeoConfirmSheet · GraphCanvas). PRD-H-2의 추상 설계를 **실제 코드베이스 위에 얹는 통합본**.
> **대전제**: 현재 구현은 이미 강한 HITL primitives를 갖고 있다(dedup 배지 · 거버넌스 제안 · Critic 검수 · 보강 소싱). **이걸 다시 만들지 않는다.** 재사용·재배치하고, 부족한 패턴-적응 레이어와 상태 프레이밍만 얹는다.

---

## 0. 목적

**왜 지금 이 작업인가.** 현재 강점과 부채가 공존한다.

- **강점(보존)**: 생성 시점 `NewNodePopover` 하나에 dedup(reuse/relate/possible_duplicate/new)·거버넌스 제안(제약·공리·enum·cardinality 승인/무시)·보강 소싱(웹 opt-in)·Critic 자문이 이미 HITL로 구현돼 있다.
- **부채 ①(해소)**: 그 모든 결정이 360px 팝오버 한 곳에 집중돼 **과적재**다. 항목이 늘면 인지 부하가 급증한다.
- **부채 ②(추가)**: 도메인 인지·패턴 시드·스키마 적응(확장/분기)·크로스-구획 브릿지·통합 상태 머신이 **아직 없다**(단일 제네릭 parse).

**목표**: (1) 디자인 통일을 유지하며, (2) 집중형 팝오버는 빠른 편집용으로 보존하고, (3) 문서→도메인 온톨로지 ingestion을 위한 **가이드 여정 모드**를 추가해 과적재를 분산하며, (4) 패턴-적응 레이어와 초안→확정→발행 상태 프레이밍을 얹는다. **교체 0, 확장 위주.**

## 1. 디자인 통일 원칙 (최우선)

새 표면이 기존 화면과 한 몸으로 보이게 하는 규칙. 이 절이 본 PRD의 척추다.

1. **토큰 단일화.** 기존 shadcn CSS 변수 체계(oklch · light/dark · `--radius` 스케일 · semantic 토큰: background/card/popover/primary/muted/accent/destructive/border/ring)를 그대로 쓴다. **새 팔레트 하드코딩 금지.** 온톨로지 역할색(증상·원인·점검·조치·부품·행정)은 차트 계열 토큰 + 전용 `--role-*` 토큰으로 정의해 **라이트/다크 자동 대응**시킨다(SVG·배지·엣지 모두 동일 토큰 참조).
2. **primitive 재사용.** 신규 표면도 기존과 동일한 shadcn primitive만 쓴다 — Popover · Sheet · Card · Badge · Tabs · Command · Sidebar. **새 UI 라이브러리 도입 금지.** 아이콘은 `lucide-react` 단일 소스 유지(다른 아이콘·인라인 SVG 아이콘 금지).
3. **관습 재사용.** 팝오버 관습(트리거 근처 등장 · Esc/외부클릭 닫기 · 최대 360px · radius 12 · shadow-lg · 헤더 드래그 이동), CommitBar 배지 색규칙(ADD/MOD/DEL), Sheet 바텀시트(변경내역·NeoConfirm·EntityResolution) 패턴을 신규 표면이 그대로 계승한다.
4. **컨펌 카드 단일 문법.** 기존/신규 모든 HITL이 **하나의 카드 해부구조와 배지 taxonomy**를 공유한다(§3). 거버넌스 카드·dedup 배지·Critic·보강·도메인·패턴·드리프트·브릿지·용어가 전부 같은 시각 언어로 읽힌다.

## 2. 두 진입 모드 (Quick / Guided)

| | Quick — 빠른 추가 | Guided — 문서→도메인 온톨로지 |
|---|---|---|
| 용도 | 노드 1~2개 즉석 추가 | 대량 입력·새 도메인 ingestion |
| 표면 | 기존 `NewNodePopover`(변경 없음) | 여정 스테퍼 + 컨펌 독 |
| 컨펌 | 팝오버 프리뷰 한 번 | 단계별 카드로 분산 |
| 상태 | 강점 보존 | 과적재 해소 |

- **모드 전환**: 입력량·의도로 자동 제안하되(짧은 입력→Quick, 문서/CSV/대량→Guided 제안), **최종 선택은 사용자**(팝오버·독 상단 토글). 강제 없음.
- Guided는 새 로직 덩어리가 아니라, **Quick 팝오버가 한 화면에 쌓아둔 결정들을 시간축으로 펼친 것**이다. 내부 카드는 대부분 재사용.

## 3. 공통 컨펌 문법 (전 표면 공유)

**카드 해부구조(4단, 고정 순서)**
1. `AI 판정` — 무엇을 감지/제안했는지 (eyebrow: 단계/유형)
2. `근거` — 출처 · 신뢰도 · 원문 근거 스팬 (투명성, 항상 노출)
3. `미리보기` — 이 결정이 캔버스/그래프에 미치는 효과
4. `액션` — 주 액션(권장) + 대안 + 건너뛰기. 액션 라벨 = 결과 동사(예 "통합 승인"→"통합됨").

**배지 taxonomy(색=판정 강도, semantic 토큰 매핑)**
- 동일/reuse · 연관/relate · 중복가능/possible_duplicate · 신규/new (기존 dedup 계승)
- 확장/extend · 분기/fork · 통과/pass · 차단/block (신규 추가)
- 각 배지에 confidence 병기. 애매(주의색)일수록 사용자가 더 신중히 보게.

이 문법을 기존 거버넌스 카드·dedup 배지 스타일을 **정규화**해 만든다(신규 시각 언어 창작 아님).

## 4. 기능 (Feature) — 목적 · 기능 · UX · UI 특징(마이그레이션)

### F1. 통합 상태 머신 — 초안 → 확정 → 발행
- **목적**: 지금 분산된 상태(커밋=CommitBar, 발행=NeoConfirmSheet)를 하나의 가시적 상태로 통일.
- **기능**: 3-state 세그먼트 + 변경 카운트 + 검증 칩(CQ/연결). 각 상태 클릭 시 해당 상세로.
- **UX**: "지금 내 온톨로지가 초안/확정/발행 중 어디인가"를 한눈에. 초안=미확정임을 상시 인지.
- **UI 특징(마이그레이션)**: 기존 **CommitBar**(ADD/MOD/DEL 카운트·되돌리기·변경내역·커밋·푸시)를 **상단 상태 스트립으로 승격**하되 버튼·배지 색규칙 유지. **NeoConfirmSheet**(Cypher 프리뷰·진행률·롤백)는 "발행" 단계의 상세 시트로 그대로 연결. 신규 컴포넌트 최소, 재배치 위주.

### F2. 도메인 요약 & 패턴 선택 (Guided 진입)
- **목적**: 단일 제네릭 parse → 도메인 인지 후 맞는 패턴을 시드로.
- **기능**: 입력 분석 → 도메인·신뢰도·혼합 비율·CQ 미리보기 → 패턴 선택([이 패턴]/[다른]/[혼합→분할]).
- **UX**: "이 문서는 진단 82% — 진단 패턴으로 만들까요?" 컨펌 후 시작.
- **UI 특징(마이그레이션)**: 기존 `NewNodePopover`의 **로딩 5단계**(텍스트 파싱→엔티티→관계→매칭→계층)를 앞단에 "도메인 인지" 단계로 **확장**하고, 프리뷰 진입 전 도메인 요약 카드를 삽입. Quick 모드에선 팝오버 내, Guided에선 독으로.

### F3. 패턴 발견 & 학습 캐시
- **목적**: 패턴을 미리 넣지 않고 런타임에 발견 + 재사용으로 수렴.
- **기능**: 캐시 조회 → miss 시 공개 온톨로지 검색(retrieve)→적응→(없으면)합성. 출처 노출. 확정 시 캐시 승격.
- **UX**: "저장소에서 X 온톨로지 발견 → 당신 데이터에 적응 → [이걸로]" + 출처 투명 + "다음엔 재사용" 안내.
- **UI 특징(마이그레이션)**: 기존 **EmptyState의 도메인 템플릿 5종**을 패턴 레지스트리로 **승격**(정적 시작점 → 동적 발견 캐시). 발견 카드는 기존 **보강 소싱(웹 opt-in) UI 패턴**을 재사용 — 출처·신뢰도 표기 관습 동일.

### F4. 가이드 여정 & 컨펌 독 (과적재 해소의 핵심)
- **목적**: 팝오버에 뭉친 결정을 캔버스 위 단계별로 분산해 인지 부하를 낮춘다.
- **기능**: 좌측 **여정 스테퍼**(단계 진행·완료 체크) + 우측 **컨펌 독**(현재 단계 카드). 기존 dedup·거버넌스·Critic·보강 카드를 독으로 재호스팅.
- **UX**: 한 번에 하나의 결정, 근거와 함께, 캔버스 반영을 즉시 확인. 사용자는 컨펌만.
- **UI 특징(마이그레이션)**: 여정 스테퍼는 shadcn **Sidebar**(collapsible; 모바일 자동 Sheet 오버레이)로 구현하고, 좌측 영역을 **Explorer와 모드 전환**으로 공유(수동 편집=Explorer 트리, Guided=여정). 컨펌 독은 기존 **RightPanel(탭 구조)을 겸용**하거나 Sheet. 팝오버 내부 카드(거버넌스 승인/무시, dedup 배지, Critic 자문)를 **독립 카드 컴포넌트로 추출**해 팝오버·독 양쪽에서 재사용 — 신규 로직 아님, 재배치.

### F5. 라이브 캔버스 조립
- **목적**: "로딩 후 일괄 생성"의 블랙박스를 "만들어지는 게 보이는" 경험으로.
- **기능**: 추출 진행 중 노드·관계가 캔버스에 점진 렌더. 병합·해소가 실시간 반영.
- **UX**: 신뢰·이해도 상승 — 무엇이 왜 생기는지 보임.
- **UI 특징(마이그레이션)**: 기존 **GraphCanvas**(React Flow + ELKjs 자동 레이아웃) + `focusNode` 하이라이트 + LOD 줌을 재사용하되, 확정 시 일괄 생성 대신 **스트리밍 partial을 순차 반영**. 기존 5단계 로딩 인디케이터의 "진행 표시" 개념을 진행형 렌더로 계승. ELKjs는 점진 추가에 맞춰 incremental relayout(전면 재배치로 튀지 않게).

### F6. 맥락 주입 용어 해소
- **목적**: 미정의·모호 용어(예 VV)를 **이 도메인에서의 뜻**으로 정확히.
- **기능**: 감지 → 도메인+인접 노드 맥락을 주입한 질의로 해소 → 후보 → 확정 → 노드 정의·용어집에 재주입. **도메인 스코프**(전역 강제 안 함).
- **UX**: "VV — 이 맥락(반도체 설비 메인트)에선 밸브 92%" 확정 → 이후 추출·검색에 재주입.
- **UI 특징(마이그레이션)**: 기존 **보강 소싱**(미정의 개념·웹 opt-in·기본 꺼짐) 흐름을 그대로 쓰되, 결과를 "개념 정의"가 아니라 "**용어 sense 해소(도메인 스코프)**"로 프레이밍하고 확정 뜻을 세션/도메인 용어집으로 승격. 카드 UI는 기존 보강 카드와 **동일 스타일**.

### F7. 스키마 드리프트 결정 — map / extend / fork
- **목적**: 패턴에 없는 새 요소를 확장할지·분기할지 판정하고 사용자 컨펌.
- **기능**: 3분기 판정(기존 매핑 / 패턴 확장 / 새 구획 분기) + 권장 배지 + 각 선택 미리보기.
- **UX**: "행정 승인 흐름 감지 — 현재 패턴 밖. 새 구획으로 분리?(권장)".
- **UI 특징(마이그레이션)**: 기존 **거버넌스 제안 카드**(승인/무시 + evidence + 배지)의 UI 문법을 그대로 **재사용**해 3-액션 결정 카드로. 새 시각 언어 없음.

### F8. 크로스-구획 브릿지
- **목적**: 데이터엔 있으나(`isBridge`) UX가 없는 브릿지를 표면화.
- **기능**: 여러 구획에 등장하는 동일 대상을 브릿지 후보로 제안 → 컨펌. 캔버스에 구획 레인 + 브릿지 엣지 시각 구분.
- **UX**: "펌프447이 [메인트]·[행정] 양쪽에 — 브릿지로 연결?".
- **UI 특징(마이그레이션)**: 기존 **EntityResolutionSheet**(possible_duplicate 병합 UI) 패턴을 브릿지 제안에 재사용. 캔버스는 기존 엣지 3종(is-a·instance-of·relation) 체계에 **브릿지 엣지 1종 추가**(점선+역할색), 구획은 배경 레인/그룹으로 구분 — 다구획에서 hairball로 안 보이게.

### F9. 6요소 인스펙터 정리 + 근거 노출
- **목적**: "구성요소 최종 뷰" 완성 + 어트리뷰션 가시화(투명성).
- **기능**: 노드/관계의 6요소를 속성·제약·근거·관계로 정리. 각 항목의 출처·evidence·confidence 노출.
- **UX**: 무엇이 어디서 왔는지(문서/사용자/웹/추론) 한 탭에서 확인.
- **UI 특징(마이그레이션)**: 기존 **RightPanel 섹션**(description·subclasses·properties·constraints·instances·relations)을 **속성·제약·근거·관계 4탭으로 재편**(대분류만 변경, 섹션 위젯 재사용). 어트리뷰션은 이미 데이터 모델에 신설돼 있으므로(PRD-E) "근거" 탭으로 **노출만** — 신규 데이터 아님.

## 6. 통과조건 (수용 시나리오)

- **Quick 모드**: 짧은 입력 → 기존 팝오버 그대로 → 즉시 추가. **동작·성능 회귀 0.**
- **Guided 모드**: 문서 입력 → 도메인 요약 컨펌 → 패턴 발견/선택 → 진행형 생성(병합·용어 해소 카드) → 드리프트/브릿지 컨펌 → 검증 → 커밋 → 발행. 각 단계 **컨펌 시에만** 반영.
- **디자인**: 두 모드가 동일 토큰·primitive·카드 문법으로 보인다(스크린샷 대조 시 이질감 없음).
- **과적재 해소**: Guided에서 어떤 단일 화면도 3개 초과 결정 유형을 동시에 요구하지 않는다.
- **상태 가시성**: 사용자가 언제나 초안/확정/발행 위치를 안다.
