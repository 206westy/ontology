# Ontology Studio v6 — 성능 로드맵: AI 역할 전환 (그림쟁이 → 모델 수호자)

> 작성일: 2026-06-25
> 성격: **기능 추가가 아니라 엔진의 성능 성숙 로드맵**
> 벤치마크: Palantir Foundry / Ontology / AIP
> 척추 결정: **AI를 "추출기"에서 "모델 수호자(Critic)"로 전환** — 여기서 시작

---

## 0. 한 줄 요지 (북극성)

> **스케치북 → 운영 온톨로지.**
> AI가 매 입력마다 새 그래프를 *낳는* 도구에서, **지속되는 하나의 진실(모델)을 수호·접지·강화하는 부조종사**가 되도록 엔진을 성숙시킨다.

성능 = 그래프를 얼마나 예쁘게/빠르게 그리느냐가 **아니라**, 결과 온톨로지가 **믿고 의사결정에 쓸 수 있을 만큼 충실하고·접지되고·일관된가**.

---

## 1. 문제 재정의 — "성능"이란 무엇인가

현재 사용자 체감: **텍스트 입력 → 그래프 생성 → 구경.** 백엔드 능력은 풍부하나(파싱·보강·ER·dedup·임베딩·파티션·커밋/롤백·Neo4j 적재·Text2Cypher) 사용자에게는 "좋은 스케치북"에 머문다.

근본 원인은 **AI와 온톨로지의 관계 방향**:

| | 화살표 | 결과 |
|---|---|---|
| **현재 우리** | `AI → 온톨로지` (AI가 매번 모델을 상상해서 낳음) | 입력마다 일관성 없는 새 그래프. 모델이 AI를 규율하지 못함. 믿고 쓸 수 없음 |
| **팔란티어** | `온톨로지 → AI` (모델이 AI를 접지·구속) | AI가 타입·권한·검증으로 둘러싸인 행동 공간 안에서만 작동. 신뢰 가능 |

**v6의 1차 무기 = 이 화살표를 뒤집는 것.** 그리고 결국 `온톨로지 ⇄ AI` 양방향 루프(깨끗한 모델이 AI를 정확하게, 정확한 AI가 모델을 깨끗하게)로 만든다.

---

## 2. 팔란티어 벤치마크 (왜 Critic이 먼저인가)

팔란티어 온톨로지 3층:

- **Semantic(명사)**: Object Type이 통합 파이프라인의 **실데이터에 바인딩**. 모델이 태생적으로 접지됨.
- **Kinetic(동사)**: 의사결정이 검증·권한·write-back을 가진 **Action**으로 닫힘.
- **Dynamic(AIP)**: LLM이 온톨로지를 **통해서만** 작동. AI의 도구 = 온톨로지의 Action/Function. 모든 제안은 계보 추적 + 검증 게이트.

**핵심 통찰: 팔란티어에서 AI는 온톨로지를 "만드는 자"가 아니라 온톨로지 "안에서 작동하는 행위자"다.** 온톨로지가 AI를 접지·구속·신뢰 가능하게 만든다.

→ 그러려면 **먼저 "지속되고 깨끗한 단일 모델"이 존재**해야 한다. 모델이 엉성하면 접지(Grounder)도 운영추론(Operator)도 틀린 모델 위의 모래성. **그래서 척추의 첫 마디는 Critic** — AI가 모델을 깨끗하고 일관되게 지키는 역할.

## 3. AI 역할 사다리

| 단계 | AI 역할 | 현재 우리 | 팔란티어 | v6 목표 |
|---|---|---|---|---|
| 1. 추출 | 텍스트 → 구조 **초안** | ✅ 여기 고정됨 | 데이터엔지니어 + AI 보조 | 유지(시작점) |
| **2. 비평** | 기존 모델 대비 일관성·중복·설계위반 **검수**, 리팩터 제안 | 일부(gap-detector) | 모델 검증 | **★ Phase 1 — 척추** |
| 3. 접지 | 개념을 **실데이터에 바인딩**·정합 유지 | ❌ | ⭐ 파이프라인(최대 격차) | Phase 2 |
| 4. 운영추론 | 완성·접지된 모델 위 **질의·행동 제안**(가드레일) | ❌ | ⭐ AIP | Phase 3 |
| 5. 관리 | 성장 모델의 **드리프트·거버넌스** 지속 관리 | ❌ | 거버넌스/계보 | Phase 4(상시) |

## 4. 전략 척추: Critic 먼저 (Phase 1 상세)

**전환 명제:** AI의 출력은 더 이상 "새 그래프"가 아니라, **지속되는 단일 모델에 대한 검증된 diff 제안**이다.

### 4.1 단일 지속 모델을 "진실"로 격상
- 현재: 입력 → 새 그래프. 전환: 항상 "현재 모델"이 존재하고, 모든 입력은 그 모델에 대한 **diff 제안**.
- 자산 재사용: `pendingChanges` / Ontology Git(commit·rollback)을 **AI 워크플로의 중심**으로. 산출물이 아니라 규율자.

### 4.2 생성 = 검증된 제안 (Critic 파이프라인)
새 추출분이 들어오면 AI가 **확정 전에** 자동 검수하고, "검수 리포트 + 수정 제안"으로 프리뷰에 표시:

1. **중복·동의어 충돌** — 기존 `similarity` / ER / dedup 자산으로 "이미 모델에 있는 개념인가" 판정 → 재사용 강제.
2. **설계 원칙 위반** — 클래스 vs 인스턴스 혼동, 별모양 허브, 고아 노드, 관계 의미 부정확(같은 문서에 나왔다는 이유의 0-정보 관계).
3. **일관성** — 같은 개념 다른 이름, 모순 관계, 타입 불일치.
4. **기존 모델 정합** — 재사용 vs 신규(parse=assist 동일성).

### 4.3 모델 헬스 상시 점수화
- `HealthDashboard` 자산 활용 → 모델 전체의 충실도·일관성 점수를 **상시 표시**. "성능"이 눈에 보이게(개선의 피드백 루프).

### 4.4 온톨로지 컨벤션 명문화
- AI가 검수할 **설계 규칙 집합**을 코드화(closed-world 검증, 클래스/인스턴스 구분, 관계 정량화 요구 등). v4 온톨로지 전문가 제안 활용.
- 규칙은 데이터·도메인별로 확장 가능하게.

### Phase 1 완료 정의
- [ ] 모든 신규 입력이 "현재 모델 대비 diff 제안 + Critic 리포트"로 들어온다(맹목적 새 그래프 생성 제거).
- [ ] 중복/충돌/설계위반이 확정 전 자동 플래그된다.
- [ ] 모델 헬스 점수가 상시 노출되고 입력 전후로 변동이 보인다.
- [ ] 동일 개념이 두 번 생성되지 않는다(재사용률 측정 가능).

## 5. 단계별 로드맵

| Phase | AI 역할 | 무엇이 좋아지나 | 핵심 산출 | 주요 자산 재사용 |
|---|---|---|---|---|
| **P1. Critic** | 모델 수호자 | **충실도·일관성·신뢰** | 검수 파이프라인, 단일 진실 모델, 헬스 점수, 컨벤션 | similarity, ER, dedup, gap-detector, validation, HealthDashboard, Ontology Git |
| **P2. Grounder** | 접지자 | **접지**(팔란티어 최대 격차) | 개념↔실데이터 바인딩, 인스턴스 현재성, 데이터 매핑 | import/batch, instances, embeddings, 기존 데이터 연결 |
| **P3. Operator** | 운영추론자 | **작동**(돈 의사결정) | 가드레일 질의·행동 제안, 근거경로, 진단형 RAG | RAG entrypoint, Text2Cypher, provenance |
| **P4. Steward** | 관리자(상시) | **확장 시 일관성 유지** | 드리프트 감지, 거버넌스, 계보, 버전 정책 | commits, partitions, constraints |

> 순서 근거: 깨끗한 모델(P1) 없이 접지(P2)·운영(P3)은 모래성. P4는 P1~P3 전 과정에 상시 적용되는 가로축.

## 6. 성능 지표 (북극성 메트릭) — "성능 개선"을 측정 가능하게

| 축 | 지표 | 측정 방법 | 개선 방향 |
|---|---|---|---|
| 충실도 | 별모양 지수(최대노드 관계 집중도) | 단일 노드 방사 관계 수 / 전체 | ↓ |
| 충실도 | 추출 정밀도·재현율 | 골든셋(라벨된 문서) 대비 | ↑ |
| 일관성 | 동일개념 재사용률 | 신규 생성 대신 기존 매칭 비율 | ↑ |
| 일관성 | 중복·충돌 잔존율 | 확정 후 남은 중복 쌍 수 | ↓ |
| 신뢰 | provenance 커버리지 | 출처 달린 노드·관계 비율 | ↑ |
| 신뢰 | confidence 보정도(calibration) | 예측 confidence vs 실제 정답률 | 정합 |
| 신뢰 | 설계위반 건수 | Critic 규칙 위반 수 | ↓ |
| 접지(P2~) | 데이터 바인딩률 | 실데이터에 묶인 노드 비율 | ↑ |
| 운영(P3~) | 진단 정확도 | 과거 케이스 대비 근본원인 적중률 | ↑ |
| 운영(P3~) | 근거경로 제공률 | 결론에 추적 가능한 경로가 붙은 비율 | ↑ |

**골든셋 구축**(라벨된 문서 + 기대 온톨로지)이 P1의 숨은 선결 과제 — 성능을 숫자로 말하려면 정답지가 필요.

## 7. 기존 코드 자산 → Critic 척추로 재배치

| 자산 | 현재 위치 | Critic 역할로 재배치 |
|---|---|---|
| `similarity.ts` | 산발 사용 | 중복·동의어 충돌 1차 게이트 |
| Entity Resolution / dedup | 별도 큐 | 확정 전 자동 검수 단계로 통합 |
| `gap-detector.ts` | 보강 탐지 | 설계위반·빈약노드 검수기로 승격 |
| validation / constraints | 사후 패널 | 입력 시점 사전 검수로 당김 |
| HealthDashboard | 별도 시트 | 상시 헬스 점수 + 입력 전후 델타 |
| Ontology Git(commit/rollback) | 저장 기능 | AI 워크플로의 중심 규율자 |
| provenance(source_type/confidence) | 토대만 | 모든 검수 판단의 근거 기록 |

→ **새로 만드는 것보다 "재배치 + 통합"이 핵심.** 기능 수는 늘리지 않고 엔진의 성능을 올린다.

## 8. 리스크

| 리스크 | 심각도 | 대응 |
|---|---|---|
| Critic 오탐(과잉 플래그)으로 전문가 피로 | High | 심각도 정렬 + 그룹핑, 자동 확정 금지(제안만), 무시 가능 |
| 골든셋 부재로 성능을 숫자로 못 말함 | High | P1 초기에 소규모 골든셋부터 구축 |
| 멀티스테이지 LLM 호출 증가(비용·속도) | Medium | 결정론적 검수 우선, LLM은 정성 판단에만, 병렬화 |
| "단일 진실 모델" 전환이 기존 UX 흐름과 충돌 | Medium | pendingChanges/Undo 호환 유지, 점진 전환 |
| 도메인(반도체) 특수 규칙을 일반 규칙이 못 잡음 | Medium | 컨벤션을 도메인별 확장 가능하게 설계 |
