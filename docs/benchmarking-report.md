## 1. PRD 핵심 요구사항 요약

### 1.1 제품 비전
- **한 줄 요약**: 도메인 전문가가 코드/쿼리 없이 온톨로지를 구축할 수 있는 그래프 편집 스튜디오
- **핵심 가치**: 사용자는 "클래스", "인스턴스", "공리" 같은 용어 없이, 자기 지식을 자유 형식으로 입력하면 LLM이 구조화
- **아키텍처**: "온톨로지의 Git" — Frontend(Next.js) → Staging(Supabase) → Production(Neo4j)

### 1.2 6대 사용자 여정
| # | 여정 | 핵심 인터랙션 |
|---|------|-------------|
| J1 | 최초 온톨로지 생성 | 캔버스 더블클릭 → 자유 입력 → LLM 구조화 → 프리뷰 → 확정 |
| J2 | 기존 노드 편집 | 노드 클릭 → 우측 패널에서 직접 편집 또는 AI 보조 |
| J3 | 관계 연결 | 노드 A→B 드래그 → 관계 팔레트 선택/입력 → 연결 |
| J4 | 계층 이동 | 노드를 다른 노드 위에 드롭 → is-a 관계 자동 생성 |
| J5 | Neo4j 푸시 | 커밋 바 → diff 확인 → Cypher 자동 생성 → 실행 |
| J6 | 점진적 확장 | 기존 그래프 위에 새 데이터 추가 → LLM이 기존 컨텍스트 참조 |

### 1.3 화면 구조 (4대 영역)
| 영역 | 크기 | 역할 |
|------|------|------|
| Explorer (좌측) | 260px 고정 | 계층 트리 + 검색 |
| Graph Canvas (가운데) | flex: 1 | React Flow 기반 그래프 시각화 |
| Right Panel (우측) | 320px 고정 | 선택된 노드의 상세 정보 (Description, Subclasses, Properties, Relations, Constraints, Instances) |
| Commit Bar (하단) | 38px 고정 | 변경사항 추적 + Neo4j 푸시 |

### 1.4 Supabase 스키마 (10개 테이블)
`classes`, `properties`, `instances`, `instance_values`, `relation_types`, `edges`, `axioms`, `axiom_classes`, `commits`, `commit_details`

### 1.5 LLM 통합 포인트
- 자유 입력 구조화 (Claude Sonnet)
- 노드 설명 생성 (Claude Haiku)

## 2. 벤치마킹 분석

### 2.1 Protege (온톨로지 전문 에디터)

**개요**: Stanford 대학에서 개발한 오픈소스 온톨로지 편집기. OWL 2 완벽 지원.

**강점**:
- OWL 2 Web Ontology Language 완벽 지원 + HermiT/Pellet 추론 엔진 내장
- 커스터마이징 가능한 UI로 초보자/전문가 모두 대응
- 온톨로지 병합, 다중 엔티티 리네임 등 리팩토링 기능
- RDF/XML, Turtle, OBO 등 다양한 포맷 지원
- WebProtege 버전으로 협업(공유, 토론, 알림) 지원

**약점 (우리 PRD 관점)**:
- **전문가 지향 UI**: 클래스/프로퍼티/공리 개념을 전제 — 우리 사용자(도메인 전문가)에게 진입장벽
- **자유 형식 입력 없음**: 사용자가 온톨로지 구조를 직접 알고 정의해야 함
- **LLM 통합 없음**: 자연어 → 온톨로지 자동 변환 기능 없음
- **그래프 시각화 제한적**: 시각화 플러그인(OntoGraf, VOWL 등)이 있지만 인터랙티브 편집과는 분리됨

**우리 PRD에 참고할 점**:
- 추론 엔진 개념 → MVP 이후 공리 위반 자동 감지 기능에 참고 가능
- 온톨로지 포맷 export(OWL/RDF) → 현재 "범위 밖"으로 명시되어 있으나, 향후 상호운용성 위해 고려
- 리팩토링 기능(이름 일괄 변경) → 우리도 rename propagation 지원 필요

### 2.2 Neo4j Bloom / Neo4j Browser

**개요**: Neo4j의 그래프 데이터 시각화 도구. 코드 없이 그래프 탐색 가능.

**강점**:
- **자연어 검색**: 검색바에 자연어 패턴 입력 → 그래프 패턴 자동 매칭 (near-natural language)
- **Perspective 시스템**: 데이터 표시 방식을 정의하여 특정 관점에 집중 가능
- **Scene 저장/공유**: 레이아웃과 스타일 상태를 저장하고 읽기 전용으로 공유
- **Histogram 기반 필터/스타일 규칙**: 데이터 분포에 따라 시각적 규칙 적용
- **Bloom 2.0의 코드리스 편집**: 편집 UI 개선

**약점 (우리 PRD 관점)**:
- 이미 존재하는 Neo4j 데이터를 **탐색**하는 도구 — 온톨로지를 **설계/구축**하는 도구가 아님
- 자유 형식 텍스트 → 자동 구조화 기능 없음
- 스키마(온톨로지) 수준의 편집이 아닌 인스턴스 수준의 데이터 탐색에 집중

**우리 PRD에 참고할 점**:
- **자연어 검색 바** → 우리 AI 보조 입력창의 UX 참고 (자연어 → 그래프 조작)
- **Perspective (관점)** 개념 → 대규모 온톨로지 시 특정 도메인만 필터링하는 뷰 (향후 기능)
- **Scene 저장** → 우리 커밋 시스템과 유사한 "상태 스냅샷" 개념
- **Histogram 스타일 규칙** → 노드 크기를 인스턴스 수에 비례시키는 우리 PRD 규칙과 유사

### 2.3 yEd Graph Editor

**개요**: yWorks의 무료 데스크톱 다이어그램 편집기. 자동 레이아웃 강력.

**강점**:
- **다양한 자동 레이아웃**: Force, Hierarchical, Orthogonal, Tree, Radial 등
- **OWL 파일 직접 임포트**: XSLT 스타일시트로 온톨로지 파일 변환 가능
- **HTML 라벨/멀티라인 텍스트**: 노드 내부에 풍부한 정보 표시
- **GraphML 포맷 지원**: 표준 그래프 교환 포맷

**약점 (우리 PRD 관점)**:
- 범용 다이어그램 도구 — 온톨로지 시맨틱(is-a, property, constraint)에 대한 도메인 지식 없음
- LLM/AI 통합 없음
- 웹 기반이 아닌 데스크톱 앱 (별도 설치 필요)

**우리 PRD에 참고할 점**:
- **ELKjs와의 연관성**: yEd의 레이아웃 엔진이 ELK(Eclipse Layout Kernel) 기반 — 우리가 사용하는 ELKjs와 동일 계열
- **Hierarchical 레이아웃** → 우리 is-a 트리 구조에 가장 적합한 알고리즘
- **자동 라벨 포지셔닝** → 엣지 라벨이 겹치지 않도록 자동 배치하는 UX

### 2.4 Obsidian Canvas / Graph View

**개요**: 마크다운 기반 지식 관리 도구. 양방향 링크 + 그래프 뷰.

**강점**:
- **양방향 링크(Bidirectional Links)**: 노트 간 연결이 자동으로 양방향 — 관계 탐색 용이
- **Canvas**: 무한 캔버스에서 노트/이미지/링크를 자유롭게 배치
- **Graph View**: 전체 지식 구조를 노드-링크 다이어그램으로 시각화
- **AI 클러스터링**: 2025년 기준 AI 기반 지식 갭 탐지 기능 추가
- **플러그인 생태계**: 1,000+ 플러그인 (Advanced Canvas 등)

**약점 (우리 PRD 관점)**:
- 노트 기반 — 온톨로지의 클래스/프로퍼티/인스턴스 구분 없음
- 구조화된 스키마가 아닌 자유 형식 링크
- 관계에 타입(relation type)이 없음 — 모든 연결이 동일

**우리 PRD에 참고할 점**:
- **양방향 링크 자동 생성** → 우리도 엣지 생성 시 양쪽 노드 패널에 자동 반영 필요 (incoming/outgoing 표시)
- **Canvas의 자유로운 배치** → 사용자가 노드 위치를 자유롭게 조정 + 자동 레이아웃 병행
- **Graph View의 필터링** → 노드 타입/깊이별 필터 기능 (향후 참고)

### 2.5 tldraw / Excalidraw (캔버스 UX)

**개요**: 무한 캔버스 기반 화이트보드 도구. tldraw는 SDK 중심, Excalidraw는 앱 중심.

**강점**:
- **직관적 캔버스 UX**: 드래그, 줌, 패닝이 자연스러움
- **tldraw**: React 컴포넌트 기반 렌더링 — 캔버스 위에 인터랙티브 UI 삽입 가능
- **Excalidraw**: 핸드드로잉 스타일의 친근한 시각적 언어
- **AI 통합 (2025~)**: tldraw의 computer.tldraw.com — 멀티모달 AI + 캔버스 결합
- **실시간 협업**: 내장 WebSocket 기반 동시편집

**약점 (우리 PRD 관점)**:
- 화이트보드 도구 — 구조화된 그래프 데이터 모델 없음
- 노드/엣지의 시맨틱(타입, 방향, 속성) 지원 안 함
- 자동 레이아웃 없음 (수동 배치만)

**우리 PRD에 참고할 점**:
- **더블클릭으로 요소 생성**: tldraw/Excalidraw의 핵심 패턴 — 우리 PRD의 "캔버스 더블클릭 → 새 노드 팝오버"와 동일
- **팝오버 위치**: 마우스 근처에 뜨는 컨텍스트 메뉴 패턴 — 우리와 동일한 접근
- **캔버스 힌트**: Excalidraw의 하단 단축키 힌트 바 — 우리 PRD의 하단 힌트 바와 동일
- **프레이머 모션 애니메이션**: tldraw의 부드러운 전환 효과 — 우리도 framer-motion으로 구현 예정

### 2.6 Notion DB의 관계형 뷰

**개요**: 데이터베이스 간 Relation/Rollup으로 구조화된 데이터 관리.

**강점**:
- **인라인 편집**: 테이블 셀을 클릭하면 바로 편집 — 저마찰 UX
- **다중 뷰**: 같은 데이터를 테이블/보드/캘린더/갤러리/차트로 보기
- **Relation 속성**: DB 간 연결을 속성으로 정의
- **Rollup**: 연결된 DB의 값을 집계 (합계, 평균 등)

**약점 (우리 PRD 관점)**:
- 그래프 시각화는 네이티브 미지원 (IVGraph 등 서드파티 필요)
- 온톨로지 계층(is-a) 표현 어려움
- 관계 타입이 단순 (이름 있는 1:N 연결만)

**우리 PRD에 참고할 점**:
- **인라인 편집 UX** → 우리 Right Panel의 프로퍼티/Description 인라인 편집 패턴에 직접 참고
- **[+] 버튼으로 행 추가** → 우리의 "[+ 프로퍼티 추가]", "[+ 인스턴스 추가]" 패턴과 동일
- **배지/태그 시스템** → 우리의 프로퍼티 타입 Badge, 상태 enum Badge와 유사

### 2.7 AI 기반 지식그래프 도구 (2025~2026 최신)

**Neo4j LLM Knowledge Graph Builder**:
- 문서 → LLM → 지식그래프 자동 생성
- 커뮤니티 요약, 로컬/글로벌 검색, 커스텀 프롬프트 지원
- 우리 J1(자유 입력 → 구조화)과 유사한 접근

**Graphiti (Zep)**:
- 시간적 컨텍스트 그래프 — 사실의 변화를 추적
- Pydantic 모델로 엔티티/엣지 타입 사전 정의(prescribed) 또는 자동 학습(learned)
- 우리의 relation_types 팔레트(사전 정의) + LLM 자동 추출(학습) 접근과 유사

**Lettria**:
- AI 기반 그래프 온톨로지 편집기
- 우리 제품과 가장 직접적인 경쟁 제품

**시사점**:
- LLM + 지식그래프 조합은 2025~2026년 주요 트렌드 (시장 규모: 2024년 $1.31B → 2028년 $3B+)
- "prescribed + learned 온톨로지" 패턴이 표준화되는 추세
- 우리 제품의 차별점: **인터랙티브 그래프 편집 + LLM 보조**의 결합 (대부분의 도구는 둘 중 하나만)

## 3. 벤치마킹 종합: 우리 PRD에 대한 시사점

### 3.1 우리 PRD가 잘하고 있는 점

| 항목 | 설명 |
|------|------|
| 자유 형식 입력 → LLM 구조화 | Protege/yEd 등 기존 도구에 없는 핵심 차별점 |
| 더블클릭 → 팝오버 패턴 | tldraw/Excalidraw의 캔버스 UX 모범 사례와 일치 |
| Git-like 커밋 시스템 | 기존 온톨로지 도구에 없는 변경 이력 관리 |
| 인라인 편집 UX | Notion 스타일의 저마찰 편집 경험 |
| 관계 팔레트 재사용 | Graphiti의 "prescribed ontology" 패턴과 일치 |
| 노드 타입별 색상 체계 | Neo4j Bloom의 Category 스타일링과 유사한 직관적 시각화 |

### 3.2 보완이 필요한 점 (향후 고려)

| 항목 | 벤치마킹 출처 | 우선순위 | 제안 |
|------|-------------|---------|------|
| 미니맵 | React Flow 내장, Neo4j Bloom | P1 | 대규모 그래프 탐색 시 필수. React Flow MiniMap 컴포넌트 활용 |
| 키보드 단축키 | Protege, Obsidian | P2 | 파워유저를 위한 단축키 체계 (Del 삭제, Ctrl+Z Undo 등) |
| 검색 → 캔버스 포커스 | Neo4j Bloom의 검색 UX | P1 | Explorer 검색 결과 클릭 시 캔버스 줌+패닝으로 해당 노드 포커스 |
| 노드 그룹핑/클러스터 | yEd, Obsidian Graph View | P3 | 관련 노드를 그룹으로 묶어 접기/펼치기 |
| Undo/Redo 스택 | 모든 도구 | P1 | 현재 PRD에 "되돌리기"가 1건만 — 다단계 Undo/Redo 필요 |
| Export 기능 | Protege(OWL), yEd(GraphML) | P3 | 온톨로지 export (OWL/RDF/JSON-LD) — PRD에 "범위 밖"으로 명시되어 있음 |
| 온톨로지 템플릿 | Protege의 프리셋 | P2 | 빈 캔버스 시작 시 도메인 템플릿 제공 (반도체, 제조 등) |
| 자동 완성 제안 | Neo4j Bloom의 검색바 | P1 | AI 보조 입력창에 기존 클래스/관계 자동완성 (Command/cmdk 활용) |

### 3.3 PRD에서 빠져있거나 모호한 부분

| 항목 | 상세 |
|------|------|
| 노드 삭제 플로우 | 노드 삭제 시 하위 클래스/인스턴스 처리 방법이 명시되지 않음 (cascade? promote?) |
| 대규모 그래프 성능 | "범위 밖"으로 명시했지만, 노드 100개 이상 시 기본적인 가상화 필요 |
| 오프라인/에러 처리 | Supabase 연결 실패 시 로컬 버퍼링 전략 없음 |

| 모바일/태블릿 | "범위 밖"이지만, 반응형 최소 대응(패널 숨기기) 정도는 고려 필요 |
