## 0. 한줄요지

"다이나믹 레이어"는 추상적인 제3의 레이어가 아니라 **(a) 보안(PRD-PF-A) + (b) 상태 라이프사이클 + (c) 이벤트-트리거 자동화**로 분해되며, 이 PRD는 (b)(c)를 담당해 "신규 데이터 유입/스케줄 → 결정함수 실행 → 액션 제안"의 준실시간 자동화 파이프라인과 객체별 경량 상태머신을 구체적인 테이블·수용기준으로 확정한다.

---

## 1. 목적

팔란티어 공식 온톨로지 문서는 **semantic + kinetic** 2종 레이어만 정의한다. "dynamic 레이어"는 우리 팀 내부에서 편의상 써온 비공식 개념이며, 사용자 인터뷰에서 이 표현이 "시스템이 알아서 다 해준다(완전자동화)"는 오해로 번지는 것을 확인했다. 이 PRD의 1차 목적은 **그 오해를 걷어내는 것**이다 — dynamic이라는 이름의 레이어를 만들지 않고, 실제로 필요한 두 가지 구체적 기능(상태 관리, 이벤트 반응)만 명시적으로 설계한다.

2차 목적은 PRD-PF-B(결정함수)와 PRD-PF-G(액션보드)를 잇는 **접착 레이어**를 만드는 것이다. 결정함수는 "호출되면 판정한다"까지만 정의하고, 액션보드는 "제안을 사람이 승인/거절한다"까지만 정의한다. **"언제 결정함수를 호출할 것인가"**와 **"판정 후 객체 상태가 어떻게 바뀌는가"**를 정의하는 조각이 빠져 있었고, 이 PRD가 그 빈틈을 채운다.

3차 목적은 **정직한 실시간 경계 긋기**다. 현재 아키텍처(Supabase=SoT, Neo4j=발행, 배치·커밋 지향, 스트리밍 인프라 없음)로는 초저지연 실시간이 불가능하다. 이를 숨기지 않고 MVP 스코프를 "준실시간(폴링/스케줄/DB 변경감지)"로 명시하여 세일즈·고객 기대치를 사전에 관리한다.

| 신규 요소 | 목적 | 비고 |
|---|---|---|
| 스케줄러 (cron worker, 예: Supabase Cron / pg_cron 또는 별도 워커) | 스케줄 트리거 실행 | 신규 상시 프로세스 1개, 최소 침습 |
| 폴링 워커(데이터셋 변경 감지) | `dataset_updated` 이벤트 소스 | MVP는 폴링 기반, DB LISTEN/NOTIFY는 M3 이후 검토 |
| 레이트리밋 카운터(Redis 또는 Supabase 카운터 테이블) | 자동화 폭주 가드 | 신규 인프라지만 경량(카운터 증분/TTL만) |
| 경량 상태머신 검증기(라이브러리 or 자체 룰 엔진) | 전이 가드 평가 | 신규 코드지만 서버 유틸 수준, 별도 서비스 아님 |

### 3-3. 명시적으로 도입하지 않는 것

- 메시지 큐(Kafka/RabbitMQ 등) — 초저지연 스트리밍이 필요해지는 시점(§5-5, §8)까지 보류.
- BPMN급 워크플로우 엔진 — 상태머신은 라이브러리(예: XState 유사 경량 룰)로 충분, 별도 오케스트레이션 플랫폼 도입 안 함.

## 4. 방향 (마일스톤)

**M1 — 데이터모델 + 수동 트리거**
`triggers`/`automation_runs`/`object_state_defs`/`instance_state_log` 테이블 마이그레이션. UI에서 "지금 실행" 버튼으로 트리거를 수동 발화 → 결정함수 호출 → 액션 제안 생성까지 엔드투엔드 1개 경로 확보. 상태머신은 웨이퍼(pending→pass/fail) 1종만.

**M2 — 스케줄/폴링 자동 발화**
cron 워커 + 데이터셋 변경 폴링 워커 도입. 레이트리밋 가드 적용. `automation_runs` 대시보드(실행 이력·성공/실패·감사) 최소 뷰.

**M3 — 상태머신 확장 + 전이 훅**
작업지시(open→in_progress→closed), 이상항목(감지→검토→해결) 상태머신 추가. 전이 시 자동화 훅(예: `closed` 전이 시 재발 방지 함수 재실행) 연결. 시각언어 상태 뱃지(원/점/파선/실선 특색과 일관된 아이콘 체계) UI 반영.

**M4 — 감사·되돌리기 강화 + 신뢰성 하드닝**
자동 실행 전량 되돌리기 UI, 감사로그 검색/필터, 레이트리밋 정책 커스터마이징(워크스페이스별), 트리거 신뢰성 지표(§2) 계측·알림.

## 5. 방법론

### 5-1. "다이나믹 레이어" 재정의 확정

팔란티어 공식 온톨로지는 semantic(무엇이 존재하는가)과 kinetic(무엇을 할 수 있는가) 2종으로 구성된다. 우리가 이전 전략문서에서 언급한 "dynamic 레이어"는 공식 개념이 아니라, semantic/kinetic만으로는 표현하기 애매했던 **"시간에 따라 변하는 것"**을 가리키는 내부 은어였다. 이를 방치하면 사용자에게 "제3의 신비한 레이어가 알아서 움직인다"는 인상을 주므로, 이 PRD에서 다음과 같이 확정 분해한다.

| 분해 요소 | 정의 | 담당 PRD |
|---|---|---|
| (a) 보안 | 접근제어, 권한, 데이터 격리 | PRD-PF-A |
| (b) 상태 라이프사이클 | 객체(인스턴스)가 시간에 따라 거치는 상태와 허용된 전이 | **이 PRD (§5-3)** |
| (c) 이벤트-트리거 자동화 | 외부 이벤트/스케줄이 결정함수(kinetic) 실행을 유발하는 규칙 | **이 PRD (§5-2)** |

즉 "dynamic"은 새 레이어가 아니라 **semantic(상태를 가진 인스턴스) + kinetic(그 상태를 판정하는 결정함수)를 이어주는 운영 규칙 집합**이다. 이후 사내·대외 문서에서 "다이나믹 레이어"라는 표현은 폐기하고 "자동화·상태 라이프사이클"로 통일한다.

### 5-2. 이벤트-트리거 자동화

**흐름:**

```
[이벤트 발생]                [트리거 평가]              [결정함수 실행]           [결과 처리]
데이터셋 갱신     ─┐                                                          ┌─ 판정=이상없음 → automation_run 기록만
스케줄(cron)      ─┼─▶ triggers 매칭 ─▶ 레이트리밋 가드 ─▶ PRD-PF-B 결정함수 ─┼─ 판정=조치필요 → PRD-PF-G 액션 제안 생성
인스턴스 생성/갱신 ─┘   (event_type,scope)  (초과시 skip+기록)   실행           └─ 판정=상태전이 필요 → §5-3 상태머신 가드 평가 → instance_state_log 기록
```

핵심 원칙: **트리거는 "실행 여부"만 결정한다. 트리거 자체가 판정 로직을 갖지 않는다.** 판정은 항상 PRD-PF-B 결정함수에 위임한다. 이는 로직 중복을 막고, 사람이 검토할 지점을 결정함수 하나로 좁힌다.

**신규 테이블: `triggers`**

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid PK | |
| `workspace_id` | uuid FK | |
| `ontology_id` | uuid FK | 대상 온톨로지 |
| `name` | text | 사용자 표시명 |
| `event_type` | enum(`dataset_updated`, `schedule`, `instance_created`, `instance_updated`, `manual`) | 이벤트 소스 종류 |
| `event_config` | jsonb | 예: `{dataset_id}`, `{cron: "*/5 * * * *"}`, `{class_id, property}` |
| `target_function_id` | uuid FK → 결정함수(PRD-PF-B, `constraints.kind='decision'`) | 실행할 판정 로직 |
| `scope` | jsonb | 대상 인스턴스 필터(파티션, 클래스, 속성 조건) |
| `action_template_id` | uuid FK → PRD-PF-G 액션 템플릿, nullable | 제안 생성 시 사용할 카드 템플릿 |
| `rate_limit` | jsonb | 예: `{max_runs_per_hour: 12, cooldown_seconds: 60}` — 자동화 폭주 가드 |
| `enabled` | boolean | |
| `created_by` | uuid FK → users | |
| `created_at` / `updated_at` | timestamptz | |

**신규 테이블: `automation_runs`**

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid PK | |
| `trigger_id` | uuid FK → triggers | |
| `status` | enum(`queued`, `running`, `succeeded`, `failed`, `skipped_rate_limit`, `skipped_disabled`) | |
| `started_at` / `finished_at` | timestamptz | |
| `input_snapshot` | jsonb | 실행 시점 입력 참조(인스턴스 ID 목록, 데이터셋 버전 등 — 재현·감사용) |
| `output` | jsonb | 결정함수 판정 결과(스코어/통과·불통과/근거) |
| `action_proposal_id` | uuid FK → PRD-PF-G `action_items`, nullable | 제안이 생성된 경우만 값 존재 |
| `state_transition_id` | uuid FK → `instance_state_log`, nullable | 상태 전이가 발생한 경우만 |
| `error` | text, nullable | |
| `actor` | enum(`system`, `user`) | 감사 추적 — 수동 발화(M1)는 `user`, 자동 발화(M2+)는 `system` |

**★정직 경계★**: `automation_runs`가 만드는 것은 **제안(`action_items` 또는 상태 전이 후보)까지**다. 이 테이블 자체가 인스턴스의 최종 승인 상태나 외부 시스템 write-back을 확정하지 않는다. 확정은 PRD-PF-G의 사람 승인 액션이 별도로 수행한다.

### 5-3. 상태 라이프사이클 (경량 상태머신)

**신규 테이블: `object_state_defs`**

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid PK | |
| `ontology_id` | uuid FK | |
| `class_id` | uuid FK → classes | 이 상태머신이 적용되는 클래스(예: 웨이퍼, 작업지시) |
| `states` | jsonb | `[{key, label, badge_style}]` — 시각언어(원/점/파선/실선) 매핑 포함 |
| `initial_state` | text | |
| `transitions` | jsonb | `[{from, to, trigger:'user'|'automation', guard}]` |

**신규 테이블: `instance_state_log`**

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid PK | |
| `instance_id` | uuid FK → instances | |
| `from_state` / `to_state` | text | |
| `actor` | enum(`system`, `user:<uuid>`) | |
| `triggered_by_run_id` | uuid FK → automation_runs, nullable | 자동화 훅으로 전이된 경우 |
| `reason` | text, nullable | 사람이 수동 전이 시 사유 입력(강제 아님, closed→open 재오픈 시 권장) |
| `created_at` | timestamptz | |

인스턴스의 "현재 상태"는 정규화 원칙에 따라 `instance_state_log`의 최신 행에서 파생한다(별도 `current_state` 비정규 컬럼은 캐시 최적화 필요 시 M4에서 검토, MVP는 파생 조회).

**예시 상태머신 3종:**

| 대상 | 상태 순서 | 자동화 훅 예 |
|---|---|---|
| 웨이퍼 | `pending → pass / fail` | PRD-PF-F SPC 결정함수 판정 완료 시 자동 전이(사람 승인 불필요 — 판정 자체가 결과이지 "액션 확정"이 아니므로 §8에서 경계 재확인) |
| 작업지시 | `open → in_progress → closed`(+`closed → open` 재오픈) | 이상항목 해결 결정함수가 "재발 없음" 판정 시 `closed` 전이 후보를 액션보드에 제안 → 사람이 확정 |
| 이상항목 | `감지 → 검토 → 해결`(+`검토 → 기각`) | 트리거가 `감지` 상태 인스턴스 생성 → 담당자 배정 시 `검토`로 사람이 수동 전이 |

허용되지 않은 전이(예: `closed → in_progress` 없이 바로 `pending`)는 `transitions` 정의에 없으면 애플리케이션 레이어에서 차단하고 DB에 커밋되지 않는다(§2 지표: 100% 차단).

**시각언어 반영**: 상태 뱃지는 기존 특색(원=클래스, 점=인스턴스, 파선=제안/미확정, 실선=확정)과 일관되게 — 예: 자동화가 만든 상태 전이 "제안"은 파선 뱃지, 사람이 확정한 전이는 실선 뱃지로 표현해 "AI가 제안했을 뿐 확정하지 않았다"는 사실을 시각적으로도 정직하게 전달한다.

### 5-4. 감사 · 되돌리기

- 모든 `automation_runs`는 불변 로그(append-only)로 남는다. 실패/스킵도 기록해 "왜 안 돌았는지"를 추적 가능하게 한다.
- 상태 전이는 `instance_state_log`에 append-only로 남고, "되돌리기"는 삭제가 아니라 **역방향 전이를 새 로그 행으로 추가**하는 방식(커밋 히스토리와 동일 사상 — git revert이지 git reset이 아님).
- 액션 제안이 아직 미확정(파선) 상태면 PRD-PF-G 액션보드에서 그냥 기각(dismiss) — 어떤 인스턴스 데이터도 변경되지 않았으므로 "되돌리기"랄 것도 없음.
- 액션 제안이 확정(실선)되어 인스턴스/그래프에 반영된 후에는 기존 커밋(commits) 인프라의 되돌리기 경로를 그대로 사용 — 자동화가 만든 커밋도 사람이 만든 커밋과 동일하게 취급되어 일관성 유지.

### 5-5. 실시간 경계 — 정직 명시

현재 아키텍처는 Supabase(SoT) ↔ Neo4j(발행)의 배치·커밋 지향 왕복 구조이며, 이벤트 스트리밍 인프라(Kafka류)나 subsecond 처리 경로가 없다. 이 PRD의 MVP 범위는 명시적으로 **준실시간(near-real-time)**이다:

- `schedule` 트리거: cron 정확도(±1분 수준).
- `dataset_updated` 트리거: 폴링 주기(기본 제안치 5분, §8 열린결정) 또는 DB 변경감지(M3+ 검토).
- 초저지연(subsecond~수초) 스트림 처리(예: 설비 센서의 실시간 관리도)는 **이 PRD의 스코프 밖**이며, 별도 인프라(메시지 큐, 스트림 처리기)와 별도 PRD가 필요하다. 세일즈/데모 시 "실시간"이라는 단어 대신 "준실시간 자동 판정"으로 표현할 것을 권고한다.

### 5-6. 리스크

| 리스크 | 완화책 |
|---|---|
| 자동화 폭주(트리거가 무한 반복 실행되거나 서로를 유발) | `rate_limit`(시간당 최대 실행 수 + 쿨다운) 필수 설정, 트리거→트리거 직접 체이닝 금지(자동화는 항상 결정함수 경유), M2부터 알림 |
| 준실시간 지연에 대한 사용자 기대치 불일치 | 온보딩·UI에 폴링 주기/다음 실행 예정 시각 명시, "실시간" 대신 "준실시간" 용어 고정 |
| 상태머신 복잡도 증가(도메인마다 다른 상태 정의 요청 쇄도) | MVP는 3종 프리셋 + 커스텀은 M4 이후, `transitions` jsonb 구조로 확장 가능하게 설계하되 UI 편집은 후순위 |
| 트리거 신뢰성(워커 다운, 실행 누락) | `automation_runs` 상태 계측 + M4 알림, 워커는 idempotent 재시도 설계(동일 이벤트 중복 실행 방지 키) |
| 자동화가 "확정"까지 넘어가는 오해/오남용 | 코드 레벨에서 `automation_runs`가 직접 `action_items.status='approved'`를 쓰는 경로를 원천 차단(PRD-PF-G 권한 모델과 연동) |

## 6. 수용기준

- [ ] `triggers`, `automation_runs`, `object_state_defs`, `instance_state_log` 마이그레이션이 Drizzle 스키마에 반영되고 SoT(Supabase)에서 조회 가능하다.
- [ ] 수동 트리거 발화 → PRD-PF-B 결정함수 호출 → 판정 결과가 `automation_runs.output`에 기록된다.
- [ ] 판정 결과가 "조치 필요"인 경우 PRD-PF-G `action_items`에 파선(미확정) 제안이 정확히 1건 생성된다.
- [ ] 스케줄 트리거가 cron 설정대로 자동 발화하고, 폴링 트리거가 데이터셋 갱신 후 설정 주기 내 발화한다.
- [ ] 레이트리밋 초과 시 실행이 `skipped_rate_limit`로 기록되고 결정함수가 호출되지 않는다.
- [ ] 웨이퍼/작업지시/이상항목 3종 상태머신이 `object_state_defs`에 시드되고, 정의되지 않은 전이 시도가 API 레벨에서 거부된다.
- [ ] 모든 상태 전이가 `instance_state_log`에 append-only로 기록되며, actor(system/user)가 구분된다.
- [ ] 미확정 제안은 삭제/기각만으로 원복되고, 확정 후 변경은 기존 commits 되돌리기 경로로 복원 가능하다.
- [ ] 어떤 코드 경로에서도 `automation_runs` 또는 트리거 실행이 `action_items`를 사람 개입 없이 `approved`로 전이시키지 않는다(코드 리뷰 + 통합테스트로 검증).
- [ ] UI 어디에도 "실시간"이라는 단어가 준실시간 자동화 설명에 단독으로 쓰이지 않는다(카피 리뷰).

## 7. 결론

"다이나믹 레이어"라는 이름은 폐기하지만 그 이름이 가리키던 필요 — 시간에 따라 움직이는 온톨로지 — 는 실재하며, 오히려 이름을 없애고 (b)상태 라이프사이클/(c)이벤트-트리거 자동화로 쪼갠 편이 더 정직하고 더 구현 가능하다. 이 PRD는 신규 인프라를 최소(스케줄러 1개, 폴링 워커 1개, 경량 레이트리밋 카운터)로 억제하면서 기존 결정함수(PRD-PF-B)·액션보드(PRD-PF-G)·커밋 히스토리(git-for-data) 자산을 접착해, "이벤트→판정→제안→사람 확정"이라는 우리 북극성에 부합하는 자동화 루프를 완성한다. 핵심은 **자동화가 결코 승인선을 넘지 않는다**는 것을 데이터모델(파선/실선 뱃지, `action_items.status` 권한 분리)로 강제하는 것이며, 이것이 팔란티어식 "제안 후 사람 검토" 원칙과도 정합한다.

## 8. 열린결정 / 불가기능

**열린결정 (다음 스텝에서 확정 필요):**
1. 폴링 주기 기본값(5분 제안) — 워크스페이스 규모별 차등 적용 여부, DB 부하 실측 후 조정.
2. `dataset_updated` 감지 방식: MVP 폴링 vs M3 DB 변경감지(Supabase Realtime/트리거 함수) 전환 시점.
3. 레이트리밋 기본 정책값(시간당 최대 실행 수, 쿨다운) — 도메인별(SPC vs 정비) 다를 수 있어 워크스페이스 설정 UI 필요 여부.
4. 상태머신 정의를 UI에서 사용자가 직접 편집 가능하게 할 시점(M4 이후 후보) — 그 전까지는 프리셋 3종 + 코드 레벨 확장.
5. 동일 인스턴스에 복수 트리거가 동시에 다른 상태 전이를 제안할 때의 충돌 해소 규칙(우선순위/병합 UI).
6. `automation_runs` 보존 기간·파티셔닝 정책(감사 요구사항에 따라 장기 보관 필요 시 별도 아카이빙 설계).
7. PRD-PF-B 결정함수 엔진의 완성 시점과 이 PRD의 M1 착수 시점 간 동기화(선행 의존성 리스크).

**불가기능 (이 PRD 스코프에서 명시적으로 제외):**
1. ★완전 자율 실행★ — 자동화가 사람 승인 없이 액션을 최종 확정하는 것은 어떤 마일스톤에서도 지원하지 않는다.
2. 초저지연(subsecond~수초) 실시간 스트리밍 처리 — 별도 인프라(메시지 큐/스트림 처리기)와 별도 PRD 필요, 이 PRD는 준실시간까지만.
3. BPMN급 복합 워크플로우(병렬 게이트웨이, 서브프로세스, 장기실행 사가) — 경량 상태머신만 지원.
4. 트리거 조건에 사용자 임의 코드(스크립트) 실행 허용 — 보안상 사전정의된 이벤트 타입 + 결정함수 호출만 허용.
5. 외부 시스템(SAP 등)으로의 자동 write-back — 이 PRD는 액션 제안 생성까지만 담당하며, 외부 시스템 반영은 사람의 수동 작업 또는 별도 커넥터 PRD의 몫이다.
