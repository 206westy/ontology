## 0. 한 줄 요지
문제정의→데이터→온톨로지→키네틱(함수)→보드의 **모든 단계 우측에 동일한 AI 패널을 붙여**, 각 단계에서 "지금 무엇이 충분하고 무엇이 부족한지"를 근거와 함께 **제안(초안)**하되 확정은 항상 사람이 하는 코파일럿을 만든다.

---

## 1. 목적
- 사용자 요구("모든 단계에 AI가 붙어서 데이터 충분성·결측·추천 함수를 자동 제안")를 **단일 패널 UX + 단계별 툴콜**로 구현한다.
- **신규 모델(LLM) 개발이 아니라 기존 AI 자산의 재배치**가 원칙이다. parse/assist/enrich/discover-pattern은 이미 존재하므로, 이를 단계 컨텍스트에 맞춰 호출·조합하는 **오케스트레이션 레이어**를 얹는다.
- 두 개의 신규 핵심 동작만 추가한다: **(2) 데이터 충분성 진단**, **(4) 문제유형→키네틱 함수 추천**. 나머지는 기존 파이프라인을 단계에 배선한다.
- 정직 경계 유지: 완전자동 금지, 모든 제안에 근거·출처(provenance) 표기, 근거 없으면 **'모름'** 반환.

---

## 2. 목표 & 지표

| 목표 | 지표 | 목표치(초기) | 측정 방법 |
|---|---|---|---|
| 전 단계 AI 부착 | AI 패널이 뜨는 단계 수 | 5/5 | 셸 라우트별 패널 렌더 로그 |
| 데이터 충분성 진단 실효성 | 진단→사용자 채택(컬럼 추가/데이터 연결) 비율 | ≥ 40% | 진단카드 accept 이벤트 |
| 함수 추천 정확도 | 추천 함수 중 사람이 confirm한 비율 | ≥ 50% | ActionCard(applied)/추천 수 |
| 과신뢰 억제 | 제안 중 '근거 표기' 비율 | 100% | provenance 필드 누락 0건 |
| 환각 억제 | 근거 부족 시 '모름' 반환율(정상 케이스) | 오판 리콜 기준 관리 | 골든셋 회귀 |
| HITL 보존 | AI가 사람 컨펌 없이 반영한 변경 | 0건 | plan-actions 감사 로그 |
| 응답 성능 | 단계 진입→첫 제안 표시 P50 | ≤ 4s | 클라 계측 |

> 지표는 **채택률·컨펌률** 중심이다. "AI가 많이 제안했는가"가 아니라 "사람이 유용하다고 확정했는가"를 성공으로 본다.

## 3. 기술 스택 — 기존 AI 자산 재배치표

신규 구축을 최소화한다. 각 단계 코파일럿은 아래 자산을 **호출·조합**할 뿐이다.

| 단계 | 재사용(기존 자산) | 신규(최소 추가) | 산출물 |
|---|---|---|---|
| ① 문제정의 | `discover-pattern`(patterns 카탈로그), `assist` | 문제→목표/필요개념 요약 툴콜 프롬프트 | 목표 초안·필요 개념 리스트·유사 패턴 |
| ② 데이터(핵심) | PRD-PF-D 컬럼 프로파일, `enrich`, pgvector | **`sufficiency` 진단 툴콜**(충분성 스코어+결측 근거) | 충분성 카드·결측 데이터 추천 |
| ③ 온톨로지 | `parse`(Stage1 엔티티→Stage2 grounded 관계), `assist`, `critic/review.ts` 9룰 | 컬럼→클래스/속성 매핑 어댑터 | OntologyAction 제안(add_class/property/relation) |
| ④ 키네틱(핵심) | PRD-PF-B Function 엔진, `term/relation_glossary`, patterns | **`function-recommend` 툴콜**(문제유형→함수 템플릿) + 자연어 규칙→조건식 초안 | 함수 추천·조건식 draft |
| ⑤ 보드 | (신규 얇게) 함수 출력 스키마 → 뷰 매핑 규칙 | 뷰 추천 룰(임계/시계열/분포→차트 타입) | 대시보드/액션보드 뷰 제안 |
| 공통 | `ActionCard`(pending/applied/ignored/skipped), `plan-actions.ts` 시뮬레이터, provenance(attributions) | 단계 무관 **CopilotPanel** 셸 컴포넌트 | HITL 컨펌카드·미리보기 |

- **툴콜**: Vercel AI SDK v6 + `@ai-sdk/openai` v3, zod4 스키마로 구조화 출력. 자유 서술이 아니라 `OntologyAction`/`FunctionDraft`/`SufficiencyReport` 등 **타입화된 제안**만 반환.
- **프론트**: PRD-PF-C의 우측 패널 슬롯에 shadcn 기반 `CopilotPanel` 단일 컴포넌트. 단계는 `stage` prop으로 스위칭.

## 4. 방향 (마일스톤: 단계별 AI 동작)

각 마일스톤은 "AI가 무엇을 읽고 → 무엇을 제안하고 → 사람이 무엇을 확정하는가"로 정의한다.

### M1. 공통 셸 — CopilotPanel + 컨펌 파이프
- 우측 고정 패널. 상단 "AI 제안" / 하단 "근거·출처" 2단.
- 모든 제안은 `ActionCard`로 렌더 → `plan-actions.ts` 미리보기 → 사람이 **적용/무시/보류**.
- 근거 부족 신호 시 카드 대신 **'모름 + 무엇이 있으면 판단 가능'** 배지 표시.

### M2. ① 문제정의 단계 AI
- **읽음**: 사용자가 입력한 자연어 문제 서술.
- **제안**: (a) 목표 문장 초안, (b) 사전 액션/필요 개념 리스트, (c) `discover-pattern`으로 유사 문제·패턴(patterns 카탈로그) 추천.
- **확정**: 사람이 목표/개념을 편집·승인 → 다음 단계 컨텍스트로 저장.

### M3. ② 데이터 단계 AI (핵심)
- **읽음**: 연결된 데이터셋의 **컬럼 프로파일**(PRD-PF-D: 타입·결측률·카디널리티·예시값) + 문제정의 컨텍스트.
- **제안**:
  - **충분성 스코어**(0~100) + 판정 근거. "이 문제를 풀기에 컬럼이 충분한가."
  - **결측 데이터 추천**: 문제유형별 필수 컬럼 체크리스트 대비 부족분. 예) SPC 이상탐지 → {측정값, 측정시각, 설비ID, 규격상·하한(스펙)} 필요, 현재 `측정시각`·`스펙` 결측 → "설비 로그에 timestamp 컬럼, 품질기준서에 USL/LSL 연결 필요".
- **확정**: 사람이 컬럼 매핑 보완 / 추가 데이터셋 연결 / "이대로 진행" 선택.
- **주의**: 충분성은 **결정론적 룰(도메인 템플릿의 필수 컬럼 매트릭스)** 우선, LLM은 컬럼 의미 매칭(임베딩)에만 사용 → 오판 리스크 축소.

### M4. ③ 온톨로지 단계 AI
- **읽음**: 승인된 컬럼 + 문제 개념.
- **제안**: `parse` 재사용으로 컬럼→클래스/속성, grounded 관계 초안. 시각언어(원=클래스, 점=인스턴스, 파선=제안, 실선=확정)로 렌더.
- **검수**: `critic/review.ts` 9룰이 제안을 사전 검수(고아 속성, 중복 클래스 등) → 경고 배지.
- **확정**: `OntologyAction` 카드 HITL 컨펌 → 스케치(파선)→운영(실선) 승격.

### M5. ④ 키네틱(함수) 단계 AI (핵심)
- **읽음**: 문제유형 + 확정된 온톨로지 속성.
- **제안**:
  - **문제유형→함수 추천**: 'SPC 이상탐지'→관리도 규칙 함수(Nelson/Western Electric 룰), '출하판정'→임계 판정 함수, 'FDC'→설비 파라미터 이탈 감시. 템플릿 라이브러리(SPC/FDC/정비)에서 소환.
  - **자연어 규칙→조건식 초안**: "3σ 벗어나면 이상" → `value > mean + 3*sigma` draft를 **PRD-PF-B 엔진 스키마**로 반환(속성 읽어 판정하는 kinetic Function).
- **확정**: 사람이 조건식·임계값 편집 → PRD-PF-B에서 실행 등록.
- **근거**: 추천마다 "왜 이 함수인가"(문제유형 매칭 + 참조 템플릿 ID) 표기.

### M6. ⑤ 보드 단계 AI
- **읽음**: 함수 출력 스키마(스칼라/시계열/불리언 판정/분포).
- **제안**: 출력형태→뷰 매핑. 판정(불리언)→액션보드/알림카드, 시계열→관리도 차트, 분포→히스토그램. dataviz 원칙 준수.
- **확정**: 사람이 보드에 뷰 배치 승인.

## 5. 방법론 (프롬프트/툴콜 설계 · 리스크)

### 5.1 툴콜 스키마(예시, zod4)
- `SufficiencyReport { score:number, verdict:'충분'|'부족'|'모름', requiredColumns:{name, role, present:boolean, matchedTo?:string}[], missing:{what, why, howToGet}[], evidence:string[] }`
- `FunctionDraft { problemType, templateId, name, conditionExpr, thresholds, rationale, sources:string[] }`
- `OntologyAction`(기존 재사용), `ViewSuggestion { outputShape, view, rationale }`
- 모든 스키마에 `sources`/`evidence` 필수 → provenance 강제.

### 5.2 프롬프트 원칙
- **결정-우선(Decision-first)**: 충분성/함수 추천의 뼈대는 도메인 템플릿의 필수 컬럼·규칙 매트릭스(코드/설정). LLM은 (a) 컬럼 의미 매칭, (b) 자연어→조건식 번역, (c) 문장 요약만 담당.
- **환각 억제**: "근거 컬럼/템플릿을 인용할 수 없으면 verdict='모름'". 스키마 검증 실패 시 카드 미표시.
- **재현성**: 동일 입력→동일 제안 위해 temperature 낮춤, 템플릿 ID 고정 참조.

### 5.3 리스크 & 완화

| 리스크 | 영향 | 완화 |
|---|---|---|
| **과신뢰**(사용자가 AI 제안 무비판 수용) | 잘못된 온톨로지·함수 운영 반영 | HITL 강제, 파선(제안)/실선(확정) 시각 구분, "근거 보기" 기본 노출, 자동 적용 0건 |
| **충분성 진단 오판** | 필요한데 "충분"으로 통과 / 있는데 "부족" | 결정론 룰 우선, LLM은 매칭만, verdict='모름' 허용, 골든셋 회귀 |
| **함수 추천 정확도**(도메인별) | 엉뚱한 함수 제안 | 템플릿 라이브러리 필수(SPC/FDC/정비), 커버 못하는 문제유형은 추천 대신 '수동 작성 안내' |
| **성능**(단계마다 LLM 호출) | 지연·비용 | 캐시(컬럼 프로파일 해시), 단계 진입 시 1회 호출·수동 재실행 버튼 |
| **provenance 누락** | 정직 경계 위반 | 스키마 레벨 필수 필드, 누락 시 렌더 차단 |

## 6. 수용 기준
1. 5개 단계 모두에서 우측 CopilotPanel이 렌더되고, 각 단계 고유 툴콜을 호출한다.
2. 데이터 단계: 컬럼 프로파일 입력 시 충분성 스코어 + 결측 컬럼 리스트 + 각 항목 근거를 반환한다. 근거 없으면 '모름'.
3. 키네틱 단계: 문제유형 선택 시 최소 1개 이상 함수 템플릿 추천 + 자연어 규칙→조건식 draft를 PRD-PF-B 스키마로 반환한다.
4. 모든 AI 제안은 ActionCard로 뜨고, plan-actions 미리보기를 거쳐 **사람 컨펌 없이는 반영되지 않는다**(감사 로그 0 자동반영).
5. 모든 제안 카드에 provenance(근거·출처) 필드가 채워져 있다(누락 0건).
6. 온톨로지 제안은 critic 9룰 검수 결과 배지를 함께 표시한다.
7. 골든셋(문제유형×데이터셋 샘플)에서 충분성 verdict 오판율이 회귀 기준 내.

---

## 7. 결론
E는 **신규 모델이 아니라 배선(orchestration) PRD**다. 이미 있는 parse/assist/enrich/discover-pattern/critic/plan-actions를 단계 컨텍스트로 재배치하고, 진짜 신규는 **충분성 진단**과 **문제유형→함수 추천** 두 개의 결정론+LLM 하이브리드 툴콜뿐이다. 이렇게 하면 "모든 단계에 AI가 붙는다"는 요구를 충족하면서도 북극성(온톨로지의 Git·스케치→운영·AI=Critic)과 정직 경계(완전자동 금지·근거 표기·모름)를 지킨다. 핵심 성공 조건은 **도메인 템플릿 라이브러리의 품질**이다 — 여기에 충분성 룰과 함수 추천이 모두 의존한다.

## 8. 열린 결정 / 불가 기능

### 열린 결정
1. **충분성 필수 컬럼 매트릭스의 소유권**: PRD-PF-D(데이터) vs 본 PRD(E) 중 어디에 템플릿을 둘 것인가 → E는 소비, D가 프로파일만 제공하는 경계 확정 필요.
2. **함수 템플릿 라이브러리 범위**: 1차 릴리스에 SPC/FDC/정비 3종만 vs 더 확장. 커버리지 밖 문제유형의 UX(추천 없이 수동 안내) 확정 필요.
3. **단계 간 컨텍스트 전달 방식**: 각 단계 승인 결과를 세션 상태(zustand) vs 서버 저장 중 무엇을 소스로 볼지.
4. **LLM 호출 시점**: 단계 진입 자동 1회 vs 사용자가 '분석' 버튼 눌러야 실행(비용·과신뢰 트레이드오프).
5. **충분성 스코어 산식 공개 여부**: 사용자에게 룰 기반 감점 항목을 투명 공개할지, 스코어만 보여줄지.
6. **보드 뷰 추천 룰의 위치**: 본 PRD의 얇은 룰 vs PRD-PF-G(보드)로 이관.

### 불가(또는 하지 않는) 기능
- **완전자동 반영**: AI가 사람 컨펌 없이 온톨로지/함수/보드를 확정하는 기능은 제공하지 않는다(북극성 위반).
- **근거 없는 단정**: 인용 가능한 컬럼/템플릿이 없으면 추천을 지어내지 않고 '모름'을 반환한다.
- **커버리지 밖 도메인 함수 자동 생성**: 템플릿에 없는 문제유형에 대해 조건식을 임의 창작하지 않는다(수동 작성으로 안내).
- **데이터 실측 없는 충분성 확언**: 컬럼 프로파일이 없는 상태에서 "충분/부족"을 단정하지 않는다.
- **AI가 그림쟁이 역할**: AI는 Critic(제안+수호자)이지 사용자를 대신해 온톨로지를 완성하는 주체가 아니다.
